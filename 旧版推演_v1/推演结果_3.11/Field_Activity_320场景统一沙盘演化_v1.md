# Field × Activity 融合版六文档：统一沙盘演化与长期运行审计 v1
## 0. 审计目标
本报告严格基于用户提供的 6 份 Sonnet 融合版文档（已整合 GPT v1.2 稳定性层与 Gemini 极端场景层）进行审计。目标有四个：
1. 在**概念自洽**上检查本体层、runtime projection、桥接层、组合规则层是否一致。
2. 在**状态写回**上检查 durable/ephemeral、WritebackLedger、ContextState、ControlState 与通知/义务/关闭广播是否闭环。
3. 在**极端场景**与**工程故障**上做系统级沙盘演化，覆盖掉线、乱序、重复、重启、分区、时钟漂移、海量并发、世界级灾害等。
4. 形成一份**统一沙盘演化总表**：成功要给证据；失败/部分失败要给证据、原因与问题代码。

本次共构造并逐项记录 **320 个场景**：**100 个正常场景 + 220 个极端场景**。

## 2. 方法
本报告不假定文档外隐含规则。判断标准只有三类：
- **通过**：文档中已有足够明确的状态结构、规则优先级和运行路径，可以闭环。
- **部分通过**：有主要路径，但存在字段漂移、例外未写清、实现边界未明确或跨文档不一致。
- **失败**：缺关键状态机、缺真源对象、缺幂等/并发/时间语义，或文本内部直接矛盾。

### 3.3 由文本直接推出的结构性问题
- **I01**：旧/新状态名并存：foreground_field_id vs ForegroundStack；activity_profile vs current_activity；attention_policy vs baseline_attention_policy；current_object_ref 顶层 vs ActivityInstance.object_ref。
- **I02**：baseline attention 不可写回的原则，与 attention_policy 衰减/旧字段修改逻辑冲突。
- **I03**：ForegroundStack 已引入，但 ContextClosedEvent、重启恢复、若干示例代码仍按单值 foreground_field_id 编写。
- **I04**：durable/ephemeral 写回边界未完全形式化；哪些状态必须先入 WritebackLedger 仍有灰区。
- **I05**：ResourceLease 只在总索引中成型，尚未在桥接层与组合规则层闭合 acquire/renew/release/expire/fence。
- **I06**：overload_mode 已声明，但 admission control / shed / degrade 的触发阈值与执行矩阵不足。
- **I09**：并发保护覆盖不均：agenda 有乐观锁，obligation 有 CAS，但 roles/participants/metadata/ForegroundStack/notifications 无统一并发策略。
- **I13**：Notification TTL 与终局约束存在，但清扫器/压缩器/批队列一致性链条未完全形式化。
- **I14**：大量时间规则依赖绝对时间戳；缺 monotonic clock / clock skew / partition-time semantics。
- **I18**：系统重启恢复段仍引用旧字段（attention_policy、foreground_field_id、activity_profile），与 v1.3 权威状态结构不完全一致。

## 5. 结论先行（跨 320 场景的总体判断）
1. **概念骨架总体成立，但字段迁移未收口**：Sonnet 版已经建立了本体层→projection→bridge→combination 的层级，但正文仍有大量旧字段残留（`foreground_field_id`、`activity_profile`、`attention_policy`、`current_object_ref` 顶层写法）。这会导致实现时出现“双真源”漂移。
2. **写回最强的是 event-first durable 链，但 durable 边界仍不够清**：`WritebackLedger` 的 event-first + replay 设计很强；真正的问题是哪些状态必须 durable、哪些只是 ephemeral，还没有在所有状态对象上写死。
3. **极端场景里最容易破的是时间语义、通知终局、前景栈一致性、排他资源与灾害本体**。
4. **当前文档已经足以支撑“小到中规模”的多智能体持续演化原型，但还不够支撑“海量 agent + 多副本 + 长期运行 + 现实灾害世界”这一更强目标。**

## 7. 跨场景综合发现
- **C1 概念双真源风险**：凡是涉及 `ForegroundStack/current_activity/baseline_attention_policy/ActivityInstance.object_ref` 的场景，只要正文仍写旧字段，就会出现“元文档已升级，但运行示例未升级”的问题。对应 I01/I03/I18。
- **C2 attention baseline 硬冲突**：文档一方面声明 baseline attention 不可写回，另一方面又保留 `decayAttentionPolicy` 直接改 `agent.control.attention_policy` 的示例。这不是细节，而是硬冲突。对应 I02。
- **C3 写回边界不足**：`WritebackLedger` 很强，但 durable/ephemeral 的边界没完全写死，导致 `resume_token`、`sleep_debt`、失败动作、call 生命周期、close 审计等都出现灰区。对应 I04。
- **C4 运行时稳定性层仍有空洞**：`overload_mode`、`Notification TTL sweeper`、全局 fan-out 预算、后台作业顺序、全局 wakeup 去同步等都未完整制度化。对应 I06/I13。
- **C5 并发/多副本/时间语义是最弱环节**：agenda 乐观锁与 obligation CAS 只是局部修复；roles/participants/notifications/ForegroundStack/ResourceLease/双活 agent/clock skew 仍缺统一语义。对应 I05/I09/I14/I15。
- **C6 物理世界与灾害层不够**：place_id 与物理场并集已经比普通聊天系统强，但对于被点名、拍肩、疏散、地震、火山、火灾、停电、道路封锁、医院超载等现实世界强约束，还缺一整层 hazard/feasibility/evacuation 模型。对应 I07/I16。

## 8. 最终判定
### 8.1 现在这套文档已经能稳定支持什么
- 单前景场 + 中断压栈 + 返回恢复
- persistent / bounded / co_located / direct / group 的大部分常规交互
- P0 生存规则、Hysteresis Lock、睡眠保护、deadline/obligation、ContextClosedEvent、基础幂等通知
- event-first durable 写回与基础崩溃恢复
- 中小规模多 agent 的正常协作场景

### 8.2 还不能宣称已经稳定支持什么
- 海量 agent 下的全局 admission / fairness / backpressure / overload state machine
- 双活 agent、多副本、网络分区、乱序 receipt、全局时钟漂移、对象级冲突解决
- ResourceLease 支撑的排他资源（speaker/moderator/host token）
- 世界级灾害、物理强制行为、复杂空间划分、身体可行动性约束
- 数年尺度的 ledger compaction、memory reconciliation、schema migration

### 8.3 建议的下一步（按优先级）
1. **先做 v1.3.1 收口补丁**：把六份文档中所有旧字段残留统一清掉，彻底以 `ForegroundStack/current_activity/baseline_attention_policy/ActivityInstance.object_ref` 为唯一权威。
2. **把 attention baseline 冲突修掉**：要么保留不可写回公设并删掉直接改基线的 decay 示例；要么把 decay 改成单独的、用户/agent 自主确认的“人格更新流程”。
3. **把 ResourceLease 正式下沉到桥接层与组合规则层**：至少补 acquire / renew / release / expire / fencing token。
4. **把 overload_mode 变成正式状态机**：给出进入/退出阈值、shed 规则、全局 fan-out 上限、跨 context wakeup 预算、后台作业优先级。
5. **把 durable/ephemeral 边界写成总表**：逐项列出 ControlState / ContextState / Notification / Obligation / resume_token / sleep_debt / close audit 的 durability。
6. **增补 World Hazard Layer**：至少支持 hazard type、scope、severity、affected place_ids、evacuation_order、shelter_state、infrastructure_state。
7. **给时间与事件加版本语义**：monotonic tick、event epoch、context generation、notification version、receipt ordering。
8. **定义后台作业 orchestration**：TTL sweeper、archive sweeper、deadlock checker、snapshotter、reconciler 的执行顺序与预算。
