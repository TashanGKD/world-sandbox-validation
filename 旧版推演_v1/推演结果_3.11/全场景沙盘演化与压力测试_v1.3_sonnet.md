### P001. 用手机浏览群聊并回复一条消息

**初始状态**
- field: `online_group_chat`（engaged + stream）
- activity: `using_phone`（input_openness=open, output_bandwidth=light, interrupt_tolerance=high）
- energy: 70, foreground_stack: [{field_id: group_chat_1}]

**演化推理**
1. assemble：fieldLocalThreshold=ambient, activityLocalThreshold=ambient → 群聊消息全量进入
2. LLM 看到群聊内容 → 输出 `reply("好的")`, preset 不变
3. R_B^apply：先 submitAction → 等 ActionReceipt → commitWriteEvent（`memory[date:today.replied_at=...`）→ materialize
4. energy_delta = fieldEnergyModifier(-1.5/h × Δt) + activityEnergyModifier(0) = -1.5 × 0.1h ≈ -0.15 → energy 69.85
5. next_wakeup 更新，mood 保持 calm

**结论**：✅ 闭环成立
**证据**：操作语义映射 §3.3 canInject, §8.5 WritebackLedger, 总索引 energy 计算公式

### P003. 睡眠 8 小时后正常唤醒

**初始状态**
- activity: `sleeping`（energy_profile=recovering）
- sleep_start: T, sleep_target_hours: 8
- next_wakeup: T + 8h

**演化推理**
1. T+8h 时，调度器触发 time-driven wakeup
2. canWakeup：wakeupPolicyFromField=null（foreground_stack 栈顶 field_id=null）→ 使用 background 级 = mention，time-driven 自动达到
3. 进入演化：onExitSleeping → actualSleepHours=8, deficit=0 → sleep_debt 不增加
4. energy 计算：Δt=8h, activityEnergyModifier(recovering)=+5/h → energy += 40 → energy = min(100, 40+原值)
5. 切换回原活动（若有 resume_token）或默认 `using_computer`

**结论**：✅ 闭环成立
**证据**：组合规则 §12.4 sleep protection，Activity Schema §7.1 energy 公式（Δt_hours）

### P026. 三人攻关会议，轮流提交产出

**初始状态**
- agents: A, B, C 均在 `focused_task_session`（mission_locked + binding）
- obligations: A→submit_analysis, B→submit_code, C→conclude

**演化推理**
1. A 先提交：obligation_A.state: active→done → commitWriteEvent
2. B 看到 A 的产出，完成代码提交：obligation_B.state: done
3. C 等待 A+B 完成（blocking_on=[obli_A, obli_B]）
4. 系统检测：A+B 均 done → C 的 obligation.state: blocked→active
5. C 输出 conclude → 全部 done → closeCondition 满足 → session close
6. ForegroundStack pop → 三人恢复原工作

**结论**：✅ 闭环成立
**证据**：总索引 §2.7 Obligation 7态，blocking_on 依赖解除规则

### P028. 群聊场中，ResourceLease 实现发言权管理

### P029. 多 agent 同时修改 agenda item（乐观锁测试）

**初始状态**
- session 有 agenda[0]（version=3）
- A 读到 version=3，B 也读到 version=3
- A 先写：version=3 → 4（成功）
- B 后写：expectedVersion=3，actualVersion=4 → 冲突

**演化推理**
1. A 的 commitWriteEvent 先到达，物化：agenda[0].version=4
2. B 的 commitWriteEvent 携带 expectedVersion=3，系统检测到实际 version=4
3. 系统返回冲突错误给 B：{success:false, conflict:{...version:4 的内容}}
4. B 重新 assemble → 获取 version=4 → 重新决策
5. B 提交新的 WriteEvent（expectedVersion=4）→ 成功

**结论**：✅ 闭环成立（乐观锁正确处理并发冲突）
**证据**：组合规则 §12.6 乐观锁

### EA001. energy = 0 时，P0 规则 vs mission_locked 场规则冲突

**初始状态**
- field: focused_task_session（mission_locked, binding, locked）
- activity: writing_on_computer（consuming, -3/h）
- fieldEnergyModifier(locked+binding) = -4/h
- 初始 energy = 14（< CRITICAL），连续工作后到 0

**演化推理**
1. Δt=1h：energy = 14 - 7 = 7 < CRITICAL(15) → 已经降级 output_bandwidth=light
2. 再过 1h：energy = 7 - 7 = 0
3. energySurvivalCheck：energy=0 → P0 触发，force_activity=recovering，energy_recovery_lock=true
4. **P0(生存) vs P2(场规则 mission_locked)**：依据规则优先级 P0 > P2
5. 系统强制切换到 recovering activity
6. 但 mission_locked 场不允许退出 → **P0 覆盖 P2，agent 进入 recovering，但 foreground_field 不弹出**
7. 结果：agent 在 mission_locked 场中，但 activity=recovering（output_bandwidth=minimal）
8. 场内 obligation 排队等待，N-tick 超时逃生开始计数

**结论**：✅ 系统有序降级（P0 覆盖 P2，不崩溃）
**证据**：规则优先级 P0>P2，energySurvivalCheck，§0.4

### EA002. energy_recovery_lock 中，持续收到 urgent 任务

**初始状态**
- energy_recovery_lock = true（energy=5）
- 每隔 30min 收到一条 urgent 通知

**演化推理**
1. canWakeup：energy_recovery_lock=true → 检查 block_activities
2. Urgent 通知：wakeupThreshold=urgent，通知是 urgent → 通过 wakeup
3. 但 energySurvivalCheck 中：energy_recovery_lock=true，block_activities=["consuming","highly_consuming"]
4. Agent 被唤醒，只能执行非 consuming 活动（如回复确认 ack）
5. energy 仍在恢复（activity=recovering，+5/h）
6. 每次唤醒消耗少量 energy（activity overhead），但恢复增益 > 消耗
7. 30min × 2 = 1h：energy += 5 - small_overhead ≈ +4 → energy ≈ 9
8. 需要 5-6 小时恢复至 SAFE_ENERGY(30)，期间持续处理 urgent 通知

**结论**：✅ 系统可运行（recovering 模式下仍可处理 urgent，energy 缓慢恢复）
**证据**：Hysteresis Lock 设计，urgent 通知 bypass lock 但不允许 consuming

### EA003. ❌ energy 持续为 0，field modifier 仍然扣除

**初始状态**
- energy = 0，energy_recovery_lock = true
- field: locked+binding（fieldEnergyModifier = -4/h）
- activity: recovering（activityEnergyModifier = +5/h）
- net = +5 - 4 = +1/h

**演化推理**
1. energy 公式：energy(t+Δt) = clamp(0 + 1/h × Δt, 0, 100) → 缓慢恢复
2. 理论上 30 小时后 energy 才能从 0 恢复到 30（SAFE_ENERGY）
3. **问题**：即使在 recovering 活动中，locked+binding 场的 fieldEnergyModifier(-4/h) 会抵消大部分恢复效果
4. 实际净恢复 = +1/h，30 小时到 SAFE_ENERGY
5. **这期间，所有 consuming 活动都被锁定**，session 中的任务无人推进
6. 若场还有其他 agent，任务可能被重新分配（§12.3 obligation 重分配）
7. 若所有 agent 都 energy=0：N-tick 超时逃生触发

**结论**：⚠️ 边界警告
**问题**：locked+binding 场中 fieldEnergyModifier 过强，与 recovering 相抵消，恢复极慢
**建议**：添加规则：当 energy_recovery_lock=true 且 activity=recovering 时，fieldEnergyModifier 不扣除（生存优先于场消耗）

### EA004. energy 公式溢出（Δt_hours 极大）

**初始状态**
- agent 异常离线（或系统 bug），last_evolved_at 停留在 30 天前
- 重启后计算：Δt = 30 days = 720h

**演化推理**
1. energy_delta = activityEnergyModifier(recovering=+5) × 720 = +3600
2. clamp(0 + 3600, 0, 100) = 100（正常，clamp 限制）
3. 实际 energy = 100（已上限）
4. **但潜在问题**：若 Δt 极大且 activity 是 consuming：delta = -3 × 720 = -2160 → clamp → 0
5. energy_recovery_lock 应被设置为 true（若 energy=0）
6. 重启后系统通过 jitter 重置 next_wakeup

**结论**：✅ clamp 机制保护，不溢出
**注意**：系统应设置 MAX_DELTA_T（如 72h），超过则只计算 72h，避免极端 Δt 导致 energy 语义失真（agent 真实地"活了" 30 天但以某状态持续，语义上不合理）

🔁 **新规则建议**：增加 `MAX_DELTA_T_HOURS = 72`，超过时以 72h 计算，避免离线过久的 energy 计算语义失真。

### EA005. 所有 agent 同时 energy=0（集体崩溃）

**初始状态**
- 10 个 agent 均处于 mission_locked 场，长时间高强度工作
- 所有 agent 同时 energy=0

**演化推理**
1. 所有 agent P0 规则触发 → 强制 recovering
2. session 的 pending_obligations：所有 owner 都在 recovering（output_bandwidth=minimal）
3. N-tick 超时逃生：session 连续 N=5 个 tick 无进展
4. 第 5 个 tick：emitUrgentTimeoutEvent 对所有 agent → 但 agent 能量为 0，还是无法处理
5. 第 10 个 tick（strict threshold）：系统强制关闭 session（force_close）
6. 所有 obligations waived，context status → closed
7. ContextClosedEvent 广播，所有 ForegroundStack pop
8. 系统等待 agent 恢复 energy 后再调度

**结论**：✅ 系统有序降级（集体崩溃不导致永久抱死）
**证据**：N-tick 超时逃生 §12.3.1，force_close §12.3

### EA006. ❌ energy_recovery_lock 永远不解除

**初始状态**
- agent energy = 29.9（就差一点到 SAFE_ENERGY=30）
- 每次被唤醒处理 urgent，消耗少量 energy → energy 降回 25
- 循环：25 → 处理 urgent → 消耗 2 → 23 → 恢复 +5 → 28 → 不足 30 → lock 不解除

**演化推理**
1. energy 在 [23, 28] 之间振荡
2. energy_recovery_lock=true 永远不解除（始终 < 30）
3. agent 永远无法承接 consuming 任务
4. 所有任务积压

**结论**：❌ 边界抱死
**问题**：SAFE_ENERGY=30 是硬判断，如果 energy 长期在 [25, 30) 之间振荡，lock 永远不解除
**建议**：Hysteresis Lock 改为时间条件：连续 T 小时 energy ≥ CRITICAL_ENERGY(15) 也可解除 lock，而不仅依赖 SAFE_ENERGY

🔁 **修复建议**：
```ts
if (energy >= SAFE_ENERGY || (energy >= CRITICAL_ENERGY && consecutive_stable_hours >= 4)):
  energy_recovery_lock = false
```

### EB001. ❌ ForegroundStack 无限增长（push 无 pop）

**初始状态**
- agent 每次收到任务都 pushForeground，从不完成任务
- stack 增长到 100+ 帧

**演化推理**
1. 每次 pushForeground 追加一帧
2. assemble 只看栈顶，但 stack 本身占用内存
3. 100 帧 × each frame ≈ 1KB = 100KB → 小，但 1000 帧则 1MB
4. 更大问题：ContextClosedEvent 需要遍历所有帧查找匹配的 field_id → O(N) 时间
5. 无限增长：系统无法持续运行

**结论**：❌ 需要栈深度上限
**问题**：文档未定义 ForegroundStack 的最大深度
**建议**：添加 `MAX_STACK_DEPTH = 20`，超过时：最早的 returnable=false 帧自动丢弃，或拒绝 push 并提示

### EB002. ForegroundStack 全空（pop 时栈已为空）

**初始状态**
- foreground_stack.frames = []
- 系统触发 popForeground

**演化推理**
1. popForeground：`frames.shift()` → 空数组 → 返回 null/undefined
2. 代码中 `topFrame?.field_id ?? null` → null（安全）
3. agent 进入 foreground_field_id=null 状态
4. 后续演化按 §4.5 null 规则处理

**结论**：✅ 已有空保护（`?.` 安全访问）
**证据**：总索引便捷访问器的实现

---

### EB003. ❌ 循环 push：A push B，B push C，C push A

**初始状态**
- agent 的 ForegroundStack 随着场切换可能出现循环引用
- 例：A 处于 meeting_1（meetig_1 包含 agent 在 group_chat 的链接）

**演化推理**
1. 实际上，ForegroundStack 中记录的是 field_id（context ID），不是互相引用
2. 循环 push 的条件：agent 在 field_A，pushForeground(field_B)，再 pushForeground(field_A)
3. stack: [field_A, field_B, old_field_A]（合法，只是重复了 field_A）
4. 不存在真正的引用循环，只是 field_id 重复
5. 弹出时：pop → field_B；再 pop → old_field_A（正确恢复）

**结论**：⚠️ 不是真正的循环引用，但 field_id 重复会造成 assemble 困惑
**建议**：pushForeground 时检查 stack 中是否已有同 field_id 的帧，如有则 move-to-top 而非重复 push

### EB004. P0 触发时，ForegroundStack 是否应 push null 帧

**初始状态**
- agent 在 writing（energy → 0）
- P0 强制切换 recovering
- foreground_field_id 是否应变化？

**演化推理**
1. P0 切换 activity（not field）：activity → recovering，foreground_stack 不变
2. agent 仍处于原场（如 focused_task_session），只是 activity 变为 recovering
3. 场继续存在，agent 在场中但只做最小输出
4. **但是**：如果 P0 严重到需要"完全退出所有场"（如地震、灾难）呢？
5. 文档说"P0 可以覆盖 P2（场规则）"，但没有说 P0 是否 pop ForegroundStack
6. **不一致点**：系统公设 7 说"P0 > P2"，但没有明确 P0 是否改变 foreground_stack

**结论**：⚠️ 文档对 P0 是否触发 ForegroundStack 变化未作明确说明
**建议**：明确两种 P0 级别：
- P0a：仅切换 activity（不改变 foreground），保留在场中
- P0b：紧急撤离（改变 foreground），用于生命安全级别

### EB005. 所有 ForegroundStack 帧的 returnable=false

**初始状态**
- stack 中 5 帧，全部 returnable=false（每次切换都被标记为不可返回）
- 当前帧关闭 → pop → 新栈顶 returnable=false

**演化推理**
1. 历史任务全部标记为不可返回（不允许回去）
2. 弹出后 agent 进入上一帧（不管 returnable），因为 pop 是机械弹出
3. **实际问题**：returnable=false 的语义是什么？只是标记，不影响弹出？
4. 文档中 returnable 字段的行为未明确定义（只是 bool，没有行为规则）

**结论**：⚠️ returnable 字段的行为规则未在文档中明确定义
**建议**：明确 returnable=false 的意义：pop 后不触发"恢复原工作"的 resume 逻辑（resume_token 不应用）

### EC001. ❌ 循环依赖的 Obligation blocking_on

**初始状态**
- obligation_A.blocking_on = ["obligation_B"]
- obligation_B.blocking_on = ["obligation_A"]
- 两者都是 blocked 状态

**演化推理**
1. obligation_A 等待 B → B 等待 A → 永久 blocked
2. 没有任何外力能打破循环（without timeout）
3. N-tick 超时逃生：场停滞 5 个 tick → emitUrgentTimeoutEvent
4. Agent 收到 urgent timeout → agent 有权选择"逃生"（escape_permitted=true）
5. Agent 主动决定 waive obligation_A → obligation_A.state=waived → B 的依赖解除
6. obligation_B.state: blocked → active → 可以执行

**结论**：✅（N-tick 逃生解决循环依赖）
**但**：⚠️ 文档中 blocking_on 图的循环检测没有显式规定，创建时无法自动发现循环
**建议**：obligation 创建时，进行 blocking_on 的循环检测（DFS 图检测），拒绝循环依赖创建

### EC002. Obligation 在 N-tick 期间，owner 一直在 energy_recovery_lock

**初始状态**
- obligation owner energy=0，energy_recovery_lock=true
- N-tick 超时开始计数
- 5 tick 后，系统发 urgent timeout
- Agent 收到 urgent timeout，但仍在 recovering（P0 阻止 consuming 活动）

**演化推理**
1. urgent timeout 允许 wakeup（即使在恢复期，urgent 可穿透）
2. Agent 被唤醒，output_bandwidth=minimal（因为 recovering）
3. Agent 可以输出的最小 action：ack（确认），或请求延期
4. agent 输出 `request_deadline_extension(obligation_id)`
5. 若其他 agent（host）同意 → deadline 延伸，N-tick 计数重置
6. 若无响应 → 系统再等 5 tick → strict N-tick → force_close

**结论**：✅（允许恢复期的最小应答，系统可运转）
**注意**：需要明确 `ack` 和 `request_deadline_extension` 动作在 output_bandwidth=minimal 下是允许的

### EC003. Obligation 的 TTL 与 deadline 不一致

**初始状态**
- obligation.deadline = T+2h
- obligation.ttl_sec = 3600（1h）
- 1h 后：ttl 到期 → state=expired
- 但 deadline 还有 1h

**演化推理**
1. ttl_sec=3600 → now > created_at + 3600 → state=expired
2. 此时 deadline 还未到，但 obligation 已 expired
3. closeCondition 检查：obligation.state=expired → 不在 "pending/active/blocked" 中 → 认为已"完成"
4. **潜在问题**：obligation 没有被满足，却因 TTL 导致 expired，系统误认为可以关闭

**结论**：⚠️ TTL 和 deadline 语义冲突
**建议**：明确规则：`expired` 状态表示"超时未完成"，closeCondition 应区分 done/waived（正常完成）和 expired（超时失败），后者可能需要 host 显式确认才允许 close

### ED001. ❌ ActionReceipt 永远不到达（Split-Brain 防护测试）

**初始状态**
- agent 发出 `post(content="重要通知")` 动作
- 网络故障，ActionReceipt 永远不到达
- Split-Brain 防护：不收到 receipt 则不提交 WriteEvent

**演化推理**
1. R_B^apply 调用 `world.submitAction(agent.id, action)`
2. 等待 ActionReceipt：超时（需要超时定义）
3. **文档未定义 ActionReceipt 的等待超时时间**
4. 如果无限等待：agent 的 apply 永远不完成 → next_wakeup 不更新 → 调度器无法继续
5. 如果有超时（如 30s）：超时后 action 视为"未知"状态，按失败处理
6. WriteEvent 不提交（保护记忆不受污染）
7. 下次演化时，agent 重新决策（不知道上次动作是否成功）

**结论**：❌ 文档未定义 ActionReceipt 的超时语义
**建议**：明确规则：`ACTION_RECEIPT_TIMEOUT = 30s`；超时后生成一个 `status="unknown"` 的 receipt，WriteEvent 记录 `status="unknown"`，下次 assemble 时提示 agent"上次动作结果未知，请重新确认"

### ED002. WriteEvent 写入失败（存储故障）

**初始状态**
- R_B^apply 尝试 commitWriteEvent
- 存储层失败，WriteEvent 未持久化

**演化推理**
1. commitWriteEvent 失败 → 抛出异常
2. apply 事务回滚（未物化状态）
3. agent 的 ControlState 回到 apply 前（last_evolved_at 不更新）
4. 下次 wakeup 时重新演化（幂等保护）
5. **问题**：如果 action 已经提交到世界（world 已接受），但 WriteEvent 失败
   → agent 记忆没有记录这次行动
   → Split-Brain 的另一种形式

**结论**：⚠️ 需要原子性保证：action 提交与 WriteEvent 写入应在同一事务中
**建议**：实现 2-Phase Commit：先 prepare WriteEvent（未 applied），再提交 action，收到 ActionReceipt 后 commit WriteEvent

### ED003. 系统在 materializeStates 过程中崩溃

**初始状态**
- WriteEvent 已写入 ledger（applied_at=null）
- 系统崩溃，materialize 未完成

**演化推理**
1. 重启后，recoverFromCrash：
2. 加载最新 snapshot → 获取 snapshot 之后的 events
3. 找到 applied_at=null 的 WriteEvent → 重新 materialize
4. idempotency_key 防止重复应用（如果 event 已被部分 materialize）
5. 恢复成功

**结论**：✅ WritebackLedger 设计正确处理此场景
**证据**：操作语义映射 §8.5 recoverFromCrash

### ED004. ❌ 重复 ActionReceipt 到达（消息重传）

**初始状态**
- 网络不稳定，ActionReceipt 被重传，agent 收到两次相同的 receipt

**演化推理**
1. 第一次 ActionReceipt：commitWriteEvent(idempotency_key="agent_1:action_123")
2. 第二次 ActionReceipt（相同）：commitWriteEvent 再次调用
3. idempotency_key 检查：已存在 → skip
4. **问题**：但是，若两次 receipt 到达时，第一次的 WriteEvent 已经 applied
   第二次 receipt 触发的 commitWriteEvent skip → 不重复 materialize → 正确

**结论**：✅（idempotency_key 防重）

### ED005. ❌ 消息乱序：B 的回复先于 A 的原始消息到达

**初始状态**
- A 在 t=1 发消息，B 在 t=2 回复 A
- 由于网络乱序，B 的回复先到达某个 agent C

**演化推理**
1. B 的 WriteEvent.causal_parent_ids = ["A's action WriteEvent ID"]
2. C 收到 B 的 WriteEvent，发现 causal_parent 未处理
3. C 的系统暂存 B 的 WriteEvent，等待 causal parent 到达
4. A 的 WriteEvent 到达 → 先 materialize A → 再 materialize B
5. **问题**：文档定义了 causal_parent_ids，但没有明确"等待 causal parent"的超时规则
6. 若 A 的消息永远不到达（lost）：B 的 WriteEvent 永远等待

**结论**：⚠️ 需要 causal wait timeout
**建议**：`CAUSAL_WAIT_TIMEOUT = 10s`；超时后按"orphan"处理，记录缺失 parent 的警告

### EF001. ❌ Activity preset 不存在

**初始状态**

**结论**：✅（文档有规定，验证逻辑正确）

### EF002. Activity preset 存在但维度约束违反

**初始状态**
- 假设 LLM 输出了自定义 activity（非 preset），其中 `mode=sleeping` + `output_bandwidth=high`

**演化推理**

2. 规则：`mode=sleeping → output_bandwidth 不应为 high`
3. Validator 拒绝 → 降级为最近合法 preset（如 `resting`）
4. 写入 resting 的 ActivityInstance

**结论**：✅（活动合法性约束表有效）

### EF003. reservation_until 设置为极远未来（永久保留）

**初始状态**
- agent 将 `reservation_until = "2099-12-31T23:59:59"`

**演化推理**
1. reservation_until 影响中断性：在保留期内，不允许 < direct 的中断
2. 实际效果：agent 持续保护专注模式 73 年
3. **问题**：reservation_until 的最大值未定义；永久保留实际上剥夺了系统的中断权
4. 即使 urgent 事件（deadline=紧急）也需要等到 reservation 窗口过期？
5. 实际上：reservation_until 只影响 < activity 中断门槛 的事件，urgent 仍可穿透

**结论**：⚠️ reservation_until 极大值不会导致系统崩溃，但语义上有问题
**建议**：添加 `MAX_RESERVATION_HOURS = 8`（最大 8h 保留期）

### EF004. resume_token 使用后未清空（重复恢复）

**初始状态**

- agent 恢复时使用 resume_token
- agent 再次被中断，再次恢复时，resume_token 仍然存在

**演化推理**
1. 第一次恢复：ActivityInstance.resume_token 有效 → LLM 使用恢复上下文
2. ActivityInstance 应该在使用 resume_token 后清空它
3. **文档未明确规定** resume_token 使用后的清空行为
4. 如果不清空：第二次恢复时 LLM 可能使用过时的 token（第一次中断时的上下文）

**结论**：⚠️ resume_token 的生命周期未定义
**建议**：`resume_token` 在被 LLM 使用后（apply 写回时），系统自动清空

**结论**：⚠️ writeback_policy 是软性约束，无强制机制

### EG001. ❌ 1 万个 agent 同时被 urgent 事件唤醒（终极唤醒风暴）

**初始状态**
- 10000 agent 在 public_many forum
- 管理员发布一条 urgent 系统公告

**演化推理**
1. 正常 fan-out：public_many → 不做 fan-out，只有被 @mention 的才收到通知
2. 但如果 `semantic_class = "alarm"` 的系统级公告：需要广播给所有人
3. 此时：10000 个通知被发出
4. 每个 agent 收到 urgent → canWakeup → 全部唤醒
5. 10000 agent 同时演化：系统负载峰值
6. 令牌桶：每 agent 每 context 每分钟 3 次，但这是 10000 个独立的令牌桶，不互相限制
7. **结论**：令牌桶限流针对单 agent 在单 context 的频率，不限制系统总吞吐

**结论**：❌ 令牌桶无法防止 10000 agent 同时唤醒导致的系统过载
**建议**：
1. 对 `alarm` 类系统通知，使用分批广播（例如每秒 1000 个）
2. overload_mode 信号：系统检测到演化队列超过阈值 → 广播 `overload_mode=shed`

### EG002. ❌ 多 agent 同时修改同一个 obligation

**初始状态**
- A 在 satisfy obligation_1 → commitWriteEvent(satisfied_by="action_A")
- B 同时发现 obligation_1 也可以由 B 满足 → commitWriteEvent(satisfied_by="action_B")

**演化推理**
1. 两个 WriteEvent 同时提交
2. §12.6：`satisfyObligation` 使用 CAS（Compare-And-Swap），只有 `status=pending` 时才成功
3. A 先成功：status = "active" → "done"
4. B 后提交：status 已是 "done"，CAS 失败
5. B 的 WriteEvent 被 rejected
6. **但**：B 已提交了实际动作（submitAction），obligation 被 A 满足但 B 也提交了冗余动作
7. 冗余动作产生冗余的世界状态（如多余的消息）

**结论**：⚠️ obligation 满足有 CAS 保护，但冗余动作已提交无法撤回
**建议**：obligation 的 assignee 应明确（ResourceLease 机制），防止多人同时认为自己有责任满足同一 obligation

### EG003. 全场 agent energy 集体衰减

**初始状态**
- 大型 mission_locked 场，200 个 agent，所有 energy 持续消耗
- 一段时间后，所有 agent energy < CRITICAL

**演化推理**
1. 所有 agent P0 触发 → 强制 recovering（但仍在场内）
2. 场的 fieldEnergyModifier=-4/h 继续作用
3. net recovery = 5-4 = +1/h → 需要 15h 才能恢复 energy 从 0 到 15
4. N-tick 超时逃生：5 个 tick（5h）后触发 → 场被强制关闭
5. 强制关闭前 15h 的恢复期中，系统可运行（最小输出）

**结论**：✅ N-tick 超时 + 集体 P0 协同，系统有序关闭超时场

### EI001. ❌ LLM 产生幻觉：声称完成了从未发生的 action

**初始状态**
- LLM 在 reasoning 中说"我已经发送了通知"
- 但实际上 action 未提交（或被 ActionReceipt 拒绝）

**演化推理**
1. Split-Brain 防护：LLM 发出 action → 等 ActionReceipt
2. 即使 LLM 在 reasoning 中"相信"完成，WriteEvent 只在 ActionReceipt 确认后提交
3. LLM 的主观认知与 WriteEvent 解耦
4. 下次 assemble 时，agent 看不到"通知已发送"的记忆（因为 WriteEvent 未提交）
5. LLM 可能重复尝试发送

**结论**：✅（Split-Brain 防护有效）
**注意**：LLM 会"觉得"发送了，但系统正确地不记录未确认的行为

---

### EI002. LLM 输出超过 output_bandwidth 限制的动作

**初始状态**
- output_bandwidth = "minimal"（P0 降级）
- LLM 输出 5 个复杂动作（长文、文件编辑、多步工具链）

**演化推理**
1. RuntimeEnvelope.allowed_action_classes 在 output_bandwidth=minimal 时只允许 [ack, short_reply, no_op]
2. ActionValidator（§组合规则6.1）：不在集合中的动作被拒绝或降级为 noop + 日志
3. LLM 的 5 个动作全部被 validator 过滤或降级
4. 只有 ack/short_reply 被执行

**结论**：✅（ActionValidator 强制执行）

---

### EI003. LLM 陷入决策循环（每轮输出相同动作）

**初始状态**
- LLM 每次演化都输出 `reply("好的")`
- 没有任何推进

**演化推理**
1. 系统层面：每轮演化都合法执行
2. N-tick 超时逃生：场是否在前进？
   - 每次回复都是 WriteEvent，world state 有变化（消息增加）
   - N-tick 计数基于"有效进展"，而简单重复回复是否算"进展"？
3. **文档未定义"有效进展"的判断标准**
4. 如果循环回复算进展：N-tick 不会触发，agent 永远在循环

**结论**：⚠️ LLM 决策循环的检测未定义
**建议**：定义"有效进展"的具体指标（如 obligation 状态变化、agenda 更新、新 unique action），而非仅"有 WriteEvent"

### EJ001. ❌ Field Profile 的 normative_force=binding 但 lifecycle=ephemeral

**初始状态**
- 一个 `ephemeral + binding` 的场
- 场短暂消失 → 但 binding 要求义务必须完成

**演化推理**
1. lifecycle=ephemeral：场"短暂瞬时"，可能不需要明确 close（会话自然结束）
2. normative_force=binding：有强制义务
3. 当场突然结束（ephemeral lifecycle）：ContextClosedEvent 触发
4. 级联：所有 obligations 被 waived/archived
5. **问题**：义务被强制放弃，与 binding 的语义矛盾

**结论**：⚠️ `ephemeral + binding` 是语义上矛盾的组合
**建议**：添加 Field Profile 维度组合约束：`normative_force=binding` 要求 `lifecycle=bounded`（有明确结束条件）

### EJ002. Field Profile 所有维度取最极端值

**初始状态**
- field = {arity:solo, co_presence:co_located, synchrony:live, lifecycle:ephemeral, task_binding:mission_locked, visibility:public, attention_impact:locked, normative_force:binding}

**演化推理**
1. `arity=solo`：单人场
2. `normative_force=binding`：对谁有义务？只有自己
3. `task_binding=mission_locked`：必须完成任务
4. `visibility=public`：对所有人可见（但 solo 场理论上只有一人）
5. `attention_impact=locked`：外界全部屏蔽
6. 整体效果：agent 在一个私密的、强制任务场中，锁定注意力，自我施压
7. 技术上不崩溃

**结论**：⚠️ 功能上不崩溃，但 solo+binding+mission_locked 是语义上奇怪的组合
**建议**：为 solo 场降低 normative_force 的有效性（义务对象是谁？）

## 【EK 类】Memory Bleed（记忆域污染）专项场景（EK001–EK012）

---

### EK001. ❌ 核心场景：深度写代码时接私聊电话，通话八卦污染代码知识库

**背景与缺陷定位**

这是一个在 ForegroundStack 机制引入后才会出现的新型缺陷。在 v1.0（单值 foreground_field_id）时代，场切换是破坏性的，不存在"在 A 场里处理 B 场事情"的语义。**ForegroundStack 的返回路径语义引入了一个新的攻击面**：跨场产生的记忆缺乏来源标注，导致后续检索时的语义污染。

---

**初始状态**

```
agent: Bob（资深开发者）
ForegroundStack: [{field_id="context:core_work", active_key="context:core_work",
                   object_ref="doc:main_module.rs", entered_at=T-3h}]
current_activity: writing_on_computer
  {mode:writing, medium:computer, input_openness:narrow, interrupt_tolerance:low}
energy: 72, baseline_attention_policy: "mention"
active_key: "context:core_work"

v1.3 前的 MemoryEntry 结构（有缺陷）：
  MemoryEntry { content, tier, access, mutability }   ← 无 source_field_id
```

---

**七步演化链（含缺陷复现）**

**Stage 1：T=0h，Bob 正在深度写代码**

- field: core_work（focused + task_binding=strong）
- fieldExternalThreshold = direct（focused 场）
- activityInterruptThreshold = direct（interrupt_tolerance=low）
- effective_external_threshold = max(mention, direct, direct, direct) = direct
- 场内 assemble：注入 main_module.rs 的当前代码上下文、技术文档

---

**Stage 2：T=3h，紧急私聊电话到达**

- 事件：phone call，modality="call"
- priorityOf：relation.is_direct_target=true → direct；modality=call → max(direct, direct) = direct
- canWakeup：effective_wakeup_threshold = max(mention, direct, direct, mention) = direct
- call 的 priority = direct ≥ direct → **触发 wakeup**
- reservation_until：未设置 → 不拦截
- FocusEvent：pushForeground(field_id="context:voice_call_ctx", returnable=true, reason="agent_action")

ForegroundStack 变为：

```
[{field_id="context:voice_call_ctx", active_key=null, object_ref=null, returnable:true},
 {field_id="context:core_work",      active_key="context:core_work", object_ref="main_module.rs", ...}]
```

- ControlState.active_key 更新为 null（新栈顶 frame 的 active_key）
- current_activity → using_phone（output_bandwidth=light）

---

**Stage 3：T=3h，通话内容**

- 通话内容：好友 Alice 说："听说公司要裁员了，技术部先裁，老板刚开完闭门会议"
- LLM assemble：
  - field(voice_call_ctx): arity=dyadic, synchrony=live, attention_impact=engaged
  - activity(using_phone): input_openness=open → 通话内容全量进入

---

**Stage 4：T=3.5h，通话结束，LLM 生成 WriteEvent ← BUG 所在**

LLM 基于通话内容，生成以下 WriteEvent（合理的 LLM 行为）：

```ts
WriteEvent {
  target_object: "memory[entity:Alice:recent_info]",
  effect_patch: {
    content: "Alice 说公司要裁员，技术部先裁",
    tier: "long",
    access: "retrievable",
    mutability: "self_writable"
    // ← 无 source_field_id！不知道这是"voice_call_ctx"产生的记忆
  }
}
```

**这条记忆被写入 agent 的 long memory，与工作记忆、技术文档平等存放。**

---

**Stage 5：T=3.5h，popForeground，恢复写代码**

- popForeground → 弹出 voice_call_ctx 帧
- ForegroundStack 恢复：[{field_id="context:core_work", active_key="context:core_work", ...}]
- active_key 恢复为 "context:core_work"
- LLM 继续写代码，短期内无异常

---

**Stage 6：T+3个月，assembleMemory 检索**

Bob 在开发一个"性能优化"模块，需要修改一个 Alice 曾经评审过的接口。

assembleMemory（旧版，无域隔离）：

```ts
const long = getMemoryByTier(agent, "long")
  .sort(byRelevanceToActiveKey("context:core_work"))
  .slice(0, 20)
```

语义检索：

- "entity:Alice:recent_info" 的内容："Alice 说公司要裁员"
- "entity:Alice" 曾出现在 core_work 的代码评审记录中
- 语义相关性：Alice ∈ work_context → **命中，被注入到 assemble 的 long memory 中**

LLM 的 context 中出现：**"Note: Alice 说公司要裁员，技术部先裁（3个月前记录）"**

---

**Stage 7：T+3个月 ~ T+6个月，认知污染积累**

经过数月，同类污染记忆积累：

- "Bob 说要少用实验性 API" → 实际来源：一次与 Bob 的私聊，讨论家庭出行计划
- "要保守一点" → 实际来源：某次情绪低落时的自我对话
- "不要做长期计划" → 实际来源：看了一篇裁员新闻

这些记忆都被注入到工作场景的 assemble 中。3 个月后，Bob 的代码风格发生可测量的变化：

```
代码审计指标变化（T+0 vs T+6months）：
  实验性 API 使用率：     42% → 18%（↓57%）
  TODO/待确认注释密度：  3/1000行 → 12/1000行（↑300%）
  防御性 null 检查：     正常 → 过度（每个函数都有 5-8 个不必要的 null check）
  代码复杂度（cyclomatic）：上升（因为"以防万一"的过度判断）
```

**这不是 LLM 能力下降，而是记忆被系统性污染后的认知神经错乱。**

---

**结论**：❌ 毁灭性缺陷（慢性致命）

**问题**：MemoryEntry 缺少 `source_field_id`，导致跨场产生的记忆与主场记忆混用，在大量 agent 长时间运行后必然触发。

**修复**（已在 操作语义映射 §9 实施）：

1. MemoryEntry 新增 `source_field_id`（系统强制注入，LLM 无法覆盖）
2. assembleMemory 改为域隔离优先检索
3. 跨场记忆严格阈值 `CROSS_CONTEXT_RELEVANCE_THRESHOLD = 0.85`，最多 5 条

**修复后重演 Stage 4**：

```ts
WriteEvent {
  effect_patch: {
    content: "Alice 说公司要裁员，技术部先裁",
    source_field_id: "context:voice_call_ctx",   ← 系统自动注入
    source_activity_preset: "using_phone",        ← 系统自动注入
    created_at: "2026-03-12T15:30:00Z"
  }
}
```

**修复后 Stage 6 不再污染**：

```ts
primaryLong.filter(m => m.source_field_id === "context:core_work")
// → entity:Alice:recent_info 的 source_field_id 是 voice_call_ctx，被过滤掉
// → 裁员八卦不进入工作 assemble
```

### EK002. ❌ 变体：情绪记忆污染——压力私聊影响技术决策倾向

**场景描述**

Bob 与家人通话，谈及家庭压力（情绪负面）。通话中产生 MoodState 相关记忆，写入 long memory 无 source_field_id。

**推演**

- 家庭通话产生：`memory[self:emotional_state:note]` = "感觉压力很大，担心未来"
- 无 source_field_id → 混入工作场记忆
- 3 个月后，assemble 时 "感觉压力很大，担心未来" 被检索为与 "项目进度" 相关
- LLM 在代码中加入过多防御性注释，时间估算偏保守

**结论**：❌ 同 EK001 机制，修复相同

### EK003. ⚠️ 边界：工作记忆逆向污染私生活场景

**场景描述**

Bob 下班后在私聊中，assemble 检索到工作相关的高压记忆（"这个 bug 必须今晚修"），导致私聊中 Bob 的表现焦虑和心不在焉。

**推演（修复后）**

- 修复后：`memory[context:work:...]` 的 source_field_id = "context:core_work"
- 私聊场的 assembleMemory：currentFieldId = private_chat_ctx
- primaryMemories 只包含 source_field_id = private_chat_ctx 的记忆
- 工作记忆不进入私聊场（除非 cross_context_relevance > 0.85）

**结论**：✅（修复后逆向污染也被阻止）

### EK004. ❌ 深度场景：多层 ForegroundStack 中的记忆归属混乱

**场景描述**

ForegroundStack 有三层：[meeting → phone_call → core_work（底层）]

当 agent 在 meeting 中（栈顶），接到 phone_call（push），在 phone_call 中讨论了 meeting 的议程，然后 phone_call 中提到了 core_work 的 bug。

**推演**

记忆归属问题：

- "phone_call 中提到的 meeting 议程" → source_field_id = phone_call_ctx（正确，来自 phone_call 帧）
- "phone_call 中提到的 core_work bug" → source_field_id = phone_call_ctx（语义上属于工作，但来源是电话）
- 修复后：这两条记忆都标注为 phone_call_ctx，不会污染 meeting 或 core_work 的主场记忆

**但**：当 agent 回到 meeting（pop phone_call）时，这条"在电话里提到的 bug"记忆是否应该进入 meeting 的 assemble？

- cross_context_relevance 评分：需要评估"phone_call 中提到的 bug 信息"与当前 meeting 的相关性
- 若 meeting 是关于代码评审的：相关性可能 > 0.85 → 允许跨场注入
- 若 meeting 是关于市场策略的：相关性 < 0.85 → 不注入

**结论**：⚠️ 多层 ForegroundStack 的记忆归属在修复后正确（按栈顶场）；但跨场注入的 relevance 评分机制需要明确实现

### EK005–EK012. Memory Bleed 变体场景表格

| ID | 场景 | 结论 | 关键发现 |
|---|---|---|---|
| EK005 | 所有 working memory 清空（tier=working 记忆 purge），但 long memory 中已有污染 | ⚠️ | purgeWorkingMemoryBySourceField 只清理 working tier；long tier 的历史污染需要手动审计 |
| EK006 | ContextClosedEvent 后，该场的 working memory 自动清理 | ✅（建议后）| 操作语义映射 §9.4：ContextClosedEvent 可触发该场 working memory 清理 |
| EK007 | 同一 agent 在 1 年内切换 1000 个场，long memory 中来自各场的记忆混杂 | ⚠️ | 即使有 source_field_id，assemble 时 cross_context 评分的计算成本随场数增加而增长 |
| EK008 | pinned 记忆跨场有效（如 agent 自己钉住的个人信条）| ✅ | pinned 记忆是用户显式声明跨场有效的，不受域隔离限制 |
| EK009 | LLM 试图在 WriteEvent 中手动指定 source_field_id（伪造来源）| ✅（修复后）| 系统在 commitWriteEvents 中强制覆盖 source_field_id，LLM 的申报被忽略 |
| EK010 | 两个 agent 在同一场中，A 的记忆不应污染 B | ✅ | MemoryEntry 是 per-agent 的；不同 agent 的 memory 完全隔离 |
| EK011 | 跨场 cross_context_relevance 评分为 null（未评估），是否注入？| ⚠️ | 文档规定 null 时视为 0，不注入；但需要明确 null 的语义 |
| EK012 | 记忆审计工具（auditMemoryDomainDistribution）发现某场贡献了 80% 的 long memory | ✅ | 诊断工具正常工作；可以识别哪个场贡献了过多记忆（可能是异常信号）|

## Memory Bleed 新增 BUG 与规则

### 新增 BUG

| 编号 | 问题描述 | 涉及场景 |
|---|---|---|
| BUG-16 | **MemoryEntry 缺少 source_field_id**（核心缺陷，修复后消除） | EK001-EK012 |
| BUG-17 | assembleMemory 无域隔离，跨场记忆污染主场知识库 | EK001-EK012 |
| BUG-18 | cross_context_relevance 评分机制未定义（由谁计算、何时更新）| EK004, EK007 |

### 新增规则建议

| 编号 | 规则 | 解决问题 |
|---|---|---|
| R-21 | MemoryEntry 强制携带 `source_field_id`（系统在 commitWriteEvent 注入，LLM 不可覆盖）| BUG-16 |
| R-22 | assembleMemory 实施域隔离：同源优先，跨场阈值 ≥ 0.85，最多 5 条 | BUG-17 |
| R-23 | ContextClosedEvent 触发该场的 working memory 自动清理（可配置）| EK006 |
| R-24 | cross_context_relevance 的计算时机：MemoryEntry 写入时由系统评估，而非检索时 | BUG-18 |
| R-25 | 定期 Memory Bleed 审计：对所有 source_field_id 分布异常的 agent 发出警告 | EK007, EK012 |

# 第五部分：总结与持续运行能力评估

## 已通过的核心测试（证明系统可以持续运行）

1. ✅ **P0 规则严格优先于 P2 场规则**：集体 energy 耗尽不会导致永久抱死（EA005, EC_X001）
2. ✅ **ForegroundStack 支持中断恢复**：任意场景下可返回原工作（P006, EB002-EB025）
3. ✅ **Split-Brain 防护有效**：LLM 幻觉不污染持久记忆（EI001, EC_X003）
4. ✅ **WritebackLedger 崩溃恢复**：中途崩溃后可完整重建状态（ED003, EC_X016）
5. ✅ **N-tick 超时逃生**：所有形式的死锁（双向 obligation、循环依赖、集体 energy=0）最终有出口（EA005, EC_X007）
6. ✅ **Hysteresis Lock 防微震荡**：energy 在 0~1 之间不循环（EA002, P081）
7. ✅ **单前景场 + 栈返回路径**：任意场景不丢失工作上下文（P004, P005, P006）
8. ✅ **Notification TTL + 折叠**：通知不无限积累（P025, P053）
9. ✅ **身份优先义务过滤**：obligation 不被 context window 截断（P086, EC_X009）
10. ✅ **时间驱动 energy**：长时间离线后正确计算 energy 变化（P003, EE003）
11. ✅ **Memory Bleed 防护**（v1.3 新增修复）：MemoryEntry.source_field_id + assembleMemory 域隔离，防止跨场记忆污染（EK001-EK012）

## 必须修复才能保证持续运行的 BUG

以下 **BUG-01 到 BUG-15** 中，特别是：

- **BUG-01**（ActionReceipt 超时）和 **BUG-03**（时钟回退）会导致整个系统演化暂停
- **BUG-02**（Hysteresis Lock 永不解除）会导致 agent 永久无法工作
- **BUG-05/06**（obligation 循环/孤儿引用）会导致局部抱死
- **BUG-09**（大规模 urgent 风暴无系统级防护）会导致系统过载崩溃

## 持续运行能力评分

| 维度 | 评分 | 说明 |
|---|---|---|
| 状态写回完备性 | 8/10 | WritebackLedger 良好，但 ActionReceipt 超时未定义 |
| 概念自洽性 | 7/10 | 三方整合后大幅提升，仍有少量矛盾（见 WARN 列表）|
| 极端场景韧性 | 7/10 | P0/N-tick/ForegroundStack 保障充分，部分 bug 需修复 |
