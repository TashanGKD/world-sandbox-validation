# 三方沙盘整合分析与根因修复方案 v1

> 整合来源：
> - 本方沙盘（v1.3_sonnet，306 场景，15 BUG + 17 WARN）
> - 第三方沙盘（320 场景，18 问题代码，87 失败 + 112 部分通过）

## 0. 两份沙盘的核心差异与互补性

| 维度 | 本方沙盘（v1.3_sonnet） | 第三方沙盘（320 场景） |
|---|---|---|
| **强项** | 极端场景推演深度（Energy/Stack/Obligation/Network/Physical） | 系统性覆盖（5×Field × 7×Activity 基线矩阵 + 工程故障） |

| **互补性** | 针对已定义机制进行极端情况压力测试 | 针对文档文字进行严格扫描，找到字段名漂移等 "文档腐烂" 问题 |

## 2. 根因归并：18 个症状 → 7 个根因

所有问题最终归并到 **7 个根本原因**：

```
ROOT-A: 字段迁移未收口（文档腐烂）
ROOT-B: 写回边界未形式化
ROOT-C: 时间语义系统性缺失
ROOT-D: 基础设施概念声明了但没有操作定义
ROOT-E: 物理世界与实体约束的建模边界未声明
ROOT-F: Permission/Role 矩阵未展开
ROOT-G: 多个已知 open issues 需要明确声明为 v2 边界
```

### 根因详解与症状映射

| 根因 | 症状代码 | 本质 |
|---|---|---|
| **ROOT-A** 字段迁移未收口 | I01, I02, I03, I18 | v1.3 升级时新增了 ForegroundStack/ActivityInstance/baseline_attention_policy，但旧字段名在大量正文中残留 |
| **ROOT-B** 写回边界未形式化 | I04, I15, BUG-01 | 没有一张表明确说：哪些状态对象是 durable（必须入 WritebackLedger）、哪些是 ephemeral（可重算）。导致每个场景都要"猜" |
| **ROOT-C** 时间语义缺失 | I14, BUG-03, BUG-13/14 | 系统依赖 wall clock，未定义：monotonic 时钟要求、Δt < 0 时的保护、clock skew 容忍、MIN_WAKEUP_INTERVAL |
| **ROOT-D** 基础设施声明不落地 | I05, I06, I09, I13, BUG-02, BUG-09, BUG-12 | ResourceLease、overload_mode、并发策略、Notification 清扫器、Hysteresis Lock 边界条件 —— 这些在总索引/组合规则里有概念，但没有完整的 acquire/trigger/expire/fence 操作定义 |
| **ROOT-E** 物理世界边界未声明 | I07, I16, EE 类场景 | 系统从未声明"物理实体约束"是否在 v1 范围内。导致地震/火山/被强拉等场景无法判断 |
| **ROOT-F** Permission 矩阵未展开 | I17, N076/077/080 | roles 字段存在，visibility 维度存在，但 (role × action → allowed?) 的矩阵从未写出来 |
| **ROOT-G** open issues 边界模糊 | I08, I10, I11, I12, BUG-10/11/16/17 | Memory Bleed（已修复）、MoodState、mobility 拆分、ForegroundStack 深度、memory reconciliation、RelationInfo —— 这些的处理方式不清楚，导致每个相关场景都标 "部分通过" |

## 3. 根因修复策略：最小化高价值变更

### 修复原则

> **一个修复消灭多个症状。可以在底层的，绝不在表层打补丁。**
>
> 目标：7 个根因 → 7 个底层修复 → 消灭 36 个症状/BUG

---

### ROOT-A 修复：系统性字段迁移（I01/I02/I03/I18）

**问题的本质**：v1.3 的文档升级只更新了"新增章节"，没有系统性替换旧字段名。

**修复方法**：制定一张**权威字段映射表**，然后在所有文档中做一次彻底的机械替换，不遗留任何旧名。

| 旧名（废弃） | 新名（权威） | 所在状态对象 |
|---|---|---|
| `foreground_field_id` | `foreground_stack.frames[0]?.field_id` | ControlState |
| `activity_profile` | `current_activity.profile` | ControlState |
| `attention_policy`（作为基线） | `baseline_attention_policy` | ControlState |
| `control.current_object_ref` | `current_activity.object_ref` | ActivityInstance |
| `activity: string` | `current_activity.preset` | LLM 输出 |

**关于 I02（attention_policy 衰减 vs 不可写回）**：

这两条规则看似矛盾，但实际上**不矛盾**，只是表述混乱：

```
正确语义：
  baseline_attention_policy = agent 的长期人格基线
    → 可以被 LLM/用户显式修改（这是 agent 的自主权）
    → 不能被"场进入"或"活动切换"自动修改（这是公设 2）
    → 可以通过 decayAttentionPolicy 自动衰减（这是 attention_policy 衰减机制）

矛盾的根源：
  "衰减"是 LLM/系统 对 baseline 的主动修改（合法）
  "场/活动写回"是 RuntimeEnvelope 对 baseline 的被动覆盖（违反公设 2）
  两者不冲突，只是需要在文档中明确区分
```

**修复**：在公设 2 后面加一句澄清：
> "LLM 或系统自维护机制（如 attention_policy decay）可以修改 baseline，但这属于 agent 的主动行为，不是场或活动的副作用。"

### ROOT-B 修复：Durable/Ephemeral 权威分类表（I04/I15/BUG-01）

**问题的本质**：文档有 WritebackLedger 的概念，但从未给出一张完整的表，明确说每类状态对象是否需要进 ledger。

**修复方法**：增加一张 **Durable State Classification Table**，一次性解决所有灰区。

```
DURABLE（必须入 WritebackLedger，崩溃后可从 ledger replay 恢复）：
  ControlState（全量，含 ForegroundStack、current_activity、energy、mood 等）
  ContextState（agenda, status, roles, pending_obligations, conclusion）
  ObligationToken（状态变更）
  ResourceLease（acquire, renew, release, expire）
  Notification（status 变更：pending → surfaced → acked → archived/expired）
  ForegroundStack（push/pop 操作）
  MemoryEntry（写入，含 source_field_id）
  AgendaItem（版本变更）

EPHEMERAL（每轮可从 durable 重算，不需要 ledger）：
  RuntimeEnvelope（每轮 assemble 前重算）
  visibleSlice 结果（由 field + observer 决定）
  effective_threshold（由 RuntimeEnvelope 携带）
  energy_delta_model（每轮从 activity + field 派生）
  MoodState.label（从 valence/activation 派生）
  context.hot_messages（从 messages_log 派生）

APPEND-ONLY（无冲突的追加，不需要乐观锁）：
  context.messages（消息序列）
  event_log（事件历史）
  WritebackLedger.events（ledger 本身是 append-only）
```

**关于 I15（Split-Brain 的重复/丢失/乱序 receipt）**：

这是 Durable 写回的一个特殊场景。根本修复是：

```
ActionReceipt 状态机：
  "unknown"（发送后未收到 receipt）
  "success"（receipt.status = ok）
  "failed"（receipt.status = error）
  "duplicate"（幂等 key 已存在）
  "out_of_order"（causal_parent 尚未 apply）

规则：
  1. ACTION_RECEIPT_TIMEOUT = 30s
  2. 超时 → ActionReceipt{status: "unknown"}
     → WriteEvent 标注 status="unknown"，effect_patch 中记录"动作结果未知"
     → 下次 assemble 时，LLM 看到"上次动作结果未知，请确认"
  3. 重复 receipt（幂重 idempotency_key 已存在）→ 跳过，不重复 materialize
  4. 乱序 receipt（causal_parent 未 apply）→ 暂存等待，CAUSAL_WAIT_TIMEOUT=10s，
     超时则按 "orphan" 处理并告警
```

### ROOT-C 修复：时间语义形式化（I14/BUG-03/BUG-13/BUG-14）

**问题的本质**：系统把 Timestamp 当作可信的 wall clock，但没有处理以下三种真实情况：
1. 时钟回退（NTP 同步）→ Δt < 0
2. 时钟跳进（NTP 失联后同步）→ Δt >> 预期
3. 时钟漂移（不同节点的时间不一致）→ 因果顺序不可靠

**修复方法**：定义一套**时间语义规范**，作为所有时间计算的统一基础。

```
TIME SEMANTICS v1（增加到总索引公设层）：

公设 T1：时钟回退保护
  Δt = now - last_evolved_at
  Δt = max(0, Δt)  // 时钟回退时不产生负 Δt
  实现：使用 monotonic clock（process-scoped）作为 Δt 计算基础，
       wall clock 只用于 human-readable 时间戳显示

公设 T2：极端 Δt 截断
  MAX_DELTA_T_HOURS = 72
  Δt_capped = min(Δt_actual, MAX_DELTA_T_HOURS * 3600)
  超过 72h 的 Δt 视为离线过久，以 72h 计算 energy，其余时间视为 null 态

公设 T3：最小唤醒间隔
  MIN_WAKEUP_INTERVAL = 60 seconds
  next_wakeup = max(now + MIN_WAKEUP_INTERVAL, computed_next_wakeup)
  防止 next_wakeup ≤ now 导致的 busy loop

公设 T4：因果顺序不依赖时间戳
  WriteEvent 的顺序由 causal_parent_ids（逻辑时钟）决定，不由 created_at（wall clock）决定
  在分布式部署中，使用 Lamport clock 或 vector clock 作为 causal_parent_ids 的基础

公设 T5：clock skew 容忍
  CLOCK_SKEW_TOLERANCE = 5 seconds
  当 |now_A - now_B| < CLOCK_SKEW_TOLERANCE 时，视为同时发生
  超过容忍则发出 clock_skew_warning
```

这五条公设一次性解决 I14 和所有时间相关 BUG。

### ROOT-D 修复：基础设施操作语义完整化（I05/I06/I09/I13/BUG-02/09/12）

**问题的本质**：ResourceLease、overload_mode、并发保护、Notification 清扫器 —— 这些概念都有，但缺少完整的"状态机 + 触发条件 + 边界"定义。

**修复方法**：为每个基础设施机制增加**最小操作语义**（不是实现代码，而是状态转换规则）：

#### D1：ResourceLease 操作语义（I05）

```
ResourceLease 状态机：
  created → [acquire] → held
  held → [renew before lease_until] → held（刷新 token + lease_until）
  held → [release] → released
  held → [lease_until 到期] → expired（系统 sweeper 检测）
  expired → [acquire by another agent] → held（新 holder）

Fence 语义（防止过期 holder 继续操作）：
  任何 ResourceLease 相关的 WriteEvent 必须携带 lease_token
  系统在 materializeState 时验证 token 是否仍然有效
  过期 token → 拒绝 materialize，返回 lease_expired 错误

调用方：
  acquire(resource_id, agent_id, purpose, lease_duration_sec)
  renew(resource_id, agent_id, current_token)
  release(resource_id, agent_id, current_token)
```

#### D2：overload_mode 触发矩阵（I06）

```
overload_mode 触发规则：
  QUEUE_DEPTH_THRESHOLD_DEGRADE = 10_000  // 演化队列积压超过 1 万
  QUEUE_DEPTH_THRESHOLD_SHED     = 50_000  // 演化队列积压超过 5 万
  CPU_LOAD_THRESHOLD_DEGRADE     = 0.8     // CPU 使用率 > 80%
  MEMORY_THRESHOLD_DEGRADE       = 0.85    // 内存使用 > 85%

  normal:    队列 < 10k AND CPU < 80% AND MEM < 85%
  degraded:  队列 >= 10k OR CPU >= 80% OR MEM >= 85%
    效果：所有 agent 的 recommended_next_wakeup 延长 2x；output_bandwidth 降一档
  shed:      队列 >= 50k OR CPU >= 95% OR MEM >= 95%
    效果：只处理 urgent；non-urgent 通知进入 batch_queue，定时批处理

overload_mode 是全局信号，由系统 health monitor 计算并广播到所有 RuntimeEnvelope
```

#### D3：并发保护策略表（I09）

```
统一并发策略（每类 durable 对象声明一次，不重复讨论）：

对象                      | 并发策略                  | 说明
--------------------------|---------------------------|------------------
context.messages          | append-only               | 无冲突，天然安全
AgendaItem                | 乐观锁（version 字段）    | 已实现
ObligationToken           | CAS（state 字段）         | 已实现
ResourceLease             | token fence               | D1 中定义
ForegroundStack           | per-agent 单写者          | agent 是单线程，无并发
context.roles             | host-only 单写者          | 只有 host 可修改
context.participants      | append-only + host-del    | 加入 append-only，移除需 host
context.pending_obligations | CAS（by obligation id）  | 已有 satisfyObligation CAS
Notification              | append-only（写）+ CAS（状态变更）| 幂等 id 防重
WritebackLedger           | append-only               | 天然安全
```

#### D4：Notification 清扫器操作语义（I13）

```
Sweeper 执行模型（简化后的关键规则，不需要大段代码）：

扫描周期：每 60 秒执行一次（可配置）
扫描对象：所有 status = "pending" 且 created_at < now - MAX_PENDING_AGE 的通知

生命周期规则（按优先级）：
  1. TTL 到期（created_at + ttl_sec < now）→ status = "expired"
  2. collapse_key 相同且有更新版本 → 旧通知 status = "merged"
  3. source_field_id 对应 context 已 closed → status = "archived"
  4. surfaced_count > MAX_SURFACE_COUNT（如 3）→ status = "archived"（防通知轰炸）
  5. 非以上情况 → 保持 pending

Sweeper 失败处理：
  Sweeper 本身宕机不影响业务（通知只是会积压）
  Sweeper 需要 health check；宕机 > 5 分钟发告警
```

#### D5：Hysteresis Lock 边界条件修复（BUG-02）

```
原问题：energy 在 [CRITICAL, SAFE) 之间振荡时，lock 永不解除

修复：双条件解锁（满足任一即可）：
  解锁条件 A：energy >= SAFE_ENERGY（= 30）
  解锁条件 B：energy >= CRITICAL_ENERGY（= 15）且 consecutive_above_critical_hours >= 4
              （连续 4 小时不低于 CRITICAL，即使没到 SAFE 也解锁）
```

#### D6：ForegroundStack 深度管理（BUG-04/12/I12）

```
MAX_STACK_DEPTH = 20

push 时的深度检查：
  if (stack.frames.length >= MAX_STACK_DEPTH):
    // 策略 A（优先）：弹出最早的 returnable=false 帧（无需返回的历史帧）
    const oldest_non_returnable = stack.frames.findLast(f => !f.returnable)
    if (oldest_non_returnable):
      stack.frames.splice(stack.frames.indexOf(oldest_non_returnable), 1)
      push_allowed = true
    // 策略 B（fallback）：如果全部都是 returnable=true
    else:
      // 拒绝 push，但允许 P0 override（P0 强制覆盖栈顶 null 帧）
      if (reason === "system_p0"):
        stack.frames[0] = new_frame  // 强制覆盖栈顶
      else:
        return { success: false, reason: "stack_depth_exceeded" }
```

### ROOT-E 修复：物理世界边界声明（I07/I16/EE 类场景）

**问题的本质**：文档从未声明系统的"物理世界建模范围"，导致所有物理极端场景都无法确定性判断。

**修复方法**：增加一个明确的**物理世界建模边界声明**，一次性解决 26 次 I07/I16 的"部分通过"。

```
v1 物理世界建模边界声明（增加到总索引 §3 公设）：

公设 PW1：v1 的物理世界建模范围
  v1 对物理世界的建模仅包括：
    - agent 的物理位置（physical_location / place_id）
    - co_located 场的空间关系（同一 place_id）
    - 基于位置的 visibleSlice 和动作并集（低头族悖论修复）
    - medium 维度中的 face_to_face / voice_call 等物理媒介类型

公设 PW2：v1 不建模的物理事件
  以下物理事件在 v1 中无本体表示，系统只能通过以下方式间接处理：
    - 自然灾害（地震/火山/洪水）
      → 通过"网络中断 + 用户显式 P0 操作"间接体现
      → 建议：未来版本引入 EmergencySignal 系统事件类型
    - 物理强制事件（被安保拉离、身体碰撞）
      → reactive_local_override 机制可覆盖"被点名"，但无法覆盖"身体约束"
      → 建议：未来版本引入 PhysicalConstraint 本体类型
    - 生命安全级别的强制撤离
      → 依赖用户/系统触发 P0b 级别（完全退出所有场）
```

## 6. 不应该做的事（防止屎山）

以下几件事**看起来要做，但实际上不该做**：

### 6.1 不要为每个 EE 类场景（地震/火山/断电）写专门规则
因为这些场景的本质都是"网络中断 + P0 生存"，已经有足够的处理机制。
正确做法：用 PW2 公设一次性声明边界，不为每个灾害类型写专门规则。

### 6.2 不要把 I11（mobility 拆分）升级为 v1 修复
mobility 拆分成 physical_mobility + context_switchability 会增加 ActivityProfile 的维度数，
破坏现有的 7 维结构。这不是 bug，是有意的简化。声明为 L1-03 并推迟到 v2。

### 6.3 不要把 overload_mode 实现成复杂的自适应算法
overload_mode 的触发条件应该是**简单的阈值比较**，不是 ML 模型或自适应控制。
简单阈值可以在文档层定义并被验证，复杂算法变成"只有运行时才知道"的黑盒。

### 6.4 不要把 RelationInfo 维护（I08）做进 Field × Activity 体系
RelationInfo（社会关系图）是一个独立的子系统，不应该被 Field × Activity 文档包含。
正确做法：声明 "RelationInfo 由外部 Social Graph 模块注入，Field × Activity 仅消费它"。

### 6.5 不要为 Memory Bleed 以外的 memory 问题写补丁
Memory reconciliation（I12 / O-8）是一个复杂问题，
当前的"world_injection 优先于 memory"策略已经是 v1 的合理近似。
不要在没有完整设计的情况下加入更复杂的 reconciliation 逻辑。
