# 导言：为什么这份框架文档存在，以及它要回答什么

## 沙盘推演的本质目标

**沙盘推演（Sandbox Simulation）不是一种测试工具，它是对系统规则体系进行系统性、完备性验证的方法论。**

它存在的原因是：**文字层面的"规则完备性"是幻觉**。一套规则集无论描述得多么详细，只要没有经过在具体状态空间内的因果推演，就无法确认它是否真正闭环——即：在所有合法初始状态和事件序列下，系统是否总能沿着有明确规则的路径演化到合法终态。

本框架要回答的核心问题是：

> **给定一个多智能体系统的规则体系，如何完备地、有逻辑地、可溯源地验证它——包括验证什么、怎么验证、发现问题后怎么分析、以及如何发现规则体系本身的缺陷？**

这个问题的答案，由以下四个子命题构成：

1. **验证什么**：以三层目标—公理—要求体系（T-PA-G）为唯一基准，逐条验证每项目标、公理、要求是否在被测系统中成立
2. **怎么验证**：通过结构化的沙盘场景推演（有明确的四要素标准格式），将规则执行路径显式化、证据化
3. **发现问题后怎么分析**：区分"完全不满足"与"概率性不满足"，后者必须沿三层体系向上溯源，收敛到确定性根因
4. **如何发现体系本身缺陷**：检测两类异常信号——"问题无处归因"与"孤立的多个部分失败"，任一出现均指示 T-PA-G 体系本身有遗漏，需按增量原则修正

# 第一部分：理论骨架——三层目标—公理—要求体系

沙盘推演的所有设计，都以这套体系为骨干和基准。**不理解三层体系，就无法理解为什么沙盘要这样推演，以及推演结果意味着什么。**

---

## 第一章：三层体系的结构与层级关系

### 1.1 三层结构定义

```
T 层（本质目标层）
  ↓ "为了保住这些目标，系统在结构上必须满足什么"
PA 层（抽象公理层）
  ↓ "这些公理在当前具体粒子体系中如何落地"
G 层（当前系统目标层 + 具体要求）
```

**三层的关键属性差异**：

| 层级 | 对所有多智能体系统成立？ | 与具体实现绑定？ | 可跨版本复用？ |
|------|----------------------|----------------|--------------|
| T 层（5个本质目标） | ✅ 是（普遍本质目标）| ❌ 否 | ✅ 完全复用 |
| PA 层（9条抽象公理）| ✅ 是（抽象公理）| ❌ 否 | ✅ 完全复用 |
| G 层（当前系统具体要求）| 仅对本系统成立 | ✅ 是 | ⚠️ 系统升级时需更新 |

**这个差异直接决定沙盘的可扩展性设计**（详见第三部分）。

### 1.2 T 层：五个本质目标

| 目标 | 定义 | 违反后果 |
|------|------|---------|
| **T1 可定义性** | 系统必须能被形式化描述（可区分主体/状态/演化/相互作用）| 根本得不到一个系统 |
| **T2 合法性** | 任意时刻都必须处于合法状态域（值域合法、写边界合法、真源唯一、接口完备）| 后续演化结果不可信 |
| **T3 持续演化** | 在理想条件下能无限时间演化（资源不只耗不补、累积不只增不减、调度不中断）| 永久沉默、资源耗尽 |
| **T4 开放演化** | 不会在非预期条件下收敛到吸收态或无语义进展的病态循环 | 假活性、假稳态、形式运行但语义冻结 |
| **T5 有效耦合** | 主体真实受环境影响，并真实影响环境（双向，缺一不成立）| 各自自循环的 agent 集合，名义上的多主体 |

### 1.3 PA 层：九条抽象公理

| 公理 | 核心含义 | 服务 T 目标 | 若缺失 |
|------|---------|-----------|-------|
| **PA1** 存在性与主体—环境可区分 | 系统状态空间良定义，演化算子对任意合法状态全定义 | T1, T5 | T1：无法定义系统 |
| **PA2** 有界性 | 所有可能无界增长的状态量（资源、记忆、通知、义务等）必须有上界或有界控制 | T2, T3 | T3：有限时间资源耗尽 |
| **PA3** 可恢复性 | 任何可消耗关键资源，必须有从任意耗尽态可达的严格正恢复路径 | T3 | T3：资源单调耗尽停机 |
| **PA4** 持续再进入 | 任意可达状态下，有限时间内必须重新获得演化机会（时间/事件/内部调度至少其一）| T3, T5 | T3：系统永久沉默 |
| **PA5** 写权唯一性 | 任意 durable 状态分量有且仅有一个直接写权限权威者，其他只能通过 apply 路径间接影响 | T1, T2 | T2：状态写权混乱，语义身份丧失 |
| **PA6** 效应有终局 | 任何生成的 effect（通知、义务、事件、任务）必须在有限时间内进入终态 | T2, T3 | T2：状态不确定；T3：积压堵死 |
| **PA7** 信息可达性 | 任何有权接收特定信息的主体，该信息必须在有限时间内（有效期内）到达 | T5 | T5：耦合断裂 |
| **PA8** 语义进展 | 系统必须定义进展投影 P(s)；若系统进入 P(s) 无净变化的循环，必须有不依赖人工干预的逃离机制 | T4 | T4：假活性、病态循环 |
| **PA9** 观察—行动耦合 | 每个主体既能观察环境（observation），又能通过行动产生净效果（action），二者缺一不成立 | T5, T1 | T5：退化为纯被动或盲动 |

### 1.4 G 层：当前系统的具体要求（v6 三粒子体系）

G 层是 PA 公理在当前"三粒子多智能体系统"中的具体化，共 9 个目标：

| G 目标 | 具体化的 PA 公理 | 服务 T | 核心要求摘要 |
|--------|---------------|-------|------------|
| **G0** 存在性与本体结构 | PA1, PA5 | T1 | 三粒子 A+C+B、粒子三元组、S_global/S_durable 分层、Source of Truth Table |
| **G1** 合法状态域（硬不变量）| PA2, PA5, PA6 | T2 | 写隔离、基线不被场/活动污染、energy 有界、通知必须有终局、唯一真源 |
| **G2** 单步演化完备性 | PA1, PA4, PA5 | T3 | 四阶段演化（assemble→evolve→emit→apply）、无死状态、六问判准（G2-R007）|
| **G3** 持续稳定性 | PA2, PA3, PA4, PA6 | T3 | 调度链不断（next_wakeup ≠ null）、能量恢复路径、记忆/通知/context 有界 |
| **G4** 开放演化性 | PA8 | T4 | 有效进展定义（PROGRESS_ACTIONS 白名单）、进展追踪、软/强逃离机制 |
| **G5** 语义有效耦合 | PA7, PA9 | T5 | Field×Activity 分开建模、local/external 阈值分流、动作可达域、义务系统 |
| **G6** 一致性与可追溯性 | PA5, PA6（审计侧）| T1, T2 | durable 变更先写事件再物化、ActionReceipt 机制、因果顺序不依赖 wall clock |
| **G7** 认知连续性（LLM 特化）| PA2, PA4, PA5 | T3 | memory 分层+域隔离（source_field_id）、ForegroundStack 连续性、resume_token |
| **G8** 工程鲁棒性 | PA2–PA9 工程增强 | T3 | busy-loop 防护、crash recovery、split-brain、lease/cleanup、overload、N-tick 逃生 |

**三层完整追溯链示例**：

```
具体要求 G3-R003 [next_wakeup 永不为 null]
  → G3（持续稳定性）
    → PA4（持续再进入：系统不能永久失去演化机会）
      → T3（持续演化：系统能无限时间演化）
```

## 第二章：六问判准——G2-R007 的规范锚定

**闭环六问是三层体系在"单步演化完备性"上的直接落地**，其规范来源是 G2-R007：

> **G2-R007**：高层的任意一条演化规则，至少必须能回答以下六问之一：到达、看见、中断、输出、写回、再调度。不能回答任何一问的规则不应进入高层。
> PA 根源：PA1（全函数演化）+ PA4（持续再进入）+ PA5（写权唯一）+ PA7（信息可达）+ PA9（观察—行动耦合）

**一条规则只有在满足下列六问时，才算闭环规则。否则只是描述性语句，不足以驱动可演化系统。**

| 问题编号 | 对应维度 | 机制函数 | PA 根源 | 检验对象 |
|---------|---------|---------|--------|---------|
| **Q1** 事件能否到达主体？ | 场的暴露机制（Ω）| routeNotification / priorityOf | PA7 | 路由链、门槛映射、fan-out 策略 |
| **Q2** 到达后是否被真正看到？ | 活动的摄取策略（Φ）| canInject / effectiveThreshold | PA7 | assemble 过滤链、义务注入完整性 |
| **Q3** 看到后是否允许打断当前活动？ | 中断门槛 | canWakeup / P0-P4 优先级 | PA4 | 规则冲突仲裁、生命威胁 Override |
| **Q4** 若响应，输出到哪里、以何形式？ | 动作合法性（Γ）| allowedActions / effectiveActions | PA9 | 角色权限、能量门控、mobility 约束 |
| **Q5** 此次交互如何写回个体状态？ | 状态更新规则 | R_B^apply / WriteEvent-first | PA5 | Durable 分类、ActionReceipt 状态机、Memory Bleed 防护 |
| **Q6** 写回后下一步如何调度？ | 下次唤醒策略（Σ）| defaultNextWakeup / Scheduler | PA4 | next_wakeup 不变量、时间公设、N-tick 逃生 |

**六问与 T-PA-G 的对应关系**：

```
六问的六个维度，每个维度都有对应的 PA 公理支撑：
  Q1（到达）  ← PA7（信息可达性）
  Q2（摄取）  ← PA7（信息可达性）+ PA2（有界：不能无限注入）
  Q3（中断）  ← PA4（持续再进入）+ PA8（进展逃离机制）
  Q4（输出）  ← PA9（行动有效性）+ PA2（有界：output_bandwidth 受限）
  Q5（写回）  ← PA5（写权唯一）+ PA6（效应有终局）
  Q6（调度）  ← PA4（持续再进入）+ PA3（可恢复：能量有恢复路径）

六问若任一无法回答 → 对应 PA 公理未在系统中落地 → 对应 T 目标面临威胁
```

## 第三章：沙盘推演的完备定义

### 3.1 什么是沙盘推演

**沙盘推演**是一种以三层目标—公理—要求体系（T-PA-G）为基准，通过在具体状态场景中对规则执行路径进行显式的因果推导，来逐条验证目标、公理与要求是否在被测系统中成立的方法论。

它不是运行时测试（不需要代码），而是：

> **用人脑或等价推理引擎充当规则引擎，在有限的代表性场景空间中，对系统规则集的逻辑完备性、自洽性和极端稳定性进行因果检验，并将每次检验结果通过证据链溯源到三层体系中的具体条目。**

### 3.2 沙盘推演的四要素（强制标准）

每次有效的沙盘推演场景必须包含以下四个要素，缺一即为无效推演：

| 要素 | 定义 | 示例 |
|------|------|------|
| **初始状态** | 所有相关变量的具体值 | `energy=0, field.task_binding=mission_locked, foreground_stack=[ctx_1]` |
| **触发事件序列** | 有序的外部或时间驱动事件 | `t=0: urgent 通知到达 → t=30min: 第二条 urgent 到达` |
| **演化推理链** | 按照规则集对每步状态变化的显式推导 | `step1: energySurvivalCheck → energy_recovery_lock=true → step2: canWakeup(urgent)=true...` |
| **终态判定 + 证据链** | 评判结果（✅/⚠️/❌）+ 引用的 G 层要求 + 追溯到的 PA 公理 | `⚠️ 部分通过，G8-R006 Hysteresis Lock 未满足，根因：PA3 可恢复路径缺失` |

**证据链的要求**：终态判定必须同时给出：
1. 引用的 G 层具体要求（如 G3-R003）
2. 该要求追溯到的 PA 公理（如 PA4）
3. 最终服务的 T 目标（如 T3 持续演化）

这三层引用链缺一不可，否则"部分通过"无法溯源到根因。

### 3.3 可验证、可溯源、证据链完善的标准

一份合格的沙盘推演报告必须满足：

```
可验证性：
  每步演化推理链的每一步，必须引用具体的规则来源（文档章节、函数名、要求编号）
  判定结果不依赖"文档外隐含规则"，只依赖可引用的显式定义

可溯源性：
  每个 ⚠️/❌ 结果，必须能沿 G→PA→T 三层向上追溯
  "部分通过"不是终点，必须继续分析更底层哪条要求未完全满足

证据链完善性：
  结论格式 = 评级 + G层要求编号 + PA公理 + T目标 + 失败模式描述
  示例：❌ 失败 [G6-R006：LLM 等 ActionReceipt 再写回] → [PA5：写权唯一] → [T2：合法性]
        根因：Split-Brain——LLM 在 ActionReceipt 返回前就写回了 durable 记忆
```

## 第四章：失败分析的溯源逻辑

**这是沙盘推演方法论的核心。正确的失败分析路径，是将表层症状收敛到本质根因的唯一可靠路径。**

### 4.1 两种失败类型及其处理差异

#### 类型 A：完全不满足（❌ 失败）

**定义**：在所有测试到的场景中，该条目标/公理/要求均无法被满足。

**处理逻辑**：
```
直接定性 → 该条目标/公理/要求在系统中不成立
记录根因分类 → 归入根因框架（见第六章）
给出修复方向 → 针对缺失的规则定义或机制补全
```

**示例**：
> G6-R005 [durable 写回先写事件再物化] 完全不满足  
> → 系统中所有写回均直接物化，无 WriteEvent 暂存机制  
> → 根因：PA5 写权唯一性未落地（任何组件均可直接写状态）  
> → 修复：引入 WritebackLedger + WriteEvent-first 原则

#### 类型 B：概率性不满足（⚠️ 部分通过）

**定义**：在某些场景下满足，在另一些特定条件下不满足。

**关键原则**：**概率性不满足不是终点，一定意味着存在某个更基础的要求没有被完全满足。**

理由：若系统的规则体系完全正确，则在所有合法输入下，任何一条要求要么总成立，要么总不成立——"有时满足有时不满足"本身就是一个信号，说明还有未被发现的底层缺陷。

**处理逻辑（强制向上溯源）**：

```
Step 1：识别"在哪些条件下失败"
  → 记录触发失败的特定初始状态或事件组合

Step 2：沿 G→PA 方向，检索该 G 要求所具体化的 PA 公理
  → 例：G3-R003 [next_wakeup ≠ null] → PA4（持续再进入）

Step 3：检查 PA 公理是否有其他相关 G 要求
  → PA4 还具体化了：G2-R001（四阶段演化）、G3-R001/R002（调度必须存在）等
  → 逐一在"触发失败的特定条件"下测试这些关联 G 要求

Step 4：找到"在触发条件下完全不满足"的底层 G 要求
  → 该要求即为根因所在层

Step 5：若 G 层根因仍显示"概率性"，继续向上到 PA 层：
  → 检查该 PA 公理的前提条件是否在极端场景下失效
  → 直至收敛到"完全不满足"的确定性结论

输出：从"概率性失败"收敛到"在特定条件 X 下，G 要求 Y 完全不满足，PA 公理 Z 未落地"
```

**示例**：

```
观察：G4-R004 [N-tick 超时逃生] ⚠️ 部分通过
  在普通 agent 中正常；但在 public_many 场中，agent 被水群持续刷新 last_progress_at

Step 1：触发条件 = public_many 场 + LLM 持续发送无效消息

Step 2：G4-R004 → PA8（语义进展公理）

Step 3：PA8 还具体化了 G4-R001 [有效进展定义]

Step 4：在"触发条件"下测试 G4-R001：
  → "有效进展"被错误地等同于"有消息活动"
  → G4-R001 完全不满足（强任务场的进展定义未与 PROGRESS_ACTIONS 白名单绑定）

Step 5：根因确认：
  G4-R001 在 public_many+strong-binding 场中完全不满足
  → PA8：系统未定义真正的进展投影 P(s)，导致"水群"可以无限欺骗超时逻辑
  → T4（开放演化）面临威胁：系统会陷入无语义进展的循环

修复方向：PROGRESS_ACTIONS 白名单限定 last_progress_at 的刷新条件
```

```
                                    ▼
                              收敛到"完全不满足"的
                              PA 公理前提条件缺失
```

### 4.3 失败分析的记录格式

每个 ⚠️/❌ 场景，除标准四要素外，必须额外记录：

```markdown
**失败类型**：完全不满足 / 概率性不满足（注明触发条件）

**溯源链**：
  G 要求：[编号] [名称]
  → PA 公理：[编号] [名称]
  → T 目标：[编号] [名称]

**根因定性**：[一句话描述系统的本质缺陷]

**满足度评估**：
  - 完全未满足（任何条件下都失败）
  - 条件性未满足（触发条件：[描述]）
  - 已有替代机制但不完整（缺失部分：[描述]）

**修复方向**：[具体的规则补全或机制设计]
**修复层级**：G 层规则 / PA 落地机制 / T 层重新定义（极少见）
```

## 第五章：公理体系的元验证——发现 T-PA-G 体系自身的问题

**这是沙盘推演区别于普通测试的关键特性：它不仅验证系统，还能发现验证框架本身的缺陷。**

### 5.1 为什么需要元验证

T-PA-G 体系本身也可能存在缺陷——遗漏了重要的本质约束，或者公理之间存在未被察觉的逻辑缺口。沙盘推演在执行过程中，会产生两类"反常信号"，当它们出现时，应当怀疑并检验 T-PA-G 体系自身是否有问题。

### 5.2 两类异常信号

#### 异常信号 1：问题无处归因（体系存在遗漏）

**现象**：观察到系统存在明确的问题（某些场景下系统行为异常、不可预测、或存在语义矛盾），但沿 G→PA→T 向上追溯时，**在 T-PA-G 体系中找不到任何与之对应的完全不满足的目标、公理或要求**。

**含义**：该问题的根因在 T-PA-G 体系的建模范围之外——体系有**结构性遗漏**，存在尚未被建模的本质约束。

**判断标准**：
```
排除以下情况后，若仍无法归因，则触发体系缺陷检查：
  1. 是否是已知限制（v1 范围边界，见第十章）？→ 若是，则正常
  2. 是否是 G 层的具体实现问题（规则描述不清楚）？→ 若是，则完善 G 层即可
  3. 是否是 PA 公理的 G 层落地不完整？→ 若是，则补全 G 层要求即可
  4. 以上均否，且问题确实存在 → 触发体系缺陷检查
```

**示例**：
> 观察：长期运行的论坛场中，`summarizeRecentPosts` 递归摘要导致历史信息熵退化为幻觉垃圾。  
> 追溯：PA2（有界性）规定了记忆必须有界控制；G3-R006 规定了 memory 必须有压缩/遗忘机制。  
> 但是：体系中没有任何 PA 或 G 层要求规定"摘要机制的语义保真性"——信息熵退化是一种定性的语义失效，不是资源量的无界增长。  
> 判断：触发异常信号 1，提议在 PA 层补充"语义保真性"相关约束（或将其归入 T4 开放演化的语义进展要求）。

#### 异常信号 2：孤立的多个部分失败（体系存在缺口）

**现象**：同时存在两个（或多个）不同的"部分不满足"的目标或要求，但在现有 PA 体系中，这两者**没有内在逻辑关联**（即无法用现有公理将它们解释为同一根因的不同表现，也无法解释为何它们会同时出现）。

**含义**：这两者很可能有更深层的共同根因，但该根因尚未被 PA 体系建模——PA 层可能在两者之间存在**缺失的中间层公理**。

**判断标准**：
```
在以下条件下，认为两个部分失败是"孤立"的：
  1. 将两者的失败溯源到 PA 层，得到不同的 PA 公理
  2. 这两个 PA 公理之间无已知的逻辑依赖关系
  3. 在场景空间中，两者的触发条件存在高度相关性（经常同时出现）
  4. 合理推测存在某种更底层的约束，该约束同时影响两者

此时，提议引入新的 PA 公理或扩展现有 PA 公理，以统一解释两者
```

### 5.3 修正公理体系的原则与方法

当确认 T-PA-G 体系存在问题后，修正必须遵循以下原则，**确保不损耗原有的公理体系**：

#### 原则 1：增量补充，不替换

```
❌ 错误：将现有 PA 公理重新定义，改变其含义
✅ 正确：在现有公理之外新增一条 PA 公理，明确声明其与已有公理的关系
```

#### 原则 2：层级归属严格

```
新增内容必须放在正确的层级：
  - 若对所有多智能体系统普遍成立 → 放 T 层或 PA 层
  - 若仅对当前粒子体系成立 → 放 G 层
  
严禁：将仅对特定系统成立的内容提升到 T/PA 层
严禁：将普遍性约束降级到 G 层（会导致跨版本失去追溯性）
```

#### 原则 3：向上追溯完整

```
每条新增公理或要求必须完成以下检查：
  1. 它服务于哪个 T 目标？（若无，则不应进入主干）
  2. 它是否已被现有 PA 公理隐式覆盖？（若是，则只需扩展 G 层落地）
  3. 它与现有 PA 公理是否有逻辑矛盾？（若有，则需修正表述而非直接添加）
```

#### 原则 4：验证新增后的覆盖性与自洽性

```
新增后必须检查：
  ✓ 每个 T 目标至少有一条 PA 公理服务（覆盖性）
  ✓ 每条 PA 公理服务至少一个 T 目标（不存在悬空公理）
  ✓ G 层中每条要求均可追溯到 PA 公理（裁剪元原则 Gm-R001）
  ✓ 新增公理不与现有公理产生循环依赖
```

#### 修正过程的文档标准

```
新增/修改记录格式：
  变更类型：新增 PA 公理 / 扩展 G 层要求 / 修正 T 目标表述
  触发原因：[异常信号类型 + 具体观察]
  原有体系中的缺口：[哪两个要求之间存在逻辑缺口]
  新增内容：[PA 公理或 G 要求的正式表述]
  与现有体系的关系：[服务 Tx、与 PAy 的关系、具体化为 Gz]
  自洽性检查：[覆盖性验证、无循环依赖确认]
```

# 第三部分：任务分层与可扩展性设计

## 第六章：通用任务与系统特化任务的分离

**可扩展性的核心**：T 层和 PA 层与具体系统无关，G 层与系统绑定。因此，沙盘推演的任务必须对应这个层级，明确哪些任务是跨版本通用的，哪些需要随系统演化而更新。

### 6.1 通用验证任务（T + PA 层，跨版本不变）

通用验证任务只需定义一次，对任何遵循三层体系设计的多智能体系统均适用：

**通用任务 UT-1：持续演化可能性验证（对应 T3 + PA2/3/4/6）**

```
验证目标：系统能否在理想条件下无限时间演化，不因内部机制缺陷耗尽、沉默或爆炸
通用检验方法（PA 层映射）：
  UT-1a [PA2 有界性]：找到所有可能无界增长的状态量，验证每个都有上界控制
  UT-1b [PA3 可恢复性]：对每类可消耗资源，验证存在从耗尽态可达的正恢复路径
  UT-1c [PA4 持续再进入]：在任意可达状态下，验证有限时间内能重获演化机会
  UT-1d [PA6 效应有终局]：找到系统中所有 effect 类型，验证每种有明确终态
通用判定条件：以上 4 条中任一不成立 → T3 面临威胁（持续演化无法保证）
```

**通用任务 UT-2：开放演化性验证（对应 T4 + PA8）**

```
验证目标：系统不会在非预期条件下陷入无语义进展的病态循环
通用检验方法：
  UT-2a：验证系统定义了明确的进展投影 P(s)，可区分"真实推进"与"语义空转"
  UT-2b：验证当 P(s) 无净变化时，存在不依赖人工干预的逃离机制
  UT-2c：验证逃离机制在最坏情况下（软超时 + 强超时）均有明确路径
通用判定条件：P(s) 未定义，或逃离机制缺失 → T4 面临威胁（假活性/假稳态）
```

**通用任务 UT-3：有效耦合完整性验证（对应 T5 + PA7/9）**

```
验证目标：主体真实受环境影响（PA7），并真实影响环境（PA9），双向均成立
通用检验方法：
  UT-3a [PA7 信息可达]：对任意有权接收信息的主体，验证信息能在有效期内到达
  UT-3b [PA9 行动有效]：对任意主体的合法动作，验证其确实能产生可观测的世界效果
  UT-3c [动作端到端闭环]：验证 intent→emit→route→apply→receipt 的完整路径
通用判定条件：PA7 或 PA9 任一不成立 → T5 面临威胁（退化为自循环状态机集合）
```

**通用任务 UT-4：合法状态域完整性验证（对应 T2 + PA5）**

```
验证目标：系统在任意时刻都处于合法状态域，写权唯一且边界明确
通用检验方法：
  UT-4a：找到所有 durable 状态分量，验证每个有且仅有一个直接写权限权威者
  UT-4b：验证跨粒子影响只能通过 emit→route→apply 路径实现，无隐式旁路
  UT-4c：验证所有合法的跨主体影响都有明确定义的接口路径（接口完备性）
通用判定条件：任一 durable 状态分量存在多个写权者，或存在未定义的影响旁路 → T2 面临威胁
```

### 6.2 系统特化验证任务（G 层，随系统版本更新）

特化任务将 PA 公理在当前系统（v6 三粒子 + FA v1.5）中的具体落地要求逐条验证：

**特化任务 ST-G0：三粒子本体结构完整性验证（对应 G0）**

```
特化内容（v6 架构）：
  ST-G0a：验证三粒子（A 世界/C 耦合介质/B 智能体）的独立存储边界
    检验：/world/ / /coupling/ / /agents/*/ 三目录的写权限是否严格隔离
  ST-G0b：验证 S_global = S_total（v6 三粒子简化后的状态空间完整性）
    检验：V_C 是否完整承载了所有游离状态（notifications/leases/ledgers/obligations 索引）
  ST-G0c：验证 Source of Truth Table（关键概念的 durable 真源位置）
    检验：FieldProfile → /world/contexts/*/field；ActivityRuntimeProfile → /agents/*/control/current_activity/profile 等
  更换 G 层（系统升级时）：若从三粒子换回双粒子，此任务中的 S_global = V_A × {V_Bᵢ}，V_C 不再适用
```

**特化任务 ST-G1：合法状态硬不变量验证（对应 G1）**

```
特化内容（v6 架构）：
  ST-G1a：写隔离不变量 → 扫描代码中是否存在 Bᵢ 直接写 /world/ 或 /coupling/ 的路径
  ST-G1b：基线不被污染 → 验证 baseline_attention_policy 只能被 LLM 主动修改，场/活动切换不触及
  ST-G1c：energy ∈ [0,100] 有界不变量 → 所有 energy 更新路径后必须有 clamp
  ST-G1d：通知终局不变量 → 所有通知最终进入 consumed/expired/merged/archived 之一
  ST-G1e：唯一真源不变量 → Global Event Log 是 append-only 且是规范真源
```

**特化任务 ST-G2：单步演化四阶段完整性验证（对应 G2）**

```
特化内容（v6 架构）：
  ST-G2a：验证四阶段（assemble→evolve→emit→apply）完整且有序，无跳过
  ST-G2b：验证无死状态（任何合法状态有确定性的下一步）
  ST-G2c：六问判准检验 → 对任意规则，验证至少能回答 Q1-Q6 之一
  ST-G2d：动作端到端闭环 → 验证 intent→emit→route→R_A^apply→receipt 的完整路径
```

**特化任务 ST-G3~G8：持续稳定性/开放演化/语义耦合/一致性/认知连续性/工程鲁棒性**

```
（逐条对应 G3-R001~G3-R014、G4-R001~R007、G5-R001~R028、G6-R001~R006、G7-R001~R006、G8-R001~R013）
特化内容：将每条 G 层具体要求转化为"在特定场景下的确定性检验题"
  示例 ST-G3-R003：设计场景 → next_wakeup 更新路径测试：
    测试 1：LLM 未给出 next_wakeup → 系统自动填入默认值？
    测试 2：计算出的 next_wakeup ≤ now → 系统是否截断为 now + MIN_WAKEUP_INTERVAL？
    测试 3：系统崩溃重启后 → next_wakeup 从 WritebackLedger 恢复，是否有效？
```

**系统升级时的任务更新策略**：

```
当系统从 v6 升级到 vNext 时：
  不变：通用验证任务 UT-1~UT-4（T+PA 层，与系统无关）
  需更新：特化验证任务 ST-G*（重新从 vNext 的 G 层要求生成）
  检查：vNext 的 G 层是否仍然完整追溯到 T-PA 体系（防止"G 层漂移"问题）
```

## 第八章：评判标准

### 8.1 三级评判定义

| 评级 | 符号 | 定义 |
|------|------|------|
| **通过** | ✅ | 场景中引用的 G 层要求完全满足，推理链每步均有文档引用，证据链三层完整 |
| **部分通过** | ⚠️ | G 层要求主路径满足，但在特定触发条件下出现不满足；必须触发 4.2 的溯源流程 |
| **失败** | ❌ | G 层要求在场景中完全不满足；直接归入根因框架并给出修复方向 |

**重要**：⚠️ 部分通过不是终态，必须完成溯源分析收敛到 ❌ 根因。

### 8.2 判定的三项强制约束

```
约束 1：不依赖文档外隐含规则
  判断只能依据：G 层具体要求的显式文字 + 引用的 PA 公理语义
  禁止：以"按照常识应该这样"为理由判定通过

约束 2：证据链三层完整（G→PA→T）
  判定结论必须附带完整的三层引用，缺少任一层则判定无效

约束 3：⚠️ 必须有溯源分析
  部分通过结论不允许只写"部分通过"而无后续溯源
  溯源分析是评判的组成部分，不是可选项
```

### 8.3 分层通过率目标

| 任务类型 | 任务编号 | 基线（首轮）| 目标（G8 修复后）|
|---------|---------|-----------|-----------------|
| 通用任务（T+PA 层验证）| UT-1~4 | 全量场景通过 | 全量场景通过（否则 T-PA 体系本身有问题）|

## 第九章：根因归并框架

**方法论核心**：将表层症状归并到底层根因，一个底层修复消灭多个症状。

### 9.1 根因命名规范

```
ROOT-A~G：现有7大根因（详见 9.2）
ROOT-H+：沙盘发现的新根因（按需新增）

问题代码命名：
  Ix（文档级静态扫描问题，I01-I18）
  BUG-xx（场景推演发现的逻辑问题，BUG-01~21）
  漏洞x（进阶沙盘发现的工程底层漏洞，漏洞1~21）
  UT-FAIL-x（通用任务失败，触发体系元验证）
```

### 9.3 根因归并的操作步骤

```
Step 1：收集所有 ⚠️/❌ 场景的溯源结论
Step 2：按"PA 层缺失的公理"分组（相同 PA 缺失归同组）
Step 3：同组内按"G 层影响范围"进一步合并（影响同一 G 目标的归同一根因）
Step 4：命名根因（ROOT-X），描述"系统在 PA 层缺失了什么"
Step 5：设计修复方案（修复 PA 落地 = 补全对应 G 层要求 + EP 工程补丁）
Step 6：估算修复收益（每个根因修复能消灭多少症状）
```

## 第十章：完整迭代循环

### 10.1 标准迭代流程（八步）

```
【第 0 步】确认基准文档与 T-PA-G 体系版本
  → 记录：当前 G 层版本（如 v6_sonnet3 + v2.2）
  → 记录：通用任务集（UT-1~4）与特化任务集（ST-G0~G8）的最新版本
  ↓

【第 1 步】文档级静态扫描（先于场景推演）
  → 旧/新字段名并存统计（ROOT-A 的前置检测）
  → Source of Truth Table 验证（G0-R012 的前置检测）
  → 跨文档引用一致性检查
  → 输出：I-编号问题清单

【第 2 步】通用任务验证（UT-1~4，T+PA 层基础）
  → 必须在 G 层特化任务之前执行
  → 若 UT 系列出现 ❌ → 触发"异常信号 1"检查（体系遗漏）
  → 输出：T-PA 层基础健康状态确认
  ↓

【第 3 步】特化场景矩阵生成（覆盖 G0~G8 全量要求）
  → 从 G 层要求直接生成检验场景（每条 G 要求至少对应 1 个 P 类场景 + 1 个 E 类边界场景）
  → 额外添加已知高频失效点的重点场景
  → 输出：场景清单（含场景 ID、覆盖 G 要求编号、预期结论）
  ↓

【第 4 步】逐场景推演（标准格式，含完整证据链）
  → ⚠️ 部分通过：必须完成 4.2 的溯源流程
  → ❌ 失败：记录根因代码、PA 缺失
  → 检测两类异常信号（见第五章）
  → 输出：带证据链的场景推演报告
  ↓

【第 5 步】问题统计与根因归并
  → 按 PA 缺失分组，归并到根因框架
  → 按 G 目标统计通过率
  → 检测"异常信号 2"（孤立的多个部分失败）
  → 输出：根因分析报告（含收益估算）
  ↓

【第 6 步】修复（按 P0→P1→P2 优先级）
  → P0：影响系统完全不可运行（❌ 场景 + ROOT-A/B/C 根因）
  → P1：影响系统稳定性（ROOT-D/E/F 根因）
  → P2：文档清理（ROOT-G 根因）
  → 每个修复对应明确的 G 要求编号和 PA 公理
  ↓

【第 7 步】修复后重验（精准范围，非全量重跑）
  → 必须重验：所有标注了本次修复问题代码的场景 + 所有 ❌ 场景
  → 选择性重验：与修复逻辑相关的 ⚠️ 场景
  → 不需重验：无关的 ✅ 通过场景
  → 输出：通过率变化对比表
  ↓

【第 8 步】判断迭代终止条件
  if (新发现问题 > 阈值 OR 有高优新根因):
    → 返回第 3 步（下一轮场景推演）
  else if (触发了体系异常信号 1 或 2):
    → 进入体系元验证流程（第五章），修正 T-PA-G 体系后重跑
  else:
    → 声明当前版本稳定性状态，记录已知限制，存档本轮推演报告
```

### 10.2 重验的精准判定标准

```
修复有效：
  ① 原 ❌ 场景变为 ✅ 或 ⚠️（症状消除）
  ② 问题代码在新场景中出现次数 = 0（根因消灭）
  ③ 无新出现的 ❌（无副作用）

修复过度（副作用）：
  原来 ✅ 的场景变为 ⚠️ 或 ❌
  → 需回退修复方案或缩小修复范围

修复不足：
  原 ❌ 场景仍为 ❌，或症状代码在新场景中仍出现
  → 检查修复方案是否真正落到了 G 层要求中，而非仅在注释层面描述
```

## 第十一章：可演化系统的六条完备性条件

一个规则集被判定为"足以驱动可演化系统（满足 T1-T5）"，必须同时满足以下六条，每条都对应 T-PA-G 的追溯链：

```
完备性条件 1（事件到达，对应 T5+PA7）：
  所有 arity × co_presence × synchrony 组合均有明确的路由策略和门槛映射
  不存在"可能接收到但无路由规则"的 (field_type, event_type) 组合

完备性条件 2（感知有效，对应 T5+PA7+PA2）：
  effectiveThreshold 计算路径无歧义，所有影响因子均有优先级排序（无并列冲突）
  身份优先义务过滤：task_binding≥strong 时 pending_obligations 全量注入，不被 window 截断

完备性条件 3（中断可控，对应 T3+PA4+PA8）：
  P0-P4 规则优先级明确，所有规则冲突均有确定性解析路径（无"谁说了算"空白）
  生命威胁 Override 机制存在（hazard/evacuation/emergency 可突破 Hysteresis Lock）
  N-tick 超时逃生存在（PROGRESS_ACTIONS 白名单，不被水群欺骗）

完备性条件 4（动作合法，对应 T5+PA9）：
  每类动作均有 (role × field_status × energy × mobility) 四维合法性校验
  LLM 输出路径均有对应的 R_B^apply 层拦截点（防幻觉越权）
  动作端到端闭环：intent→emit→route→apply→receipt 链完整

完备性条件 5（写回完备，对应 T2+PA5+PA6）：
  所有 durable 对象均在 Durable/Ephemeral 分类表中（无灰区）
  WriteEvent-first 原则 + ActionReceipt 状态机（防 Split-Brain）
  PendingIntent 暂存区（防假性遗忘）
  MemoryEntry.source_field_id 域隔离（防 Memory Bleed）
  拓扑排序（Kahn 算法，防因果乱序）

完备性条件 6（调度终局，对应 T3+PA4+PA3）：
  next_wakeup 不变量（不为 null）+ T1-T5 时间公设（防时钟异常）
  N-tick 超时逃生（软超时 N=5 ticks + 强超时 2N=10 ticks）
  overload_mode 完整状态机（含恢复路径，防永久降载黑洞）
  energy 可恢复路径（存在 energy_profile=recovering 的活动预设）
```
