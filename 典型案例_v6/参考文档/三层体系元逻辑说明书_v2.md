# 第一章：为什么要分三层

## 1.1 核心问题

设计多智能体系统时会产生大量要求："agent 的 energy 不能超过 100"、"通知必须有 TTL"、"LLM 不能直接输出完整 ActivityInstance"……

这些要求从何而来？哪些是**不可删减的本质要求**？哪些是**当前实现方案带来的附属要求**？哪些是**工程现实的补丁**？

分不清楚会产生两类错误：

| 错误类型 | 具体表现 | 后果 |
|---|---|---|
| 把工程补丁误当本质要求 | 理论文档臃肿，换一种实现时不知道哪些要求还成立 | 设计失去灵活性 |
| 把本质要求降格为可选项 | 系统看似工作，某些条件下必然失败 | 隐患难以被发现 |

三层体系正是为了解决这个问题。

## 1.2 三层的含义与判据

```
T 层  ──  本质目标层
          ↓
PA 层  ── 抽象公理层
          ↓
G 层  ──  当前系统目标层 + 具体要求
```

**判断一个陈述属于哪一层，看它回答什么问题：**

| 层级 | 回答的问题 | 与实现绑定？ | 对所有多智能体系统成立？ |
|---|---|---|---|
| **T 层** | 系统最终要保什么性质？ | 否 | 是 |
| **PA 层** | 为了保住 T，结构上至少需要什么条件？ | 否 | 是 |
| **G 层** | 在粒子体系中，PA 怎么落地？ | 是 | 否（仅对本系统）|

**判断示例**：

- "系统必须能持续演化" → T 层（本质目标，与任何实现无关）
- "任何可消耗资源必须有恢复路径" → PA 层（结构性条件，不依赖 energy 这个概念）
- "energy ∈ [0, 100]，且存在 recovering 类活动使 Δenergy > 0" → G 层（在粒子体系中的具体落地）

## 1.3 三层之间的关系：抽象 → 公理化 → 具体化

```
T 层    是目标的抽象集合      ← "系统要保什么"
  ↓ 公理化（T 的结构性必要条件）
PA 层   是 T 的结构化约束     ← "为了保住 T，结构上至少需要..."
  ↓ 具体化（在 world/agents/meta 架构中）
G 层    是 PA 在当前粒子      ← "在 world/agents/meta 这个架构里，PA 长什么样"
         体系中的具体化
```

**重要**：PA 层是从 T 推导出来的——每条 PA 都可以证明：若不满足该 PA，则 T 层某个目标在某种条件下必然失效。G 层是 PA 在特定架构约束下的具体化——每个 G 目标都是某条 PA 的实例。若一个 G 要求找不到对应的 PA，说明它是架构引入的附属要求，不应进入理论主干（这是 Gₘ 判准的依据）。

## 1.4 物理类比：守恒律 → 动力学约束 → 具体系统方程

| 本体系层级 | 物理类比 | 含义 |
|---|---|---|
| **T 层** | 守恒律（能量守恒、动量守恒）| 对任何物理系统成立，无关实现 |
| **PA 层** | 李雅普诺夫条件、哈密顿量存在性要求 | 为了保住守恒律，演化算子必须满足的结构性条件 |
| **G 层** | 具体力学方程（$F=ma$、麦克斯韦方程组）| 在特定坐标系和模型下的具体化 |

一个物理学家不会把"$F=ma$ 是一个本质定律"和"能量守恒是一个本质定律"混为一谈。我们的三层体系做同样的事，只是对象是多智能体系统。

# 第二章：T 层 — 本质目标（5 个）

## 2.1 T 层的定义原则

进入 T 层的陈述必须满足三个条件：

1. **普遍性**：对任何可以被称为"多智能体系统"的系统成立
2. **不可绕过性**：如果违反，系统不是"跑得不好"，而是"不再是我们讨论的那类系统"
3. **无关实现**：不依赖 world/agents/meta、FieldProfile 等任何具体架构概念

## 2.2 T 层的内在逻辑：串行依赖链 + 并行约束

```
T1（可定义性）
  ↓ 必须先能被描述，才能讨论其他一切
T2（合法性）
  ↓ 能被描述之后，状态必须始终合法；非法状态使后续演化不可信
T3（持续演化）
  ↓ 合法地存在，还要能永久运行下去
T4（开放演化）
  ↓ 能持续运行，还不能是无意义循环；T3 保证活，T4 保证活得有意义

T5（有效耦合）── 并行约束，在 T1 之后始终有效
  主体必须真实地与环境相互作用，否则只是孤立状态机的集合
```

**T 层依赖总结**：

| 目标 | 前置条件 | 独立维度 |
|---|---|---|
| T1 | 无（是最基础前提）| 存在与可描述性 |
| T2 | T1（先有定义，才能谈合法性）| 任意时刻的状态合法性 |
| T3 | T1+T2（合法存在，才能谈持续）| 时间维度的持续性 |
| T4 | T3（能持续，才能谈有无意义）| 演化的语义有效性 |
| T5 | T1（需要系统被定义才能定义耦合）| 主体与环境的真实交互 |

**T3 与 T4 的关键区别**（最容易混淆）：

> T3 = 系统活着（不在有限时间内死亡）  
> T4 = 系统活得有意义（演化有语义进展，不空转、不僵死）

一个系统可以满足 T3（一直在运行）但违反 T4（永远在几个无意义状态间循环）。

## 2.3 每个 T 目标的深度分析

---

### T1：可定义性

**回答的问题**：系统能不能被描述出来？

**定义**：系统必须能被形式化描述，至少能表达：(1) 可区分的主体与环境；(2) 可描述的状态；(3) 可描述的演化；(4) 可描述的相互作用。

**为什么必要**：如果连系统有什么、它的状态是什么、如何演化都无法表达，那么"运行"、"正确性"、"稳定性"这些概念都失去着落。

**违反后果**：不会得到"一个运行得差的系统"，而是根本没有系统。

**违反案例**：  
一个"多智能体协作系统"用自然语言描述："A 负责写代码，B 负责审查，C 负责测试"。没有定义 agent 是什么、状态是什么、消息如何路由、动作如何影响世界。当试图推演"A 完成代码后通知 B"这个事件时，无法回答：通知从哪里来？经过什么路径？B 何时收到？B 的状态如何变化？整个推演从起点就失去基础，无法进行。这是 T1 缺失。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 必须首先定义相空间（主体/环境的可区分状态）才能建立力学体系 |
| 计算机 | 必须首先定义进程（主体）、内存（环境）、系统调用（接口）、调度器（演化）|
| 芯片 | 必须首先定义寄存器（状态）、ISA（演化规则）、总线（相互作用）|

---

### T2：合法性

**回答的问题**：系统能不能始终保持在有意义的状态？

**定义**：系统在任意时刻都必须处于合法状态域中。包含五个维度：状态值域合法、写边界合法、时间单调、真源唯一、接口完备。

**三维写合法性**（T2 的核心结构）：

```
值域约束：写的【结果】在合法范围内         （如 energy ∈ [0, 100]）
写权唯一：写的【主体】唯一                 （不能多个组件直接修改同一状态）
写边界：写的【影响范围】有界               （一次写操作不应越权影响无关状态分量）
```

**为什么必要**：即使系统在运行，若状态自相矛盾，后续所有演化结果都不可信。T2 失效不是"跑得慢"，而是"跑出来的结果无意义"。

**违反案例**：  
Agent B 具有直接修改 Agent A 的 `energy` 字段的权限（违反写权唯一）。B 在某轮演化中将 A 的 energy 从 20 直接设为 150（超出 [0,100] 合法域）。此后 3 小时内，所有依赖 A 的 energy 的判断（"A 是否需要休息？"、"A 的输出带宽是多少？"、"P0 是否触发？"）都建立在这个非法值之上，产生完全不可预期的行为。这是 T2 违反——不是"A 的行为有点怪"，而是"A 的所有行为从此失去语义基础"。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 守恒律（能量守恒、动量守恒）——合法状态域是满足守恒律的状态集合 |
| 计算机 | 内存安全——进程越界写入其他进程内存，B 的所有状态都不可信 |
| 芯片 | 总线仲裁——多设备同时争写，数据总线上的信号叠加为噪声，所有读取都无意义 |

---

### T3：持续演化

**回答的问题**：系统能不能永远运行下去（在理想条件下）？

**定义**：系统必须能在理想条件下持续无限时间演化，不因内部机制缺陷在有限时间内耗尽、沉默、堵塞或爆炸。

**最小要求的四个方向**：资源不会只耗不补；累积不会只增不减；调度不会中断；演化函数对任意合法状态都能给出下一步（无死状态）。

**为什么必要**：一个在有限时间内必然停机的系统，只是短程过程，不是长期自治系统。

**违反案例**：  
Agent B 的 energy 以 -3/h 的速度消耗（工作状态），理论上存在 recovering 活动（+5/h）。但 B 所在的任务场是 mission_locked，P0 规则在 energy=0 时触发，试图将活动切换到 recovering。然而 mission_locked 的 normative_force=binding 约束阻止了活动切换（recovering 意味着退出当前场）。energy 永远无法回升，B 从此永久沉默。即使有 100 条未读通知，也没有任何机制能唤醒 B。这是 T3 违反——恢复路径"理论存在"但"实际不可达"（PA3 被违反）。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 非平衡定态——系统持续耗散但靠外部输入维持，不趋向热寂 |
| 计算机 | 垃圾回收机制 + 调度器——无 GC 则 OOM；无调度则饥饿 |
| 芯片 | 时钟信号 + 看门狗定时器——没有时钟电路停转；没有 watchdog 死锁后无法恢复 |

---

### T4：开放演化

**回答的问题**：系统的运行有没有语义进展（不是空转）？

**定义**：系统不仅要能持续运行（T3），还要避免在非预期条件下收敛到吸收态、僵死态，或陷入无语义进展的病态循环。

**与 T3 的精确区别**：

| | T3 | T4 |
|---|---|---|
| 系统还在运行 | ✓ | ✓ |
| 演化有语义进展 | 不要求 | ✓ |
| 典型反例 | 完全停机/能量耗尽 | 无限空转/义务循环死锁 |

**"非预期/无意义"的判据**：由系统设计层定义的进展投影 P(s) 来界定。若系统在有限步内 P(s) 没有净变化，且这不是合法的静息期（如正常休眠），则视为非预期吸收态。

**违反案例**：  
Agent A 和 Agent B 处于义务循环死锁（A 等 B 完成义务 X，B 等 A 完成义务 Y，且两者 blocking_on 形成环）。两者的 next_wakeup 正常更新（T3 满足，PA4 满足），每 5 分钟各自演化一次。但每次演化后，A 发现 B 的义务 X 仍未完成，继续等待；B 同理。系统在这两个状态间机械来回，永无进展。所有参与者在"等"，但整个工作项目永远不会推进。这是 T4 违反——没有 PA8 的语义进展公理，系统无法自主逃离这个循环。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 系统陷入非目标极限环——粒子一直在动（T3），但永远无法到达目标区域 |
| 计算机 | 活锁（livelock）——两个进程持续地相互礼让，都没有实质推进 |
| 芯片 | 流水线气泡——时钟一直在转（T3），但大量 NOP 指令导致有效指令吞吐量趋近于零（T4 违反）|

---

### T5：有效耦合

**回答的问题**：主体与环境之间有没有真实的双向交互？

**定义**：主体必须真实地受到环境影响，并真实地影响环境。系统不是若干自循环模块的并列，而是真正发生相互作用的多主体系统。

**最小要求**：合法事件最终能影响相应主体（观察侧）；合法动作最终能改变相应世界（行动侧）；两者缺一不构成真实耦合。

**为什么必要**：若主体与环境之间没有真实耦合，"多智能体—世界系统"只是多个孤立状态机加一个空壳环境。

**违反案例**：  
设计者将所有 agent 的 memory 设计为完全私有（每个 agent 有独立的 world 副本），没有共享的 InteractionContext。Agent A 在自己的 world 副本中"发布"了一条任务分配消息，但这条消息没有任何路由路径到达 Agent B 的 world 副本。B 永远不知道 A 的动作，A 的所有动作对 B 的状态零影响。两者各自独立演化，形式上都在运行（T3 满足），但根本不是在"协作"——这是 T5 违反。

**T5 与 T1 的关系**：T5 依赖 T1（需要系统被定义才能定义耦合的方向性），但 T5 与 T3/T4 在逻辑上独立（可以有真实耦合但系统短暂存在，也可以持续演化但耦合只是名义上的）。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 粒子—场相互作用（牛顿第三定律）——主体既受场影响，也反作用于场 |
| 计算机 | 完整 I/O 系统——CPU 能读取输入（观察侧），也能向输出写入（行动侧）|
| 芯片 | 传感—控制回路——传感器读取环境（观察），执行器修改环境（行动），缺一则开环 |

## 2.4 T 层依赖图

```
T1（可定义性）
  │
  ├──→ T2（合法性）────→ T3（持续演化）────→ T4（开放演化）
  │     依赖 T1            依赖 T1+T2           依赖 T3
  │
  └──→ T5（有效耦合）
        依赖 T1（可区分是定义耦合方向的前提）
        与 T3/T4 在逻辑上独立，但实践中 T3 支撑 T5 的持续有效
```

# 第三章：PA 层 — 抽象公理（9 条）

## 3.1 PA 层的定义原则

进入 PA 层的陈述必须满足：

1. **从 T 可推导**：若不满足该 PA，则 T 层某个目标在某种条件下必然失效
2. **抽象性**：不依赖具体实现（不能包含 world/agents/meta、FieldProfile 等具体概念）
3. **结构性**：描述的是系统必须具备的**结构性条件**，而非目标本身

PA 层是 T 层与 G 层之间的桥梁：比 T 更具体（有结构含义），但比 G 更抽象（与实现无关）。

## 3.2 PA 层的内在分组逻辑

9 条 PA 公理可按功能分为 5 组：

```
定义组（服务 T1 为主）：
  PA1 ── 系统结构存在且可区分（主体/环境/相互作用路径）
  PA5 ── 状态所有权唯一（每个状态分量只有一个权威写者）

边界组（服务 T2 为主）：
  PA2 ── 所有状态量有界（防上溢/爆炸）
  PA6 ── 所有效应有终局（防永久悬挂）

生存组（服务 T3 为主）：
  PA3 ── 资源可恢复（防下溢/单调耗尽）
  PA4 ── 调度持续再进入（防永久沉默）

进展组（服务 T4，独立对应）：
  PA8 ── 语义进展可定义且可逃离（防无意义循环）

耦合组（服务 T5 为主）：
  PA7 ── 信息可达（观察侧：信息到达主体）
  PA9 ── 行动有效（行动侧：行动影响环境）
```

## 3.3 PA 层的互补对结构

**PA 层的 9 条公理不是简单并列，而是形成四对互补关系：**

```
资源稳定互补对：PA2（有界，防爆炸）↔ PA3（可恢复，防耗尽）
  → 两者合起来保证资源处于"有界且有恢复路径"的健康区间
  → PA2 单独不够：有上界但单调下降，最终还是停机
  → PA3 单独不够：有恢复但无上界，最终还是爆炸

效应生命周期互补对：PA6（防永挂）↔ PA7（防永失）
  → PA6 防止"效应永远挂起"（通知、义务等进不了终态）
  → PA7 防止"信息永远到不了"（合法接收者收不到）
  → 两者共同保证效应的"发出—接收—终局"闭环

耦合双向互补对：PA7（观察侧）↔ PA9（行动侧）
  → PA7 保证主体能观察到环境（信息到达）
  → PA9 保证主体能作用于环境（行动有效）
  → 两者共同构成 T5 有效耦合的双向闭环
  → 缺 PA7：主体是盲目执行器（不知道环境状态就在动）
  → 缺 PA9：主体是纯被动接收器（能看到一切但无法改变任何东西）

运行活性互补对：PA4（防永无机会）↔ PA8（防机会全空转）
  → PA4 保证"系统总能被再次触发演化"
  → PA8 保证"被触发的演化产生语义进展"
  → 两者层次不同：PA4 是存在性条件（有没有机会），PA8 是质量条件（机会是否有意义）
  → 缺 PA4：agent 静默，永远不被唤醒
  → 缺 PA8：agent 被不断唤醒，但每次都在空转，没有语义进展
```

## 3.4 PA→T 服务矩阵

> ● = 主要服务（缺失则该 T 直接失效）  
> ○ = 次要服务（缺失对该 T 有影响但非直接）  
> 空 = 无直接关系

```
            T1        T2        T3        T4        T5
           可定义性   合法性    持续演化   开放演化   有效耦合
PA1 存在性  ●          ○         ○          ○         ●
PA2 有界性  ○          ●         ●          ○         ○
PA3 可恢复  ○          ○         ●          ○         ○
PA4 再进入  ○          ○         ●          ○         ●
PA5 写权唯  ●          ●         ○          ○         ○
PA6 效应终  ○          ●         ●          ○         ○
PA7 信息达  ○          ○         ○          ○         ●
PA8 语义进  ○          ○         ○          ●         ○
PA9 观察行  ●          ○         ○          ○         ●
```

**读法说明**：

- PA1 主要服务 T1 和 T5——没有"主体/环境可区分"，系统无法被形式化（T1 失效），耦合方向无法定义（T5 失效）
- PA2 主要服务 T2 和 T3——无界状态破坏合法域（T2），无界增长最终耗尽资源（T3）
- PA8 独立对应 T4——PA8 是 T4 的直接公理化，没有其他公理可以替代
- PA7 和 PA9 共同服务 T5——两者缺一则 T5 不完整

## 3.5 每条 PA 公理的深度分析

---

### PA1：存在性与主体—环境可区分公理

**从 T 推导**：T1 要求系统能被形式化描述，T5 要求有真实耦合。若系统内部不能区分"谁在行动"和"谁被作用"，则 T1 无法成立（无法形式化），T5 也无法成立（无方向性的耦合不是耦合）。→ 因此必须有 PA1。

**数学表达**：

```
∃ 状态空间 S:
  S = S_agent ⊕ S_world ⊕ S_interaction   // 可分解为主体侧/环境侧/相互作用路径
  其中 S_agent ≠ ∅, S_world ≠ ∅

∧ ∀ s ∈ S_legal, ∃ s' ∈ S_legal: transition(s) → s'  // 演化算子对任意合法状态全定义
```

**含义**：PA1 不要求一定写成"世界粒子 A + 智能体粒子 B"。它只要求系统内部存在"谁在感知/行动"和"谁被感知/被作用"的基本区分，且演化函数没有死状态。

**关于工程例外**：若 LLM 或外部服务暂时无法给出下一步（如模型幻觉、网络超时），此类情况不是"合法状态但演化函数无定义"的 PA1 违反——而应由工程层（PA4 保证的持续再进入 + G8 的超时恢复）确保系统最终仍能继续演化。

**若缺失**：无法定义 agent、world、耦合；行为方向性无法定义；整个体系概念失去基础。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 相空间（phase space）的存在性 + 哈密顿演化算子的全定义性 |
| 计算机 | 进程模型（kernel/user space 分离）+ 程序语义的完整性（任何合法输入都有输出）|
| 芯片 | ISA 完整定义 + 地址空间分区（内核/用户分离）|

**违反案例**：某实现将 agent 的状态和 world 的状态混写在同一个 JSON 文件中，没有边界。两个 agent 同时修改这个文件，产生不可预期的合并。无法定义"哪条记录属于哪个 agent 的本体状态"。这是 PA1 违反。

---

### PA2：有界性公理

**从 T 推导**：T2 要求任意时刻处于合法状态域（无界状态必然越界）；T3 要求持续演化（无界增长最终在有限时间内导致停机）。→ 因此必须有 PA2。

**数学表达**：

```
∀ X ∈ { 设计时识别为"可能无界增长"的状态量 }:
  ∃ B_X ∈ ℝ⁺: ∀ t ≥ 0, |X(t)| ≤ B_X

// X 包括（不限于）：
//   资源类：energy, attention_budget, token_budget
//   累积类：memory_size, notification_count, context_count, obligation_count
```

**关键约束**：此处"可能无界增长"的识别发生在**系统设计阶段**——设计者须在设计时识别哪些状态量在理论上可无界增长并提供有界控制；这不要求运行时动态判定。

**PA2 与 PA3 的精确互补**：

```
PA2: 任意可积累状态量 X → ∃ 上界 B_X（防爆炸）
PA3: 任意可消耗资源 R → ∃ 严格正恢复路径（防耗尽）
合起来：资源处于 [R_min, B_R] 内的长时健康区间
```

**若缺失**：资源枯竭、内部积压爆炸、被历史拖死、在有限时间内丧失演化能力。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 有界 Lyapunov 函数（系统能量有上界）|
| 计算机 | 垃圾回收（防内存泄漏）+ 队列深度限制（背压机制）|
| 芯片 | FIFO 深度上界 + DDR 容量限制 + 缓冲区边界保护 |

**违反案例**：某系统每次 agent 发消息都向所有其他 agent 发通知，且无 TTL。1000 轮后，50 个 agent 的系统中每个 agent 有约 50,000 条 pending 通知。assemble 时 context window 溢出，系统无法演化。PA2 违反导致 T3 失效。

---

### PA3：可恢复性公理

**从 T 推导**：T3 要求持续演化。PA2（有界性）保证资源不超过上界，但不保证资源不低于下界。PA3 专门保证"从任意低状态都能恢复"。PA2 + PA3 合起来才能保证 T3 的资源稳定性。→ 因此必须有 PA3。

**数学表达**：

```
∀ R ∈ ConsumableResources（可被消耗的关键资源）:
  ∀ s 满足 R(s) = R_min（资源耗尽态）:
    ∃ path p: s →* s', 且沿 p 有 ΔR > 0（严格正增量）
    ∧ path p 在系统可达轨迹中存在（不只是理论存在，实际可达）
```

**关键约束**："在可达轨迹中存在"——恢复路径不能只在理论上存在，必须在实际的状态转移图中可达。若 mission_locked 约束阻止了进入 recovering 活动，则恢复路径"理论存在"但"实际不可达"，PA3 被违反。

**若缺失**：任何消耗型动作都趋向永久停机；即使有上界（PA2 满足），系统也会在到达下界后卡死。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 热力学中的可逆抽运——从低能态可以通过抽运到高能态 |
| 计算机 | 内存回收路径的存在 + 磁盘 I/O 请求的完成路径 |
| 芯片 | 充电控制器（电池从耗尽态可充电）+ 电压恢复电路 |

**违反案例**：Agent B 所在场是 mission_locked，能量 -3/h 消耗，reaching 活动（+5/h）存在但需要退出场才能进入。energy 到 0 后，P0 规则触发但被 mission_locked 阻止。恢复路径不可达，B 永久沉默。

---

### PA4：持续再进入公理

**从 T 推导**：T3 要求持续演化——系统不能永久失去演化机会；T5 要求有效耦合——agent 必须能响应环境变化（需要被再次触发）。→ 因此必须有 PA4。

**数学表达**：

```
∀ s ∈ S_reachable（任意可达状态）:
  ∃ t' > t_now, ∃ component C: scheduled_evolution(C, t')
  // 在任意可达状态下，总存在某个将被触发演化的组件和时刻
  // 注意：不要求 s 必须发生变化（静息合法），
  //       只要求"有机会演化"不被永久失去
```

**PA4 与 PA8 的精确区别**（最容易混淆的一对）：

```
PA4: ∀ s, ∃ t': scheduled_evolution(t')
     = "系统在有限时间内总会获得演化机会"
     = 只要求"有机会"，不要求"机会有意义"

PA8: if cycle C 中 P(s) 无净变化
       then ∃ escape_mechanism
     = "若系统空转，必须有逃离机制"
     = 比 PA4 更强：要求机会产生语义进展

PA4 满足的情形：agent 合法休眠，next_wakeup=now+8h，时间到了会被唤醒
PA4 违反的情形：agent 的 next_wakeup=null，永远不会被唤醒
PA8 违反的情形：agent 被不断唤醒（PA4 满足），但每次都只是发"我在想..."，无实质进展
```

**若缺失**：系统即使当前状态合法，也可能永远沉默，变成静态系统。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 庞加莱递归定理——有界系统轨迹无限多次回归 |
| 计算机 | 事件循环永不停止 + 调度器总有下一个 tick |
| 芯片 | 时钟信号（clock）永不停止 + 看门狗定时器（watchdog）防止死锁 |

**违反案例**：三个 agent A、B、C 约定：A 等 B 回复后才设置 next_wakeup；B 等 C；C 等 A。初始时，三者的 next_wakeup 都是 null（等待对方）。没有时间驱动的备用触发。系统永久沉默。三方循环等待，没有任何一个会被再次调度。

---

### PA5：写权唯一性公理

**从 T 推导**：T1 要求系统可被形式化描述（若一个状态分量可被多方随意修改，它的值无法被单一地追溯，系统不可被形式化）；T2 要求状态合法（多写者竞争会导致状态污染和矛盾）。→ 因此必须有 PA5。

**数学表达**：

```
∀ sᵢ ∈ S_durable（任意因果可追溯的状态分量）:
  |{ C | can_directly_write(C, sᵢ) }| = 1   // 直接写权限者有且仅有一个

∧ ∀ C' ≠ authoritative_writer(sᵢ):
    C' 对 sᵢ 的影响必须通过:
    C' → emit(event) → route → authoritative_writer(sᵢ).apply(event)
```

**关于"durable"的含义**：在 PA 层的抽象含义为**其变更具有因果可追溯性的状态分量**（即该分量的变化会影响系统的未来演化轨道），而非临时计算缓存。这与 G 层"durable"（持久化存储）的对应关系见 G0-R005。

**若缺失**：耦合边界消失；行为因果不可追踪；审计无法进行；系统变成不可验证的黑箱互写。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 哈密顿量的唯一性——系统演化由唯一的演化算子决定 |
| 计算机 | Rust 的所有权系统（最多一个可变引用）+ 数据库的单主写入原则 |
| 芯片 | 总线仲裁器（每个事务只有一个 master）+ 内存保护单元（MPU）|

**违反案例**：系统中，agent 的 active_key 既可以被 LLM 输出直接覆写，也可以被 ContextClosedEvent 处理逻辑自动重置。两者并发时竞争，结果取决于顺序，且无法追溯哪次写是"对的"。这是 PA5 违反。

---

### PA6：效应有终局公理

**从 T 推导**：T2 要求任意时刻状态合法（无终局的效应使系统状态不确定）；T3 要求持续演化（无终局的效应无限积压，最终耗尽资源）。→ 因此必须有 PA6。

**数学表达**：

```
∀ e ∈ Effects_generated（任意生成的效应对象）:
  ∃ t_terminal ∈ (t_created, +∞): status(e, t_terminal) ∈ TerminalStates(type(e))

// 各类型的终态集合（设计时必须显式定义）：
// Notification:      { consumed, expired, merged, archived }
// Request/Attempt:   { accepted, rejected, failed, cancelled, applied }
// Task/Obligation:   { satisfied, cancelled, expired }
// ContextClose/Event:{ applied, ignored, failed, archived }
```

**PA6 与 PA7 的精确互补**：

```
PA6: 防止"效应永久悬挂"——生成的效应最终必须进入终态
PA7: 防止"信息永远到不了"——合法接收者必须在有效期内收到信息
两者共同保证"效应—接收—终局"的完整闭环
```

**若缺失**：系统被历史效应拖死；无法区分哪些效应已完成；通知/义务无限积压。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 激发态粒子的自发辐射——任何不稳定粒子最终衰变到稳定产物 |
| 计算机 | RAII——资源生命周期有明确终点；GC finalization |
| 芯片 | 握手协议的完成（AXI 事务有 BVALID/BREADY 握手终结）|

**违反案例**：一个"帮助请求"义务被创建后，提出者离开了该场景，义务没有超时机制、没有被显式取消，永远保持 "active" 状态。六个月后，系统中有 50,000 个永远活跃的义务。每次 agent 的 assemble 都要处理这些义务，系统崩溃。这是 PA6 违反。

---

### PA7：信息可达性公理

**从 T 推导**：T5 要求有效耦合，"主体真实受到环境影响"。若合法应接收信息的主体永远接收不到该信息，则环境变化无法影响主体，T5 的观察侧失效。→ 因此必须有 PA7。

**数学表达**：

```
∀ info i, ∀ agent aᵢ where authorized_recipient(aᵢ, i)（合法接收者）:
  ∃ t_delivery ≤ t_expiry(i): received(aᵢ, i, t_delivery)
  // 信息在其有效期内到达合法接收者

// 注意：
// - 不要求"立刻到达"（可以有延迟）
// - 但必须在信息有效期 t_expiry(i) 内到达
// - 对无有效期限制的信息：∃ t_delivery < +∞ 即可（有限时间内总会到达）
```

**若缺失**：主体与环境解耦；重要事件永远无法影响相应 agent；系统退化为各自自循环的状态机。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 电磁信号在有限时间内传播到接收者（光速有限，但一定到达）|
| 计算机 | 可靠消息传递（TCP 的 at-least-once + 超时）|
| 芯片 | 中断到达 CPU 的保证（IRQ 不被无限期屏蔽或丢失）|

**违反案例**：一个紧急通知（semantic_class = hazard）被发往 agent B，告知房间有烟雾警报。但系统的注意力门槛配置错误：hazard 类通知被 attention_policy 过滤（应强制通过）。B 永远收不到通知，继续在危险环境中"正常工作"。PA7 违反（合法接收者永远未收到信息）。

---

### PA8：语义进展公理

**从 T 推导**：T4 要求系统不在非预期条件下陷入无语义进展的病态循环。若没有定义"什么是进展"的机制、没有检测"是否在进展"的手段、没有逃离"无进展循环"的路径，T4 就无法成立。→ 因此必须有 PA8。

**数学表达**：

```
// 前提：系统设计者定义目标相关的进展投影函数
∃ P: States → ProgressValue  // P(s) 是"目标相关的进展度量"

// 公理内容：
若 ∃ cycle C ⊆ reachable_trajectories, ∀ s ∈ C: P(s) = P_constant（循环内 P 无净变化）
则 ∃ escape_mechanism M:
    Prob[system exits C via M] > 0  // 非零概率逃离（非确定性）
  ∧ M 不依赖人工干预（自主触发）
```

**关键澄清——为什么用 Prob[...] > 0 而非确定性逃离**：PA8 不要求"一定逃离"（确定性太强），只要求"逃离的概率非零"（系统不被吸收）。这意味着：在每次进入无进展状态时，系统都有一个非零的机会逃离；由于这个概率始终非零，随着时间推移，系统最终会逃离（Borel-Cantelli 引理的应用）。这是"系统不成为吸收态"的正确数学表达。

**"目标相关"的精确含义**：P(s) 的定义来源于**系统设计或配置层给定的**维度（如任务完成状态、义务状态变化），而非用户在运行时主观认定的"有意义"。不同场类型下 P(s) 的定义不同：强任务场用 PROGRESS_ACTIONS 白名单；弱任务/社交场用更弱但仍可验证的进展标记。

**与 PA4 的精确区别**（再次强调，因为这对最容易混淆）：

```
PA4: 保证系统"有机会演化"（存在性）
PA8: 保证演化"产生语义进展"（质量性）
```

**若缺失**：固定点僵死；周期性空转；无新信息注入后的假活性；不断触发但无任何语义进展。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 系统避免进入非目标吸引子（attractor）——不陷入不期望的极限环 |
| 计算机 | 分布式系统的进展性质（progress property）+ 活锁检测与打破 |
| 芯片 | 流水线向前推进（不只是有时钟，而是每个周期确实执行有意义的指令）+ 死锁检测 |

**违反案例**：5 个 agent 在任务场持续对话：A "我在思考"，B "好的继续"，C "我同意"……循环 200 次，`send_message` 被频繁执行，事件日志充满消息记录。但没有 deliverable 被提交，没有 agenda 被更新，没有 obligation 被满足。`last_progress_at`（进展时间戳）从未更新。无 PA8 的系统中，这个循环永远持续，消耗无限计算资源，产出为零。

### PA9：观察—行动耦合公理

**从 T 推导**：T5 要求有效耦合，包含两个方向：主体受到环境影响（观察侧，由 PA7 保证）+ 主体的行动影响环境（行动侧，由 PA9 保证）。若 agent 只能观察不能行动（或反之），则耦合单向，不构成真实多主体系统。→ 因此必须有 PA9 来保证双向闭环。

**数学表达**：

```
∀ agent aᵢ ∈ Agents:
  // 观察侧
  ∃ ω_relevant ⊆ WorldState: aᵢ can_observe(ω_relevant)

∧ // 行动侧
  ∃ α ∈ Actions(aᵢ):
    execute(aᵢ, α) → net_effect(WorldState) ≠ ∅
    // "净效果"不为空：行动之后，WorldState 至少有一处实质性变化
```

**PA7 与 PA9 的精确分工**：

```
PA7: info → agent（环境 → 主体，观察侧）
PA9: agent → world（主体 → 环境，行动侧）

T5 = PA7 ∧ PA9：双向真实耦合
缺 PA7：主体是盲目执行器
缺 PA9：主体是纯被动接收器
```

**若缺失**：agent 退化为纯被动接收器（只观察）或盲目执行器（只行动）；真实多主体—世界系统不存在。

**跨域类比**：

| 领域 | 类比 |
|---|---|
| 物理 | 牛顿第三定律——粒子既受场影响，也反作用于场 |
| 计算机 | 完整 I/O 能力的进程（不只读，也能写）|
| 芯片 | 双向总线（bidirectional bus）——可发出请求，也可接受响应并产生结果 |

**违反案例**：一个"观察者 agent"被配置为只有 `read_messages` 权限，没有任何写操作权限。该 agent 看到所有消息，"认知"到群聊进展，但无法发出任何有效行动。它自称系统"成员"，但实际对世界状态零影响。PA9 违反（行动侧退化），T5 失效。

# 第四章：G 层 — 当前系统目标

## 4.1 G 层的定义原则

进入 G 层的陈述必须满足：

1. **可追溯性**（Gₘ 判准）：必须能追溯到至少一条 PA 公理，进而追溯到至少一个 T 目标
2. **与当前架构绑定**：可以使用 world/agents/meta、FieldProfile、notification_buffer 等具体概念
3. **具体可验证**：足够具体，使实现者可以判断某个实现是否满足该要求

G 层是 PA 在"异步耦合粒子系统"这个特定架构中的具体化。

## 4.2 G 层每个目标回答什么问题

| G 目标 | 回答的核心问题 | 具体化的 PA |
|---|---|---|
| **Gₘ** | 什么样的要求才能进入 G 层？（设计判准）| 判准（非 PA 具体化）|
| **G0** | 这个系统"是什么"？（本体结构）| PA1, PA5 |
| **G1** | 系统在任意时刻是否合法？（硬不变量）| PA2, PA5, PA6 |
| **G2** | 一次完整演化能跑完吗？（单步完备）| PA1, PA4, PA5 |
| **G3** | 系统能永远跑下去吗？（长时稳定）| PA2, PA3, PA4, PA6 |
| **G4** | 系统演化有语义意义吗？（不空转）| PA8 |
| **G5** | 主体与环境的耦合是有效的吗？（语义耦合）| PA7, PA9 |
| **G6** | 系统的状态变化能被追溯和恢复吗？（审计）| PA5, PA6 |
| **G7** | LLM agent 在中断后能恢复工作上下文吗？（认知连续）| PA2, PA4, PA5（系统特化）|
| **G8** | 在真实计算机环境中，以上目标能稳定成立吗？（工程增强）| PA2–PA9 |
| **Gf** | 未来可以如何进一步完善？（后置优化）| 各 PA 的精化 |

## 4.3 G 层内在依赖链

**核心串行链**（前者是后者的前置依赖）：

```
Gₘ（裁剪元原则）
  ↓ "什么样的要求才能进入 G 层"
G0（存在性与本体结构）
  ↓ "系统能被定义了"
G1（合法状态域 / 硬不变量）
  ↓ "系统在任意时刻都是合法的"
G2（单步演化完备性）
  ↓ "系统能完整地跑一轮"
G3（持续稳定性）
  ↓ "系统能永远跑"
G4（开放演化性）
  ↓ "系统跑的有语义意义"
```

**并行关系**（互不依赖，分别从 G0/G2 推出）：

```
G0 + G2 → G5（语义有效耦合）
G0 + G1 → G6（一致性与可追溯性）
G5 和 G6 是概念上的并行关系，互不依赖
```

**特化关系**：

```
G3 + G6 → G7（认知连续性：以 G3+G6 为前置依赖，是 LLM-based agent 的特化目标）
```

**工程层**：

```
所有 G0-G7 → G8（工程鲁棒性：在现实计算机环境中实现 G0-G7 的增强）
```

**G5 与 G6 为什么是并行关系**（容易混淆）：

| | G5（语义耦合）| G6（可追溯性）|
|---|---|---|
| 解决什么 | 系统**按预期工作**（行为语义正确）| 系统**可被验证**（状态变化可追溯）|
| 反例 | 耦合正确但不可审计 | 高度可审计但耦合逻辑有错误 |
| 前置 | G0, G2 | G0, G1 |

→ 两者解决不同维度的问题，在概念上互不依赖。但在工程实现上会局部互相支撑（如 ActionReceipt、崩溃恢复会增强动作闭环判断）。

**为什么这个顺序是必然的**：

| 层次关系 | 理由 |
|---|---|
| G0 → G1 | 没有系统定义，无法定义"什么是合法状态" |
| G1 → G2 | 没有合法状态的边界，无法定义演化是否"完备" |
| G2 → G3 | 不能完成一步演化，不可能谈无限演化 |
| G3 → G4 | 系统不活着，谈"活得有意义"无从说起 |
| G3 → G7 | 认知连续性需要 memory 有界（G3-R006）和系统持续运行 |
| G6 → G7 | 认知连续性需要 memory 写入可追溯/可恢复 |

## 4.4 G 层完整依赖矩阵

> ● = 强前置依赖（行依赖列；缺列则行失去意义）  
> ↑ = 工程增强（G8 增强 G0–G7，但不是逻辑前置）  
> 空 = 无直接依赖

```
      G0  G1  G2  G3  G4  G5  G6  G7  G8
G0    —
G1    ●   —
G2    ●   ●   —
G3    ●   ●   ●   —
G4    ○   ○   ○   ●   —
G5    ●   ○   ●   ○   ○   —
G6    ●   ●   ○   ○   ○   ○   —
G7    ○   ○   ○   ●   ○   ○   ●   —
G8    ↑   ↑   ↑   ↑   ↑   ↑   ↑   ↑   —
```

**读法说明**：

- G3 的行全是 ●（G3 依赖 G0、G1、G2）——稳定性需要存在性、合法性、单步完备性作为基础
- G4 只依赖 G3——开放演化的前置只需要系统活着（G3），不需要语义耦合（G5）
- G5 依赖 G0 和 G2——语义耦合需要本体定义（G0）和 emit 路由（G2），不依赖 G3/G6
- G6 依赖 G0 和 G1——审计需要本体存在（G0）和写权明确（G1），不依赖 G5
- G7 依赖 G3 和 G6——认知连续性需要 memory 有界稳定（G3）和 memory 写入可追溯（G6）
- G8 是工程增强（↑）——G8 增强 G0–G7 在现实中的鲁棒性，但 G0–G7 不依赖 G8

**G7 的特殊性**：G7 是系统特化目标，以 G3 和 G6 为前置依赖，但同时也直接具体化 PA2（memory 有界）、PA4（中断恢复的再进入机制）、PA5（memory 域隔离）。这意味着 G7 既有 G 层间的前置依赖，又有直接的 PA 层来源。

# 第五章：完整 T→PA→G 推导链

## 5.1 从 T 到 PA：为什么这 9 条公理是必要的

| PA 公理 | 服务的 T | 推导过程（为什么 T 需要这条 PA）|
|---|---|---|
| **PA1** | T1, T5 | T1 要求系统可被形式化 → 必须有可区分的主体/环境结构和全定义演化算子；T5 要求真实耦合 → 必须知道"谁在行动、谁被作用" |
| **PA2** | T2, T3 | T2 要求状态在合法域内 → 无界状态必然越界；T3 要求无限演化 → 无界增长在有限时间内导致停机 |

# 第七章：为什么这样分层，而不是别的分法

## 7.1 为什么需要 PA 层（不直接从 T 到 G）

如果直接从 T 目标推导 G 要求，会遇到三个问题：

**问题1：同一个 G 要求服务多个 T 目标，无法清晰溯源**

例如 G1-R007（energy ∈ [0, 100]）：
- 它服务 T2（合法性，energy 超界是非法状态）
- 它也服务 T3（持续演化，energy 有界是资源稳定的前提）

如果直接从 T 跳到 G，这条要求就挂在两个 T 目标上，追溯路径模糊。但通过 PA2（有界性），可以清晰地说：G1-R007 具体化 PA2，PA2 同时服务 T2 和 T3。PA 层提供了清晰的中间汇聚点。

**问题2：PA 层是完备性检查的"模板"**

如果某条 PA 公理没有对应的 G 要求，说明该 PA 在当前架构中没有被正确实现——这是架构缺口。PA 层充当了"是否覆盖到了所有本质需要"的检验框架。

**问题3：PA 层允许 G 层可替换**

PA 层不绑定实现。如果未来换一套架构（不是 world/agents/meta，而是另一种实现方式），T 层和 PA 层仍然成立，只需要重新推导 G 层。G 层是可替换的，T 和 PA 是稳定的。

## 7.2 为什么 T 层恰好是这 5 个目标

这 5 个目标覆盖了"多主体系统正常运作"的所有必要维度，且互不重叠：

| 目标 | 覆盖的维度 | 若删除 |
|---|---|---|
| T1 | 存在与可描述性 | 系统连定义都没有 |
| T2 | 任意时刻的合法性 | 系统随时可能进入非法状态 |
| T3 | 时间维度的持续性 | 系统在有限时间内停机 |
| T4 | 演化的语义有效性 | 系统"活着"但不做有意义的事 |
| T5 | 主体与环境的耦合 | 系统是孤立状态机，不是多主体系统 |

删去任意一个，都会有某类根本性失败无法被防止。

**T4 的必要性**（最容易被忽视）：

很多系统设计只关注 T1-T3 和 T5，忽略了 T4。但在 LLM-based 多智能体系统中，T4 的失效极其常见：

- agent 不断重复相似输出（假活性）
- 义务循环依赖导致系统卡在"进展中"但实际无进展的状态
- group chat 中消息飞但没有任何 deliverable 产生

T4 正是为了防范这类"形式上活着，实质上已经死亡"的系统状态。没有 PA8 这条公理，任何系统都无法系统性地预防和处理这类失效。

## 7.3 G 层为什么不是 PA 的直接翻译

G 层是 PA 在特定架构中的**具体化**，不是字面翻译。具体化包括：

1. **选择具体的状态量**：PA2 说"有界控制"，G 层选择 energy、memory、notification_buffer 作为具体对象。这是架构决策，不是 PA 的直接推论。

2. **确定具体实现路径**：PA4 说"持续再进入"，G 层确定通过 next_wakeup 时间戳 + 调度器来实现。这是实现决策。

3. **处理多对多映射**：同一个 G 要求可能具体化多条 PA（如 G2-R005a 具体化 PA5 + PA9）；同一条 PA 可能被多个 G 要求具体化（如 PA5 被 G0、G1、G6、G7、G8 的多个要求具体化）。

4. **识别系统特化目标**：G7（认知连续性）是 LLM-based agent 的特化，不是所有多主体系统都需要的。它是 PA2+PA4+PA5 在特定主体类型上的专业化应用，不是新 PA 公理的落地。

# 第八章：三项实用工具

## 8.1 如何审查一个新需求（4步指南）

当遇到一个新的设计需求时，按以下步骤判断它的位置：

```
Step 1：能追溯到 PA 公理吗？
  否 → 不属于 G 层主干（考虑放入应用层配置或业务规则）
  是 → 继续 Step 2

Step 2：是理想条件下的要求，还是工程补强？
  理想条件下成立 → G0–G7（根据追溯到的 PA 选择对应 G）
  只在非理想环境下需要 → G8
  （判断标准：删掉它，G0–G7 的逻辑结构是否坍塌？若不坍塌，则是 G8）

Step 3：能用参数维度取值表达吗？
  是 → 优先通过维度取值（如 cognitive_demand、interrupt_tolerance 的具体值）表达
  否 → 写成 G 层要求，附带 PA 根源标注

Step 4：检查是否已有覆盖
  查对应 G 目标下是否已有等价要求
  若有 → 补充说明而非新增，避免规范重复
  若无 → 新增，格式：[简称] 正文 → PA根源：PAx → 次目标：Gy → 层级
```

**示例1**（可追溯，进 G 层）：

```
候选需求："agent 在 energy < 20 时，next_wakeup 应拉长到 2 倍"
Step 1: 能追溯 PA3（可恢复）——energy 低时给更多恢复时间 → 可追溯
Step 2: 理想条件下也需要这样的策略 → G3
Step 3: 可以用 energy 值 + defaultNextWakeup 函数的参数表达 → 用维度参数化
Step 4: G3-R003 + G5-R012 中有相关覆盖，作为参数补充
结论：这是 G3 的一条策略参数，不是新的独立要求
```

**示例2**（不可追溯，不进 G 层主干）：

```
候选需求："agent 不应在深夜主动发起高优先级通知"
Step 1: 能追溯 PA7（信息可达）? PA7 说信息应该到达，这条说信息不应发出
        能追溯 PA9（行动有效）? PA9 不关心发出的时间
        无法找到直接 PA 支撑 →
结论：这是应用层/业务层的规则，应通过 FieldProfile 参数（如 active_hours 维度）配置
      不应进入 G 层主干
```

## 8.2 六问沙盘框架（推演前置检查）

在进行沙盘推演之前，确认前置条件：

```
前置条件确认：
✅ G0 满足：系统由哪些粒子构成（V_A、V_Bᵢ），存储归属钉死
✅ G1 满足：所有硬不变量（energy 有界、写隔离、通知终局等）
✅ G2 满足：四阶段演化（assemble→evolve→emit→apply）正确
✅ G5 满足：canInject/canWakeup/allowedActions 的门控机制
✅ 活动预设表掌握：每个 preset 对应的 7 维参数
✅ FieldProfile 掌握：每个场的 8 维参数及其对系统行为的影响
```

推演时，每处理一个事件，按以下六问检查：

```
Q1. 这个事件能到达主体吗？
    检查：canInject(e, Bᵢ) 是否为 true？
    依据：effective_external_threshold vs priority(e)
          或 isLocalEvent → effective_local_threshold vs priority(e)

Q2. 到达后会被看到吗？
    检查：通知优先级 >= effective_threshold？
    依据：baseline_attention_policy + current_stress_modifier + fieldExternalThreshold

Q3. 是否会打断当前活动？
    检查：canWakeup(e, Bᵢ) = true？
    依据：wakeup_external_threshold vs semantic_class

Q4. 若响应，输出什么动作？
    检查：allowedActions 中是否包含该动作？
    依据：角色权限 + context.status 过滤 + medium/bandwidth/mobility 约束

Q5. 动作如何写回状态？
    检查：R_B^apply 中的写回路径是什么？
    依据：control_delta → V_Bᵢ；action_event → R_A^apply → V_A

Q6. 写回后，下次何时演化？
    检查：next_wakeup 更新到什么值？
    依据：schedule_update?.next_wakeup || defaultNextWakeup(mode)
    约束：next_wakeup ≠ null（G3-R003），且 > now + MIN_WAKEUP_INTERVAL
```

**注意**：Q1-Q2 对应 PA7（信息可达），Q3 对应 PA4（持续再进入），Q4 对应 PA9（行动有效），Q5 对应 PA5（写权唯一），Q6 对应 PA4（下次再进入时间）。六问覆盖了单步演化完备性（G2）的全部维度。

## 8.3 如何验证系统满足某条 G 要求

以 G3-R003（next_wakeup 永不为 null）为例：

```
验证问题："在任意时刻，任意 agent 的 next_wakeup 是否都是有效时间戳？"

验证路径：
  1. 检查 R_B^apply 中是否有如下逻辑：
     "若 Output 未提供 next_wakeup，则系统自动填入 defaultNextWakeup(mode)"

  2. 检查 defaultNextWakeup 是否有 MIN_WAKEUP_INTERVAL 保护：
     "next_wakeup = max(computed, now + MIN_WAKEUP_INTERVAL_SEC)"
     （这是 G8-R001 busy-loop 防护的工程实现）

  3. 检查系统重启恢复流程（onSystemRestart）是否恢复所有过期的 next_wakeup
     （这是 G8-R002 crash recovery 的工程实现）

若以上任一缺失：
  G3-R003 未满足
    → PA4 被违反
      → T3 存在风险（系统可能在有限时间内停止演化）
```

# 第十章：总结

## 整个体系的一句话精髓

> **这套体系的核心思想是：把"为什么"和"是什么"分开。**  
> T 层回答"为什么"（系统最终要保什么），PA 层回答"结构上最少需要什么"，G 层回答"在这个具体架构里是什么"。  
> 任何具体要求都应该是这条链的末端——找得到它存在的理由，找得到它保护的本质目标。

## 三层体系的六项核心价值

| 价值 | 没有这套体系时的问题 | 有了这套体系后 |
|---|---|---|
| **裁剪能力** | 不知道哪条要求可以删除 | Gₘ 判准：找不到 PA 追溯的要求，不进主干 |
| **完备性验证** | 不知道是否遗漏了什么 | PA→G 映射：每条 PA 都应在 G 层有具体化 |
| **层级分离** | 理想要求和工程补丁混写 | G0-G7 是理想层，G8 是工程层，明确区分 |
| **可移植性** | 换了架构，不知道哪些要求还成立 | T 和 PA 不与架构绑定，可以重新推导 G 层 |
| **风险评估** | 不知道缺失某条要求的后果 | 每条要求可追溯到 T，知道违反的严重程度 |
| **解释能力** | 要求无法向外部解释清楚 | 任何要求都有完整的 T→PA→G 推导链 |
