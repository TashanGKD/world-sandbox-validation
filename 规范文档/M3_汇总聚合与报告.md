## 第一章：汇总工作的总体结构

### 1.1 输入与输出

```
输入（来自 M2）：
  大量场景记录文件（record_id）
  每条记录含：局部判定（P1-P5）、证据等级（E0-E3）、失败追溯链

输出（本文档产生）：
  表1 任务覆盖总表
  表2 G 层满足性矩阵
  表3 PA 层满足性矩阵
  表4 T 层满足性矩阵
  表5 关键失败总表
  表6 根因追溯表（RC-编号）
  表7 体系缺口审查表（GAP-编号，简版，深度分析见 M4）
  [辅]表8 场景分布统计表
  [辅]表9 证据等级统计表
  最终总报告摘要
```

### 1.2 分层聚合原则（禁止违反）

```
允许的聚合路径：
  G 层记录 → G 层汇总矩阵
  G 层汇总 + GEN 任务 → PA 层汇总矩阵
  PA 层汇总 → T 层汇总矩阵

禁止的操作：
  ❌ 从若干随机场景直接跳判 T 层
  ❌ 用大量 P1 稀释 P3（P3 必须先下钻到 P4）
  ❌ 把 P5 当作 P1（P5 = 证据不足，不是没问题）
  ❌ 在关键任务未覆盖的情况下给出 P1 总结论
```

### 1.3 五条核心聚合规则

| 规则 | 内容 | 违反后果 |
|------|------|---------|
| **规则 A：P4 优先** | 若存在 P4 结论（证据≥E1，非单一偶发，可定位到明确 G/PA/机制），则该对象主导判定不高于 P2 | 严重错误：忽略根本性失败 |
| **规则 B：P3 不被稀释** | 即使大量 P1，P3 也不能被平均掉；必须先下钻到 P4 根因 | 严重错误：掩盖结构性失败 |
| **规则 C：P5 非中性** | P5 是"证据不足"；大量 P5 的对象不得给出 P1 最终结论 | 严重错误：虚假的"完全满足" |
| **规则 D：关键任务缺失不给 P1** | 验证映射表（M1）标定为关键的任务簇未覆盖，最多只能给 P2 或 P5 | 严重错误：覆盖不足时虚报满足 |
| **规则 E：系统失败与体系缺口分表** | "v6 设计问题"与"T-PA-G 体系遗漏"必须分开记录，不能混为一条结论 | 错误：修复方向错乱 |

## 第二章：聚合流程（八步标准工作流）

```
第一步：检查任务覆盖（先填表1，再进行任何汇总）
  ↓ 确认关键任务已完成，不在覆盖不足时给出虚假结论
  
第二步：整理局部判定输入
  ↓ 从所有正式记录抽出"target_id + 判定 + 证据等级"，形成输入表
  
第三步：G 层汇总（填表2）
  ↓ G 层最贴近具体机制，先汇总 G，再往上聚合
  
第四步：PA 层汇总（填表3）
  ↓ 从 G 层结论 + GEN 类通用任务，聚合到 PA 层
  ↓ 注意：P3 必须收敛后才能聚合；P5 由覆盖不足引起时不等同"没问题"

第五步：T 层汇总（填表4）
  ↓ 从 PA 层稳定结论上收到 T 层
  ↓ 此时不再直接看零散场景，主要看哪些 PA 失守
  
第六步：形成关键失败总表与根因追溯表（填表5 + 表6）
  ↓ 先确定哪些失败最关键，再展开根因链（RC-编号）

第七步：体系缺口审查（填表7，必要时转 M4）
  ↓ 只在"现有链条解释不了"时进入 GAP 表
  ↓ 满足 M4 启动条件时，移交 M4 深度分析
  
第八步：生成总报告摘要
```

## 第三章：主导判定生成流程

每个验证对象（target_id）的主导判定按以下流程生成：

```
Step 1  收集同一 target_id 的全部局部判定（来自所有相关记录）
Step 2  去掉无效/重复/无正式记录的条目
Step 3  标记关键任务是否已覆盖（M1 标定的）
Step 4  检查是否存在高证据 P4（证据≥E1，非偶发，可复现）
  → 若是：主导判定至少为 P3，通常直接给 P4，向上聚合时不高于 P2
Step 5  检查是否存在可复现 P3（稳定条件下可重复出现）
  → 若是：不得被大量 P1 稀释，必须标注"需继续下钻"
Step 6  检查 P5 是否由覆盖不足引起
  → 若是：主导判定不得为 P1（必须补场景）
Step 7  生成主导判定
Step 8  附带生成"聚合理由摘要"（一句话说明为何给出该判定）
```

## 第九章：表6 根因追溯表（RC-编号）

> **用途**：把失败从"现象"追到"要求—公理—目标—本质机制"。

### 9.1 根因链的七个节点（必须完整）

| 节点 | 内容 | 不能缺少的原因 |
|------|------|-------------|
| 1. **失败现象** | 发生了什么（可观测的系统行为）| 追溯起点 |
| 2. **直接失效 G 要求** | 哪条 G 首先没成立（G编号-R编号）| 追溯第一层 |
| 3. **上位 PA 公理** | 对应哪条 PA 失守（PA编号）| 追溯第二层 |
| 4. **T 目标受损** | 最终伤害了哪条 T（T编号）| 追溯第三层 |
| 5. **机制根因** | 系统中哪个本质机制缺失/冲突/漂移 | 修复方向 |
| 6. **证据节点** | 支持这条链的关键记录编号 | 可验证性 |
| 7. **终止理由** | 为何认为已追到"本质层"，不再继续 | 防止无限追溯 |

### 9.2 机制根因标准词汇表

使用以下词汇描述机制根因，不用自造临时表述：

```
双真源（Two Sources of Truth）
非法跨边界写（Cross-Boundary Write Violation）
无界积压（Unbounded Accumulation）
恢复路径不可达（Recovery Path Unreachable）
再进入链断裂（Re-entry Chain Broken）
effect 无终局（Effect Without Terminal State）
进展投影缺失（Progress Projection Missing）
无净进展循环无逃逸（Cycle Without Net Progress and No Escape）
信息不可达（Information Unreachable）
动作不生效（Action Without World Effect）
工程异常无防护（Engineering Anomaly Without Guard）
记忆域混乱（Memory Domain Pollution / Memory Bleed）
```

```markdown
# 表6：根因追溯表

| rc_id | 失败现象 | 触发场景 | 直接失效 G | 对应 PA | 受损 T | 机制根因 | 证据记录 | 失败程度 | 可复现 | 终止理由 | 后续动作 |
|-------|---------|---------|-----------|--------|-------|---------|---------|---------|------|---------|---------|
| RC-001 | WritebackLedger 被当作第二真源使用，状态崩溃重启后不一致 | SYS-G-TRACE-014-SCN-02-REC | G6-R001（durable 变更必须进 Event Log）| PA5 | T1,T2 | 双真源 | EVD-014 | D4 根本性 | 高 | 已定位到 Event Log 与 WritebackLedger 角色边界缺失 | FP0 修复：明确 WritebackLedger 只是 Event Log 的细粒度展开，不是独立真源 |
| RC-003 | public_many 场中，agent 发送无效消息后 last_progress_at 被刷新，N-tick 超时失效 | GEN-PA-PROGRESS-007-SCN-03-REC | G4-R001（PROGRESS_ACTIONS 白名单）| PA8 | T4 | 进展投影缺失（白名单定义不精确）| EVD-007 | D2 条件性 | 是（public_many+strong-binding 场必现）| 白名单定义存在，但未明确排除普通消息 | FP1 修复：在白名单中显式声明"普通消息不算进展" |
```

### 9.4 每条根因链的叙述块（建议附在表格之后）

```markdown
### RC-001 叙述
- 现象：系统崩溃重启后，某些 durable 状态从 WritebackLedger 重建，与 Global Event Log 中的记录不一致
- 首先违反：G6-R001（所有 durable 变更必须产生可重放事件记录，写入 Global Event Log）
- 对应公理：PA5（写权唯一——WritebackLedger 不是 durable 状态的权威写者，Event Log 才是）
- 受损目标：T1（可定义性——系统状态不可追溯，形式化描述失效）、T2（合法性——重建后状态不可信）
- 本质机制问题：双真源（WritebackLedger 被误用为独立真源，与 Event Log 分裂）
- 证据：EVD-014（场景推演轨迹表明：崩溃前 WritebackLedger 有 X 条事件，Event Log 有 X+3 条，重建后丢失 3 条状态变更）
- 为什么不再继续追溯：已定位到最底层机制缺失（Event Log 唯一真源原则未落地），且此条是独立根因，不依赖更深层缺陷
```

## 第十章：表7 体系缺口审查表（GAP-编号，简版）

> **用途**：把"系统问题"和"体系问题"正式分开。本表是简版，完整的缺口分析协议见 M4。

### 10.1 进入此表的准入条件

满足以下任一条，才进入此表：

1. 存在明确失败，但无法映射到现有 T/PA/G（现有链条解释不了）
2. 同一场景同时损害两个高层目标，但现有 PA 图上无连接路径
3. 大量 P3 长期无法收敛到更底层硬缺陷
4. 只能不断新增孤立 G 补丁才能解释同一类问题（补丁膨胀）
5. 不同任务对同一公理给出稳定冲突而体系无优先级规则

```markdown
|--------|---------|-----------|-------------------|---------|-------------|-----------|---------|------------|

```

### 10.3 缺口类型枚举

```
无法归类型缺口：有失败，但现有 T/PA/G 无对应条目
双重失效无桥接型缺口：两 T 同时受损，PA 图上无连接
概率性不满足无法收敛型缺口：大量 P3，下钻后仍无法收束到 P4
要求层补丁膨胀型缺口：不断加 G 层补丁，无法上收到 PA
互斥结论型缺口：不同任务对同一公理给出稳定冲突
```

### 10.4 修正类型枚举

```
增补适用域（说明原条目在此边界不适用）
增补中间桥接公理（两条 PA/T 之间缺少连接）
增补判定条件（原条目定义不足以判定该场景）
上收若干 G 为 PA（若干 G 层补丁有共性，上收为 PA 公理）
拆分原有 PA（原 PA 承载了两类独立约束，需拆开）
明确优先级/适用边界（两条公理在边界条件下冲突）
```
