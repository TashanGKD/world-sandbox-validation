# M1：验证映射表

> **文档编号**：M1  
> **版本**：v2.0（v1.0 升级版）  
> **日期**：2026-03-13  
> **核心变化**：新增 MAP 任务类型映射；新增 TERMINAL 任务域（PA6 专属，统一收口 effect 终局性）；执行顺序升级为 EXEC-0/1/2 三批次  
> **定位**：规划阶段使用。回答：每条 T/PA/G 应该用什么类型的任务验证、靠什么证据支撑、失败后应往哪里追溯。  
> **前置要求**：已读 M0（了解三层体系与术语定义）。  
> **读完本文档你将知道**：任务规划的完整依据，以及任何失败场景的第一追溯方向。  
> **本文档不负责**：任务的具体设计（见 M2）、结论的聚合（见 M3）、任务库起始清单（见 M6）。

## 第一章：映射体系总览

### 1.1 映射的三个方向

| 方向 | 用途 | 流向 |
|------|------|------|
| **自上而下** | 从 T/PA/G 条目推导应该设计哪类任务 | T → PA → G → 任务族 → 场景 → 证据 |
| **自下而上** | 从失败场景追溯到根因所在层 | 场景失败 → G 失效 → PA 失守 → T 受损 |
| **旁向审查** | 当失败无法被现有链条解释时，进入体系缺口检查 | 失败 → 无归因 → M4 |

### 1.2 总体映射矩阵

| 层级 | 条目类型 | 核心问题 | 首选任务类型 | 首选证据等级 | 常见失败信号 | 失败后首查 |
|------|---------|---------|------------|------------|------------|---------|
| **T** | 本质目标 | 系统最终是否保住该目标 | GEN、HYB | E2–E3（多任务聚合）| 多条 PA/G 同时失效 | 对应 PA 集 |
| **PA** | 抽象公理 | 结构上是否具备该公理 | GEN、HYB | E2（不同任务域重复出现）| 结构性失败、概率性失效 | 对应 G 与机制 |
| **G** | 当前体系要求 | 当前实现是否落实该要求 | SYS、HYB | E1–E2（对象/时序/状态级）| 明确违规、缺字段、断链 | 具体对象/规则 |
| **映射层** | GEN→G 桥接 | 抽象判据到 G 层的映射是否完备无歧义 | **MAP** | E2（需同时检查 GEN+SYS）| GEN 成立但 G 无对应；G 存在但无 GEN 依据 | 层间桥接缺失、概念漂移 |

**MAP 任务的专属定位**：MAP 失败不等于 GEN 失败，也不等于 SYS 失败。MAP 失败意味着抽象判据存在但落地映射不完整——这类问题最容易导致"说的和做的不是一件事"。

### 1.3 证据使用原则

- **T 层**：不宜只靠单个局部场景判定，应依赖多条 PA/G 的聚合结论（E2-E3）
- **PA 层**：既看抽象逻辑，也看多个不同任务族上的重复表现（E2）
- **G 层**：可由更直接的对象级、时序级、状态机级场景支撑（E1–E2）

## 第二章：T 层验证映射

### T1 可定义性目标

| 项目 | 内容 |
|------|------|
| **验证问题** | 系统是否存在清晰的对象边界、状态空间、真源、演化路径与作用接口 |
| **首选任务类型** | GEN（合法性）、GEN（真源审计）、SYS（本体结构）|
| **首选任务域** | `LEGAL`、`TRACE`、`BOUND` |
| **核心证据** | 状态空间定义、真源表（Source of Truth Table）、粒子/对象结构、合法演化链 |
| **典型失败信号** | 主体/环境不可区分；真源不明（双真源）；关键对象漂浮在体系外；旧新结构并存导致状态不可判定 |
| **向下落实接口** | → PA1（存在性）、PA5（写权唯一）→ G0、G6 |
| **向上聚合逻辑** | 若 PA1 或 PA5 长期失效，则 T1 受损 |
| **备注** | T1 更适合"结构审计 + 代表性场景"共同判定，而非纯动态场景 |

**T1 推荐任务簇**：

```
GEN-PA-LEGAL-xxx  主体—环境区分审计
GEN-PA-LEGAL-xxx  状态空间分层审计（S_global / S_durable / S_runtime_total）
SYS-G-TRACE-xxx   真源唯一性审计（Source of Truth Table 验证）
SYS-G-TRACE-xxx   演化四阶段闭环审计（assemble→evolve→emit→apply）
SYS-G-LEGAL-xxx   跨粒子边界写入禁令审计
```

---

### T2 合法性目标

| 项目 | 内容 |
|------|------|
| **验证问题** | 是否存在越界值、越权写、双真源、非法 effect 状态、时间倒流、接口旁路 |
| **首选任务类型** | GEN（合法性）、SYS（硬不变量）、SYS（追溯性）|
| **首选任务域** | `LEGAL`、`TRACE`、`BOUND` |
| **核心证据** | 不变量检查记录、状态快照、写回路径审计、真源表、effect 生命周期轨迹 |
| **典型失败信号** | baseline_attention_policy 被场/活动污染；energy 越界（未 clamp）；直接跨粒子写；pending 通知永不终局；runtime 结构被误当 durable |
| **向下落实接口** | → PA2（有界）、PA5（写权唯一）、PA6（效应终局）→ G1、G6 |
| **向上聚合逻辑** | 任一硬不变量持续失效，T2 至少降为 P2 |
| **备注** | T2 是最适合"单步推演 + 不变量检查"验证的目标层条目 |

**T2 推荐任务簇**：

```
GEN-PA-LEGAL-xxx  写隔离审计（跨粒子写禁令）
GEN-PA-LEGAL-xxx  基线不变量审计（baseline_attention_policy 不被污染）
GEN-PA-BOUND-xxx  能量/资源有界审计（energy clamp、sleep_debt 上限）
GEN-PA-LEGAL-xxx  effect 终态审计（通知/义务/context 有终局）
SYS-G-TRACE-xxx   Event Log 真源边界审计（WritebackLedger 不冒充真源）
```

---

### T3 持续演化目标

| 项目 | 内容 |
|------|------|
| **验证问题** | 是否存在再进入链、恢复路径、有界控制、终局控制、无死状态 |
| **首选任务类型** | GEN（有界性、恢复性、调度）、SYS（长时运行）、SYS（工程异常）|
| **首选任务域** | `BOUND`、`RECOVER`、`LONGRUN`、`ENGINEER` |
| **核心证据** | next_wakeup 链完整性、energy 恢复轨迹、队列/记忆/义务有界性、长时场景运行轨迹 |
| **典型失败信号** | 无唤醒（next_wakeup 丢失）；无恢复路径（energy=0 后永久卡死）；消息/通知积压无界；context 只增不减；系统重放后卡死 |
| **向下落实接口** | → PA2、PA3、PA4、PA6 → G2、G3、G8 |
| **向上聚合逻辑** | 若再进入、恢复、有界、终局四者中任一长期缺失，T3 至少降级 |
| **备注** | T3 必须看长时行为；单个短场景不足以给出完全结论 |

**T3 推荐任务簇**：

```
GEN-PA-RECOVER-xxx  调度不断链（next_wakeup 非空、MIN_WAKEUP_INTERVAL）
GEN-PA-RECOVER-xxx  energy 恢复路径（energy=0 后能否到达 recovering 活动）
GEN-PA-BOUND-xxx    长时积压（notification/memory/obligation 是否有界）
SYS-G-BOUND-xxx     context 生命周期（bounded/ephemeral context 能否正常关闭）
SYS-G-ENGINEER-xxx  工程故障恢复（crash→replay→next_wakeup 重建）
```

---

### T4 开放演化目标

| 项目 | 内容 |
|------|------|
| **验证问题** | 是否定义了进展投影 P(s)；是否存在无净进展循环；是否有不依赖人工干预的逃离机制 |
| **首选任务类型** | GEN（进展性）、GEN（活锁）、SYS（进展追踪）、HYB（逃逸验证）|
| **首选任务域** | `PROGRESS`、`LONGRUN` |
| **核心证据** | PROGRESS_ACTIONS 白名单定义、last_progress_at 追踪记录、状态循环轨迹、N-tick 逃离触发证据 |
| **典型失败信号** | 形式上持续活跃但无义务/任务/世界效果净变化；recovering/working 微震荡；永远 blocked 但不退出；普通消息刷新 last_progress_at（水群欺骗）|
| **向下落实接口** | → PA8（语义进展）→ G4，部分依赖 G3/G8 |
| **向上聚合逻辑** | 若无法区分"真推进"和"空转"，T4 不能判为 P1 |
| **备注** | 关键不是"系统有没有动"，而是"动是否形成目标相关净变化" |

**T4 推荐任务簇**：

```
GEN-PA-PROGRESS-xxx  progress projection 定义验证（P(s) 是否被明确定义）
SYS-G-PROGRESS-xxx   PROGRESS_ACTIONS 白名单边界（哪些算进展，哪些不算）
GEN-PA-PROGRESS-xxx  空转循环检测（反复演化但 P(s) 无净变化）
SYS-G-PROGRESS-xxx   N-tick 软超时逃离（5 ticks 无进展→系统注入逃离信号）
SYS-G-PROGRESS-xxx   N-tick 强超时保障（10 ticks 无进展→强制退出）
GEN-PA-PROGRESS-xxx  activity 微震荡（energy ∈ [CRITICAL, SAFE) 振荡）
```

---

### T5 有效耦合目标

| 项目 | 内容 |
|------|------|
| **验证问题** | 有权信息是否在有效期内可达；动作是否真正产生世界效果；两者是否形成闭环 |
| **首选任务类型** | GEN（耦合性）、SYS（Field×Activity）、HYB（闭环验证）|
| **首选任务域** | `COUPLE`、`HAZARD`、`TRACE` |
| **核心证据** | local/external 分流路由记录、canInject/canWakeup 轨迹、effectiveActions 计算、world-side apply 记录、ActionReceipt |
| **典型失败信号** | 有权消息到不了主体；动作只停留在 agent 自述，world 无可观测效果；物理场（low-head 场景）被完全忽略；hazard 类事件无强制唤醒通道 |
| **向下落实接口** | → PA7（信息可达）、PA9（行动有效）→ G5，部分依赖 G2/G6/G8 |
| **向上聚合逻辑** | 输入侧（PA7）和输出侧（PA9）必须都成立；缺一不构成真实耦合 |
| **备注** | 极适合用"同一场景的双向闭环任务"验证 |

**T5 推荐任务簇**：

```
GEN-PA-COUPLE-xxx   信息可达（有权 agent 是否在有效期内收到信息）
SYS-G-COUPLE-xxx    local/external 分流（本场 vs 非本场的注入门槛差异）
GEN-PA-COUPLE-xxx   动作世界生效（动作提议 → emit → apply → receipt 闭环）
SYS-G-HAZARD-xxx    hazard 强制唤醒（hazard/evacuation/emergency 绕过门槛）
SYS-G-COUPLE-xxx    低头族与双场耦合（物理场点名不被数字前景场屏蔽）
HYB-FULL-COUPLE-xxx 双向闭环（同一场景内：看见世界事件 + 行动改变世界）
```

## 第三章：PA 层验证映射

### PA1 存在性与主体—环境可区分公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 系统是否具备最小本体结构与演化定义；是否存在"合法状态但演化函数无定义"的情况 |
| **首选任务域** | `LEGAL`、`BOUND` |
| **核心证据** | 粒子/对象定义完整性、状态空间图（S_global/S_durable/S_runtime_total）、四阶段演化链记录 |
| **典型失败信号** | 缺少关键对象（如 C 粒子的 V_C 未定义）；某合法状态下无法继续推演（演化函数不全定义）；状态空间层级混淆 |
| **首查 G 接口** | G0、G2 |
| **失败根因提示** | 若对象、状态或演化任一缺失，多半是 PA1 失败而非纯局部 bug |

---

### PA2 有界性公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 系统是否识别并控制所有可无界增长的状态量（资源、积累、队列、context 集合等）|
| **首选任务域** | `BOUND`、`LONGRUN` |
| **核心证据** | 队列上限、TTL 定义、分层记忆与 compaction 记录、bounded context 计数 |
| **典型失败信号** | memory 无界增长；notification 无 TTL；context 只增不减；fanout 无上限导致 O(n²) 爆炸 |
| **首查 G 接口** | G1、G3、G8 |
| **失败根因提示** | 概率性爆炸通常说明 PA2 没有真正落实，而不是"参数调小点就行" |

---

### PA3 可恢复性公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 是否能从任意资源耗尽态重新回到可演化态 |
| **首选任务域** | `RECOVER`、`ENGINEER` |
| **核心证据** | recovering 活动存在且可达、sleeping/resting 正增益计算、恢复后的 next_wakeup 链 |
| **典型失败信号** | energy 归零后永久卡死（Hysteresis Lock 无双条件解锁）；恢复态名义存在但永远不可达；恢复与工作之间微震荡（能量净零）|
| **首查 G 接口** | G3、G5、G8 |
| **失败根因提示** | "有恢复路径但永远到不了"也应视为 PA3 未满足 |

---

### PA4 持续再进入公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 任意可达状态下，是否总有 time-driven / event-driven / internal scheduling 之一保证再进入 |
| **首选任务域** | `RECOVER`、`ENGINEER`、`LONGRUN` |
| **核心证据** | next_wakeup 非空验证、pending_wakeups 追踪、事件触发链记录、N-tick 逃生触发记录 |
| **典型失败信号** | 系统静默（next_wakeup 丢失或 busy-loop）；event 到达但无合法入口；等待链断裂 |
| **首查 G 接口** | G2、G3、G8 |
| **失败根因提示** | PA4 不要求本轮一定进展，但要求不能永久失去机会 |

---

### PA5 写权唯一性公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 是否存在双真源、旁路写、未定义共享写、View 持有新信息等写权混乱 |
| **首选任务域** | `LEGAL`、`TRACE` |
| **核心证据** | 真源表（Source of Truth Table）、写路径审计、apply 边界检查、Event Log 与 Ledger 关系审计 |
| **典型失败信号** | 直接改别的粒子 durable 状态；WritebackLedger 变第二真源（应只是 Event Log 的细粒度展开）；旧新字段并存导致真源不明 |
| **首查 G 接口** | G0、G1、G6 |
| **失败根因提示** | PA5 是最常见的"表面功能正常、长期一定漂移"的根因之一 |

---

### PA6 效应有终局公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 是否存在统一的 effect 生命周期与终态定义；所有 effect 是否可观测终局 |
| **首选任务域** | **`TERMINAL`（专属，优先）**、`BOUND`、`LEGAL`、`TRACE` |
| **核心证据** | TTL 字段存在且有效、终态集（consumed/expired/merged/archived/satisfied/cancelled）、close path 完整 |
| **典型失败信号** | pending 通知永不结束（无 TTL）；bounded context 无法关闭；ActionReceipt 无失败终态定义；obligation/lease/staging 无终局路径 |
| **首查 G 接口** | G1、G3、G6、G8 |
| **失败根因提示** | 若 effect 没有终局，不仅伤害合法性（T2），也会拖垮持续演化（T3）|

> **TERMINAL 域说明**：PA6 过去分散在 BOUND/LEGAL/TRACE 三个域中验证，导致终局性验证碎片化、覆盖不系统。`TERMINAL` 是专门为 PA6 设立的统一任务域，覆盖所有 effect 类型的终局验证：
> - `notification`：consumed/expired/merged/archived
> - `obligation`：done/cancelled/expired（7态全覆盖）
> - `bounded context`：closed（含 agenda/obligation 全部满足）
> - `resource lease`：released/expired
> - `staging intent (PendingIntent)`：committed/abandoned
> - `retry queue`：成功/耗尽/放弃（有上限）
> - `关闭广播事件`：acknowledged/ignored/failed/archived

---

### PA7 信息可达性公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 路由、门槛、过滤、有效期与权限，是否共同保证"该到的人能到" |
| **首选任务域** | `COUPLE`、`HAZARD` |
| **核心证据** | delivery chain 追踪、注入门槛（canInject）计算轨迹、唤醒门槛（canWakeup）记录、订阅/成员关系验证 |
| **典型失败信号** | 有权限但信息不到；background 门槛过高吞掉关键信号；hazard 无法绕过 locked 场的外部门槛 |
| **首查 G 接口** | G2、G5、G8 |
| **失败根因提示** | 输入不到不一定是"耦合弱"，可能先是 PA7 路由链失效 |

---

### PA8 语义进展公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 是否能判定"推进/空转"；是否存在自动逃离活锁/吸收态机制（不依赖人工干预）|
| **首选任务域** | `PROGRESS`、`LONGRUN` |
| **核心证据** | P(s) 定义文本、last_progress_at 追踪记录、timeout breaker 触发证据、状态循环轨迹 |
| **典型失败信号** | 系统一直活跃但无任务/义务/世界效果净变化；无限 blocked 循环；普通消息刷新进展计数器 |
| **首查 G 接口** | G4、G8 |
| **失败根因提示** | 若只能依赖人工干预逃离，则 PA8 未满足 |

---

### PA9 观察—行动耦合公理

| 项目 | 内容 |
|------|------|
| **验证问题** | 输入侧（观察环境）和输出侧（行动改变环境）是否同时成立，并形成闭环 |
| **首选任务域** | `COUPLE`、`TRACE` |
| **核心证据** | observation path（assemble 可见内容）、allowed actions 计算、world-side R_A^apply 执行记录、ActionReceipt 回执 |
| **典型失败信号** | 只能看不能改（action 无世界效果）；动作成功只停留在 agent 自述（无 receipt）；改变了世界但 agent 无法感知结果 |
| **首查 G 接口** | G2、G5、G6 |
| **失败根因提示** | PA9 的验证必须至少包含一次"看见世界 + 改变世界"的完整闭环场景 |

## 第四章：G 层验证映射

### 4.1 G 层总览表

| G 编号 | 名称 | 直接服务 | 首选任务域 | 首选证据 | 典型失败信号 |
|--------|------|---------|-----------|---------|------------|
| **G0** | 存在性与本体结构 | PA1、PA5 / T1 | `LEGAL`、`TRACE` | 真源表、目录/对象结构、状态分层 | 对象漂浮、层级混淆、旧新结构并存 |
| **G1** | 合法状态域（硬不变量）| PA2、PA5、PA6 / T2 | `LEGAL`、`BOUND` | 不变量检查、状态快照 | 越界、越权写、双真源、无终局 |
| **G2** | 单步演化完备性 | PA1、PA4、PA5 / T3 | `LEGAL`、`COUPLE` | assemble→evolve→emit→apply 轨迹 | 缺阶段、无 route、world effect 不落地 |
| **G3** | 持续稳定性 | PA2、PA3、PA4、PA6 / T3 | `BOUND`、`RECOVER`、`LONGRUN` | 长时轨迹、next_wakeup 链、恢复曲线 | 积压、断链、无恢复、context 不关闭 |
| **G4** | 开放演化性 | PA8 / T4 | `PROGRESS`、`LONGRUN` | progress projection、循环轨迹 | 空转、活锁、无逃逸 |
| **G5** | 语义有效耦合 | PA7、PA9 / T5 | `COUPLE`、`HAZARD` | delivery、canInject/canWakeup、effective actions、receipt | 信息不到、动作不生效、低头族悖论 |
| **G6** | 一致性与可追溯性 | PA5、PA6 / T1、T2 | `TRACE`、`ENGINEER` | Event Log、receipt、重放一致性 | 无可重放记录、Ledger 冒充真源 |
| **G7** | 认知连续性（LLM 特化）| PA2、PA4、PA5 / T3 | `BOUND`、`RECOVER` | durable memory、resume token、跨轮连续性 | 重启失忆、恢复后断片、全量注入导致 OOM |
| **G8** | 工程鲁棒性 | PA2–PA9 工程层 / T3 | `ENGINEER`、`HAZARD` | 重试、幂等、延迟、脑裂、灾害演化 | 掉线后写坏、重复投递、延迟破坏闭环 |

### 4.2 G 层逐条验证提示

#### G0 存在性与本体结构

**首问**：对象是否"存在且归属清楚"？

**必测**：

- 三粒子（A/C/B）最小骨架是否完整（V_A / V_C / V_Bᵢ）
- `S_global / S_durable / S_runtime_total` 是否区分，且定义无歧义
- Source of Truth Table 是否闭合（每个关键概念只有一个 durable 真源）
- 旧结构是否被明确禁止（`foreground_field_id` 单值、裸 `activity_profile`、`current_object_ref` 顶层等）
- ActivityRuntimeProfile 是否只作为 current_activity.profile 的内部变量块，不独立成顶层本体

**失败首查**：PA1（对象不存在）或 PA5（真源混乱）

---

#### G1 合法状态域

**首问**：有没有"任一步后立刻非法"的情况？

**必测**：

- 写隔离（Bᵢ 不能直接写 /world/ 或 /coupling/ 或其他 Bⱼ）
- baseline_attention_policy 不被场进入/活动切换自动修改
- energy ∈ [0, 100] 的 clamp 不变量
- ForegroundStack 单前景场假设（单栈顶）
- **所有 effect 类型必须有终局**（TERMINAL 域覆盖点）：notification/obligation/lease/staging/retry 均有 TTL 或终态路径
- Global Event Log 是 append-only 且是规范真源（WritebackLedger 不取代）

**失败首查**：T2 直接受损

**配套任务域**：LEGAL（合法性主域）+ **TERMINAL**（effect 终局检查）

---

#### G2 单步演化完备性

**首问**：单步演化链条是否闭合？

**必测**：

- assemble：输入是否合法（memory 分层、通知过滤、义务全量注入条件）
- evolve：LLM 输出是否产生合法提议（只能输出 activity preset 名称，不能直接输出 ActivityInstance JSON）
- emit：是否只产事件不越权写（通过 Event Log 路由）
- apply：是否写回自身并触发可观测的后续效果（R_A^apply 执行、R_C^apply 路由）
- 是否存在 ActionReceipt 语义（动作成功/失败的回执）

**失败首查**：PA1（演化断裂）、PA4（无再进入）、PA5（越权写）、PA9（行动不生效）

---

#### G3 持续稳定性

**首问**：系统能否一直跑，不被积压、断链、耗尽拖死？

**必测**：

- next_wakeup 链：是否任意时刻非空，是否有 MIN_WAKEUP_INTERVAL = 60s 保护
- energy 恢复：从任意 energy=0 状态，是否可达 recovering 活动（净恢复路径）
- Hysteresis Lock 双条件解锁（energy ≥ SAFE_ENERGY=30 OR 连续 ≥4h 保持 energy ≥ CRITICAL=15）
- memory / notification / obligation 有界性（TTL、分层、compaction 均存在）
- bounded/ephemeral context 能否关闭（closeCondition 函数存在且有意义）
- **效应终局完整性**（TERMINAL 域覆盖点）：所有 effect 类型均有终局路径，无永久挂起

**失败首查**：PA2（无界）、PA3（恢复不可达）、PA4（再进入断链）、PA6（effect 无终局）

**配套任务域**：BOUND（有界）+ RECOVER（恢复）+ **TERMINAL**（效应终局）+ LONGRUN（长时稳定）

---

#### G4 开放演化性

**首问**：系统是不是在"假活性"？

**必测**：

- PROGRESS_ACTIONS 白名单是否定义（只有特定动作才能刷新 last_progress_at）
- 白名单是否与场类型匹配（task_binding=strong 场使用严格版，弱任务场使用宽松版）
- 普通消息是否不算进展（防水群欺骗）
- N-tick 软超时（5 ticks 无进展→注入 urgent 逃离信号）是否存在且有效
- N-tick 强超时（10 ticks→强制退出，returnable=false）是否存在
- Obligation 循环检测（blocking_on 形成环时系统能否检测并触发仲裁）

**失败首查**：PA8（进展投影缺失或逃离机制缺失）

---

#### G5 语义有效耦合

**首问**：系统是不是一个真实的"主体—环境耦合系统"？

**必测**：

- Field 与 Activity 分开建模，行为由 field_profile × activity_profile × agent_state 三者共同决定
- local/external 阈值分开（前景场内事件 vs 外部事件的不同注入门槛）
- effective_actions = (field 允许 ∪ 物理场允许) ∩ activity 约束 ∩ energy 约束 ∩ mobility 约束 ∩ role 约束
- 低头族悖论修复（co_located 场的物理点名不被数字前景场屏蔽）
- hazard/evacuation/emergency 语义类别存在强制唤醒通道（Life Threat Override）
- cognitive_demand 驱动 fieldEnergyModifier（v6 新增，消除 v5 二义性）

**失败首查**：PA7（信息可达断裂）、PA9（行动不生效）

---

#### G6 一致性与可追溯性

**首问**：状态是否可重建、行为是否可审计？

**必测**：

- 所有 durable 变更必须先写 WriteEvent（含 idempotency_key）再物化状态（WriteEvent-first 原则）
- ActionReceipt 确认后才提交 durable 写回（Split-Brain 防护）
- LLM 等待 ActionReceipt 期间，推理意图已暂存 PendingIntent（防假性遗忘）
- replay 时 WriteEvent 应用顺序由 causal_parent_ids（逻辑时钟）决定，不由 wall clock 决定（Kahn 拓扑排序）
- 崩溃恢复从 Global Event Log replay，不依赖内存快照

**失败首查**：PA5（写权混乱）、PA6（effect 无终局）

---

#### G7 认知连续性

**首问**：LLM 型 agent 是否拥有跨轮连续性？

**必测**：

- memory 是 durable 真状态（不能随时丢弃）
- MemoryEntry.source_field_id 域隔离（系统自动注入，LLM 不覆盖，防 Memory Bleed）
- assembleMemory：同源优先；跨场记忆须 cross_context_relevance ≥ 0.85，且条数 ≤ 5
- ForegroundStack pop 时的 Stale Frame 校验（跳过已关闭的场）
- ForegroundStack pop 时的 Location-Stack 一致性校验（co_located 场的物理位置对齐）
- resume_token：pushForeground 时系统生成，popForeground 时验证（TTL=24h）

**失败首查**：PA2（记忆无界）、PA4（中断后无法恢复再进入）、PA5（记忆域污染）

---

#### G8 工程鲁棒性

**首问**：非理想条件下（掉线/延迟/并发/灾害）系统是否仍可收敛，不写坏状态？

**必测**：

- busy-loop 防护：next_wakeup ≤ now 时不忙等（MIN_WAKEUP_INTERVAL = 60s）
- 时钟回退保护：Δt = max(0, now - last_evolved_at)（公设 T1）
- 极端 Δt 截断：MAX_DELTA_T_HOURS = 72（公设 T2）
- crash recovery：从 Global Event Log replay，重启后 next_wakeup 加 jitter 重置
- Split-Brain：StagingLedger 暂存 + ActionReceipt 状态机（success/failed/unknown/duplicate/out_of_order）
- ResourceLease：token fence + Sweeper 定期清理
- overload_mode：normal/degraded/shed 三级 + 有恢复状态机（不能只有降级没有恢复）
- 并发保护：AgendaItem 乐观锁（热点场景升级为 queued 模式）、Obligation CAS、ForegroundStack per-agent 单写者

**失败首查**：通常不是新增高层目标，而是对 G2–G6 的工程层支撑不足

## 第六章：典型失败模式与上收路径

### 6.1 典型失败模式表

| 失败模式 | 表面现象 | 首查要求 | 上收公理 | 受损目标 |
|---------|---------|---------|---------|---------|
| 双真源 | 两个位置都被当作权威状态 | G0、G1、G6 | PA5 | T1、T2 |
| 基线污染 | baseline_attention_policy 被场/活动永久改写 | G1 | PA5 | T2、T3 |
| 挂起 effect | notification/obligation 永不终局 | G1、G3 | PA6 | T2、T3 |
| 唤醒链断裂 | next_wakeup 丢失或永远不触发 | G3 | PA4 | T3 |
| 恢复不可达 | 有 recovering 活动但永远到不了 | G3 | PA3 | T3 |
| 语义空转 | 反复演化但无净进展（P(s) 无净变化）| G4 | PA8 | T4 |
| 耦合失真 | 有权信息不到、动作不生效 | G5 | PA7/PA9 | T5 |
| 动作无 receipt | agent 以为成功，但 world 无确认 | G2、G6 | PA9、PA5 | T5、T2 |
| 掉线后写坏 | 重试/重放破坏 durable 状态 | G8、G6 | PA5、PA6 | T2、T3 |
| fanout 爆炸 | 大规模广播拖垮系统 | G3、G8 | PA2、PA6 | T3 |
| Memory Bleed | 跨场记忆污染（吃火锅写进技术文档）| G7 | PA5 | T3 |
| activity 微震荡 | energy 在阈值附近反复切换 consuming/recovering | G3、G8 | PA3 | T3、T4 |

### 6.2 上收路径标准格式

```
失败现象
  → 直接失效的 G 要求（G编号-R编号）
    → 所映射的 PA 公理（PA 编号）
      → 所服务的 T 目标（T 编号）
```

只有当这条链走不通时，才进入 M4（公理体系元验证）。

## 第七章：执行顺序建议（EXEC 批次）

> **EXEC-0/1/2 是执行批次（先跑哪批），P1-P5 是判定等级（满足性结论），两套体系不可混用。**

```
【EXEC-0：基础批次（必须最先跑）】
  任务类型：GEN 全部 + MAP 核心映射任务
  验证目标：T1/T2 的结构性约束（真源/写权/终局/进展定义）
  验证 G 目标：G0（本体结构）→ G1（硬不变量）→ G6（真源账本）
  原因：若这批失败，后续所有 SYS/LEGACY 任务的判定结论都可能失真
  执行顺序：GEN-DEF-* → GEN-LEGAL-* → GEN-TERMINAL-* → GEN-PROGRESS-* → MAP-* 核心

【EXEC-1：主干批次（EXEC-0 通过后）】
  任务类型：SYS 核心机制任务（Context/ForegroundStack/Notification/Obligation/WriteEvent/hazard）
  验证 G 目标：G2（单步演化）→ G3（持续稳定）→ G5（语义耦合）→ G4（开放演化）→ G7（认知连续）
  任务域：RECOVER + TERMINAL + COUPLE + PROGRESS + TRACE
  原因：验证当前体系主干机制的 G 层落地

【EXEC-2：收尾批次（EXEC-1 通过或部分通过后）】
  任务类型：LEGACY 320 场景 + SYS 工程鲁棒性任务
  验证 G 目标：G8（工程鲁棒）+ 历史场景 T-PA-G 重新标注
  任务域：ENGINEER + HAZARD + LONGRUN
  原因：把历史证据挂入新框架；验证非理想条件下系统是否仍可收敛

【体系缺口审查（条件触发，不是固定批次）】
  触发条件：任意批次出现"失败无法归因"或"孤立多重失败"
  行动：启动 M4 协议
```

**EXEC 批次 × G 目标覆盖矩阵**：

| EXEC 批次 | 主要 G 目标 | 主要 PA 公理 | 主要任务类型 |
|----------|-----------|------------|-----------|
| EXEC-0 | G0、G1、G6 | PA1、PA5、PA6（终局定义）| GEN（全部）、MAP（核心）|
| EXEC-1 | G2、G3、G4、G5、G7 | PA2、PA3、PA4、PA6（落地）、PA7、PA8、PA9 | SYS（核心）、MAP（剩余）|
| EXEC-2 | G8 + 历史覆盖 | PA2–PA6（工程层）| SYS（G8）、LEGACY（全部）|

## 第八章：MAP 任务规划专章

### 8.1 MAP 任务的规划原则

MAP 任务回答："这条 GEN 抽象判据，在当前体系的 G 层中，对应的是哪条（哪些）具体要求？映射是否完备？"

**规划步骤**：

```
Step 1：对每条 GEN 任务，找出其在 G 层中的对应要求
  GEN-DEF-001（对象集合可区分）→ G0-R001~R015（存在性与本体结构）
  GEN-LEGAL-009（durable 真源唯一）→ G1-R012（唯一真源不变量）
  GEN-TERMINAL-033（notification 必有终局）→ G1-R010（通知必须有终局）
  GEN-PROGRESS-041（P(s) 可定义）→ G4-R001（有效进展定义）

Step 2：检查是否存在 GEN 判据无对应 G 要求（遗漏）
  → 记录为 MAP 失败信号（体系遗漏候选）

Step 3：检查是否存在 G 要求无对应 GEN 判据（孤立）
  → 该 G 要求可能是 v6 特化（正常），或应当补充 GEN 判据

Step 4：对每对 (GEN, G) 配对，设计一个 MAP 任务验证映射完备性
  命名格式：MAP-[SEQ]
```

### 8.2 核心 MAP 任务列表（从任务库 v2 提炼）

| MAP 任务 | 被验证的 GEN→G 映射 | 典型失败信号 |
|---------|-----------------|------------|
| MAP-001 | 对象-层级映射完备（PA1/PA5→G0/G1）| 同名对象在不同层有歧义 |
| MAP-002 | 状态字段→真源对象映射（PA5→G1）| 字段有双真源或无真源 |
| MAP-003 | effect 类别→终局状态映射（PA6→G1/G3）| 某类 effect 无对应终态定义 |
| MAP-007 | 进展类任务→PA8/G4 映射（PA8→G4）| 进展定义在 G 层没有落地 |
| MAP-008 | 有界性任务→PA2/G3 映射（PA2→G3）| 某类状态量无上界控制 |
| MAP-009 | 写合法性任务→T2/PA5/G1 映射（PA5→G1）| 写权限控制在 G 层不完整 |
| MAP-025 | timeout breaker 映射完备（PA8→G4）| N-tick 逃生在 G 层落地不完整 |
| MAP-026 | hysteresis lock 映射完备（PA3→G3/G4）| 迟滞锁双条件解锁在 G 层缺失 |
| MAP-027 | hazard 强制唤醒映射完备（PA7→G5/G8）| Life Threat Override 在 G 层无对应 |
