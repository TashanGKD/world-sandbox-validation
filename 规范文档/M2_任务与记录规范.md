# M2：任务与记录规范

> **文档编号**：M2  
> **版本**：v2.0（v1.0 升级版）  
> **日期**：2026-03-13  
> **核心变化**：新增 MAP（桥接）和 LEGACY（继承）任务类型；新增 TERMINAL 任务域；引入执行批次（EXEC-0/1/2）与判定等级（P1–P5）的明确区分；统一命名规范  
> **定位**：执行阶段使用。回答：任务怎么设计、场景怎么记录、单场景结论怎么判定。  
> **前置要求**：已读 M0（术语定义）+ M1（已产出任务规划单）。  
> **读完本文档你将知道**：如何设计一个可追溯、可复验的标准任务；如何记录一次场景推演并给出有证据支撑的判定。  
> **本文档不负责**：跨任务的结论聚合（见 M3）、体系缺口分析（见 M4）、任务库起始清单（见 M6）。

## ⚠️ 关键术语区分（必读）

本文档使用两套独立的"P 编号"，含义完全不同，**绝对不能混用**：

| 编号 | 含义 | 使用场景 |
|------|------|---------|
| **EXEC-0 / EXEC-1 / EXEC-2** | 执行批次（任务执行优先级，先跑哪批）| 任务设计时标注，指导执行顺序 |
| **P1 / P2 / P3 / P4 / P5** | 判定等级（满足性结论，该要求是否成立）| 场景记录时填写，表达结论 |

**助记**：EXEC = 执行顺序；P = 满足性判断（Pass/Partial/Prob-fail/Fail/Pending）

## 第一章：核心对象定义

| 对象 | 定义 | 说明 |
|------|------|------|
| **任务（Task）** | 为验证特定性质而设计的抽象测试单元 | 包含验证目标、场景骨架、判定规则 |
| **场景（Scenario）** | 任务在某组具体初始条件下的一次实例化 | 包含具体数值、事件序列 |
| **记录（Record）** | 对一个场景完整的证据化推演 | 逐步推导、判定、根因追溯 |
| **判定（Verdict）** | P1–P5 的满足性结论，附证据等级 E0–E3 | 必须有证据，不能主观断言 |

**对象关系**：

```
一个任务（Task）
  ├─ 多个场景（Scenario）
  │     └─ 一条记录（Record）
  │           ├─ 逐步推演表
  │           ├─ 判定（G/PA/T 各层）
  │           └─ 失败追溯（若有）
  └─ 汇总参与 M3 聚合
```

## 第二章：任务类型

### 2.1 五类任务类型

| 类型 | 含义 | 主要验证层 | 版本升级时 | 判定目的 |
|------|------|----------|-----------|---------|
| **GEN** | 通用任务：去掉当前体系字段名后仍成立 | T、PA | ✅ 无需重写 | 抽象判据本身是否成立 |
| **MAP** | 桥接任务：验证 GEN→SYS 的映射链是否完备 | T/PA → G | ✅ 大部分可复用 | 抽象判据到当前体系的映射是否完备、无歧义 |
| **SYS** | 体系绑定任务：依赖 v6 的具体 schema 和字段 | G | ❌ 需随体系更新 | 当前体系的具体实现是否满足要求 |
| **HYB** | 混合任务：表面依赖当前体系，判定落在 T/PA | T+G、PA+G | ⚠️ 通用部分可复用 | 同时验证 T/PA + G |
| **LEGACY** | 继承任务：将已有历史场景回收并挂入 T-PA-G 分类 | G（主）、PA | ❌ 场景内容不变，分类随体系更新 | 历史证据在新体系框架下的满足性追溯 |

**四类任务的逻辑分工**：

```
GEN（抽象判据）
    ↓ MAP 验证映射完备性
SYS（具体实现）
    ↑ HYB 跨层验证
LEGACY（历史证据）→ 挂入 SYS 类任务的 G 层分类
```

#### GEN 任务的 9 个子类（判据维度）

GEN 任务按验证的 PA 公理维度细分，每个子类对应一组核心抽象判据：

| 子类 | 核心 PA | 典型判据举例 |
|------|--------|------------|
| **定义性** | PA1 | 对象集合可区分、演化算子全定义、主体/环境边界不混淆 |
| **合法性** | PA5 | durable 真源唯一、跨粒子无旁路写、写边界不越权、时间单调不回退 |
| **有界性** | PA2 | 通知数量有界、前景栈深度有界、广播扇出有界、日志增长可截断 |
| **恢复性** | PA3、PA4 | 能量耗尽后可恢复、sleep_debt 可偿还、被中断后存在返回路径 |
| **终局性** | PA6 | notification/obligation/bounded context/lease/staging 记录必有终局 |
| **进展性** | PA8 | P(s) 可定义、消息增长≠进展、空转循环可识别、任务卡死可逃离 |
| **耦合性** | PA7、PA9 | 有权消息可达、动作能改变世界、感知与作用构成闭环、低头场景本地点名仍可达 |
| **跨层一致性** | PA5、PA6、PA8 | 概率性失败能向下钻、场景失败能向上收敛、同一失败不双真源解释 |
| **体系校验** | Gm | 失败现象可归类到现有体系、无法归类时触发缺口审查、任务库跨体系可迁移 |

#### MAP 任务的职责说明

MAP 任务不是 GEN，也不是 SYS，它专门回答：**"GEN 说抽象判据 X 要成立，在当前 v6 体系中，X 对应的是哪个 G 要求？这个映射是否完备、无歧义？"**

| MAP 失败类型 | 含义 | 与 GEN/SYS 的区别 |
|-------------|------|----------------|
| GEN 失败 | 抽象判据本身有问题 | 与体系无关，普遍性缺陷 |
| MAP 失败 | 抽象判据到 G 层的映射缺失或有歧义 | GEN 可能是对的，但体系落地不完整 |
| SYS 失败 | 体系的具体实现不满足 G 要求 | MAP 映射可能是对的，但实现有缺陷 |

#### LEGACY 任务的使用场景

当已有历史场景（如此前 320 个沙盘场景）需要纳入新框架时：

1. **不要重新设计场景**，保留原始初始状态、事件序列、推演过程
2. **补充 T-PA-G 分类标注**，找出历史场景验证的 G 要求、对应 PA 公理、服务 T 目标
3. **补充判定等级**（P1–P5）和证据等级（E0–E3）
4. **标注历史问题代码**（如 I01、BUG-03 等）

### 2.2 十个任务域

| 域代码 | 主要验证性质 | 对应 PA | 典型 G 目标 | GEN 子类 |
|--------|------------|--------|-----------|---------|
| `LEGAL` | 合法性、写权唯一、值域边界 | PA5、PA2 | G0、G1、G6 | 定义性、合法性 |
| `BOUND` | 有界性（资源/队列/记忆/义务积累）| PA2 | G3、G1 | 有界性 |
| `RECOVER` | 可恢复性（关键资源恢复路径）| PA3、PA4 | G3、G5 | 恢复性 |
| **`TERMINAL`** | **终局性（所有 effect 类型必须有终态）** | **PA6** | **G1、G3** | **终局性** |
| `PROGRESS` | 语义进展性（非空转、逃逸机制）| PA8 | G4 | 进展性 |
| `COUPLE` | 有效耦合（观察+行动双向闭环）| PA7、PA9 | G5、G2 | 耦合性 |
| `TRACE` | 一致性与可追溯性 | PA5、PA6 | G6 | 跨层一致性 |
| `ENGINEER` | 工程鲁棒性（故障/延迟/并发/重试）| PA2–PA6 | G8 | — |
| `HAZARD` | 极端物理/灾害事件的强制处理 | PA7 | G5 | 耦合性 |
| `LONGRUN` | 长时演化稳定性 | PA2、PA3、PA8 | G3、G4 | — |

> **TERMINAL 域新增说明**：PA6（效应有终局公理）是 T3 持续演化的核心保障之一，过去分散在 LEGAL/BOUND/TRACE 三个域中，导致终局性验证缺乏系统覆盖。**TERMINAL 域统一收口所有 effect 终态验证**（通知、义务、bounded context、lease、staging 记录、重试流程等），单独成域。

## 第三章：执行批次（EXEC）

> **重要**：执行批次（EXEC）与判定等级（P）是完全独立的两套体系。EXEC 指导执行顺序，P 表达满足性结论。

### 3.1 三个执行批次

| 批次 | 代码 | 执行任务类型 | 优先运行原因 | 对应 CSV 优先级 |
|------|------|------------|------------|--------------|
| **基础批次** | EXEC-0 | GEN（P0 任务）+ MAP（P0 任务）| 先确认体系本身、映射关系与真源/终局/进展判据无根本断裂；这些失败会伪化后续所有结论 | P0（CSV 中 GEN/MAP 行）|
| **主干批次** | EXEC-1 | SYS（P0 任务）+ GEN/MAP（P1 任务）| 验证当前体系主干机制：Context、ForegroundStack、Notification、Obligation、WriteEvent-first、route/apply、hazard、replay | P0（CSV 中 SYS 行）+ P1（GEN/MAP）|
| **收尾批次** | EXEC-2 | LEGACY（320 场景）+ 其余 SYS（P1）| 把旧场景挂入 T-PA-G 分类；验证历史证据是否仍符合新体系要求 | P1（SYS）+ LEGACY |

### 3.2 执行批次的判断规则

```
一个任务进入 EXEC-0 的条件（满足任一）：
  □ 是 GEN 任务且验证 PA1/PA5 相关性质（真源/写权唯一/接口完备）
  □ 是 MAP 任务且验证核心对象映射（effect 映射、真源映射、进展映射）
  □ 其失败会导致后续大量任务判定失真

一个任务进入 EXEC-1 的条件（满足任一）：
  □ 是 SYS 任务且验证 G0-G6 主干机制
  □ 是 GEN/MAP 任务但验证的是非核心路径（耦合性、进展性的进阶场景）

一个任务进入 EXEC-2 的条件：
  □ 是 LEGACY 任务（历史场景回收）
  □ 是 SYS 任务但验证 G7/G8（认知连续性、工程鲁棒性）
  □ 其失败不影响核心路径的可信性
```

## 第五章：判定等级与证据等级

### 5.1 满足性判定（P1–P5）

> **再次强调**：以下 P1–P5 是**判定等级**（满足性结论），与执行批次 EXEC-0/1/2 是完全不同的两套体系。

| 等级 | 含义 | 处理规则 | 能否作为最终结论 |
|------|------|---------|---------------|
| **P1** | 完全满足 | 归档 | ✅ 需 ≥E1 证据 |
| **P2** | 部分满足 | 标注薄弱点后归档 | ⚠️ 需标注"待完善" |
| **P3** | 概率性不满足 | **必须继续下钻到 P4** | ❌ 不是终点 |
| **P4** | 完全不满足 | 立即生成 RC- 根因链 | ✅ 需 ≥E1 证据 |
| **P5** | 暂不可判定 | 补充场景后重判 | ❌ P5 ≠ 没问题 |

**核心规则（禁止违反）**：

```
规则 A（P4 优先）：若存在 P4 结论（证据≥E1，非单一偶发），则该对象主导判定不高于 P2
规则 B（P3 不被稀释）：即使大量 P1，P3 也不能被平均掉；必须先下钻到 P4 根因
规则 C（P5 非中性）：P5 是"证据不足"；大量 P5 的对象不得给出 P1 最终结论
规则 D（关键任务缺失不给 P1）：关键任务簇未覆盖，最多只能给 P2 或 P5
```

### 5.2 证据等级（E0–E3）

| 等级 | 含义 | 适用 |
|------|------|------|
| **E0** 文本级 | 只有规范文本与逻辑推演，无状态演化证据 | 探索性分析，不作为正式归档依据 |
| **E1** 场景级 | 有明确前提、事件序列与状态变化（手工推演）| G 层单条结论 |
| **E2** 轨迹级 | 有逐步演化轨迹、状态快照、分支对照或形式表格 | PA 层结论 |
| **E3** 复验级 | 多场景/多参数/多次复演下结论稳定一致 | T 层最终结论；启动体系缺口审查 |

## 第六章：场景设计维度矩阵

### 6.1 GEN 和 MAP 任务的场景维度（通用层）

```
资源状态轴（对应 PA2/PA3）：
  energy ∈ {0, 1-14(低于CRITICAL=15), 15-29(低于SAFE=30), 30-100(正常)}
  sleep_debt ∈ {0, 1-8h, 9-24h(满载)}
  notification_buffer ∈ {空, 少量, 大量(触发 overload), 极大量(触发 shed)}

再进入状态轴（对应 PA4）：
  next_wakeup 状态 ∈ {正常, 丢失, =now(busy-loop 边界), 极远未来}
  event_trigger ∈ {time-driven, event-driven, internal scheduling}

效应状态轴（对应 PA6/TERMINAL 域）：
  pending_effects ∈ {空, 有 TTL 即将到期, 无 TTL 永久挂起(触发失败场景)}
  effect_types ∈ {notification, obligation, bounded_context, lease, staging_intent, retry_queue}

进展状态轴（对应 PA8）：
  last_progress_at 状态 ∈ {新鲜, 超过 N=5 ticks 阈值, 超过 2N=10 ticks 阈值}
```

### 6.2 SYS 任务的场景维度（v6 体系特化）

```
Field 维度（9维参数）：
  arity × synchrony × lifecycle × task_binding × cognitive_demand
  = 4 × 3 × 3 × 4 × 4 = 576 组合（代表性抽样）

  关键边界值：
    task_binding = mission_locked（最强任务绑定）
    cognitive_demand = intensive（最高认知负荷，-4.0/h）
    lifecycle = ephemeral AND co_presence = co_located（时空崩塌矛盾）

Activity 维度（7维参数）：
  mode × energy_profile × interrupt_tolerance = 12 × 4 × 4 = 192 组合

Agent 状态轴：
  foreground_stack.depth ∈ {0(空栈), 1, 2-5, 6-20(满栈 MAX_STACK_DEPTH)}
  obligation_count ∈ {0, 1, 2-5, 5+(多义务，测 assemble 截断)}
  energy_recovery_lock ∈ {true(迟滞锁激活), false}
```

### 6.3 LEGACY 任务的场景回收规则

```
LEGACY 任务不重新设计场景，而是对已有场景进行回收标注：

必须补充的字段：
  □ T 映射：该场景验证的 T 目标（T1-T5，可多个）
  □ PA 映射：对应的 PA 公理（PA1-PA9，可多个）
  □ G 映射：对应的 G 层要求（G0-G8，精确到 G3-R003 级别）
  □ 执行批次（EXEC-0/1/2）
  □ 判定等级（P1-P5，基于历史推演结果）
  □ 证据等级（E0-E3）
  □ 历史问题代码（如 I01、BUG-03，原场景报告中标注的问题代码）

允许保持不变的字段：
  ✓ 初始状态（保留原始值）
  ✓ 触发事件序列（保留原始值）
  ✓ 演化推理链（保留原始推演）
  ✓ 原始结论（保留，作为历史记录）

新增结论格式（基于 T-PA-G 框架重新判定）：
  在原始结论之后追加一节"T-PA-G 框架重新判定"，
  按标准记录模板 H 节格式填写，不修改原始内容
```

### 6.4 极端触发器（E/EE 类场景）

```
G1 不变量边界触发器（合法性压测）：
  energy = 0 精确边界 / energy = CRITICAL(15) / energy = SAFE(30) ± ε
  foreground_stack.depth = MAX(20)（满栈时 push）
  foreground_stack.depth = 0（空栈时 pop）
  effect 无终局状态（TTL=null，永久 pending）← 对应 TERMINAL 域

G3 稳定性冲突触发器（持续演化压测）：
  P0 vs P2：energy=0 且 field.task_binding=mission_locked
  Hysteresis Lock vs 生命威胁：energy=5 + semantic_class=hazard 事件
  ephemeral lifecycle + mission_locked（逻辑锁死 vs 物理散场）
  sleeping + sleep_protection vs urgent 系统灾害

G4 语义进展触发器（开放演化压测）：
  LLM 持续输出无效消息（测水群欺骗 N-tick）
  Obligation 双向 blocking_on（死锁环路）
  所有 agent energy=0 的集体崩溃

G6/G8 工程故障触发器（一致性+鲁棒性压测）：
  ActionReceipt 超时（timeout=30s，测 unknown 状态机路径）
  时钟回退（now < last_evolved_at，测 T1 公设）
  双活 Agent Replica（并发两个实例写同一状态）
  节点崩溃重启（测 WritebackLedger replay + jitter 重置）
  物理位置被强制删除（place_id 不存在，测 G7 Stale Frame）
```

## 第七章：任务设计模板

### 7.1 标准任务模板（完整版，适用 GEN/MAP/SYS/HYB）

```markdown
# [任务标题]

## A. 任务元信息
- 任务编号：[见第四章编号规范]
```
