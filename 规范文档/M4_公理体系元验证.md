> **文档编号**：M4

> **启动条件**：仅当 M3 的表7（体系缺口审查表）出现至少一条 GAP- 条目并标注"已移交 M4"时，才打开本文档。**日常沙盘推演不需要读本文档。**

## 第一章：为什么需要元验证

### 1.1 沙盘推演的双重验证能力

沙盘推演不仅验证 v6 系统，还能反过来发现 T-PA-G 体系自身的缺陷。这是本套方法论区别于普通测试的关键特性：

```
普通测试只回答：系统有没有做到 X？
沙盘推演还回答：X 是否被规范正确地定义了？
                 若 X 发生了失败，是系统问题还是规范遗漏了这类问题？
```

### 1.2 体系缺口的本质

当一个失败无法在 T-PA-G 体系的任何条目中找到归宿时，有三种可能：

| 可能性 | 含义 | 处理方式 |
|--------|------|---------|
| **A 体系遗漏** | 该失败对应一类约束，T-PA-G 体系根本没有建模 | 补充新条目（PA 或 G 层）|
| **B 体系缺口** | 两个已有条目之间缺少中间连接，导致失败落入缝隙 | 增补桥接公理或明确依赖关系 |
| **C 推演者判断失误** | 失败实际上可以归因，只是推演者没有找到对应条目 | 重新检查 M1 映射表，不启动 M4 |

**M4 只处理 A 和 B，不处理 C。**  
在进入 M4 之前，必须确认已排除 C（至少已检查 M1 中该条目的所有上收路径）。

## 第二章：启动 M4 的判定标准

### 2.1 五类触发信号

以下任一信号出现，且证据等级 ≥ E2，才可进入 M4：

| 信号 | 触发条件 | 说明 |
|------|---------|------|
| **S1 无法归类型** | 存在明确失败，但现有 T/PA/G 均无对应条目可解释 | 非推演者遗漏（已检查 M1 全量），而是体系真的没有该约束 |
| **S2 双重失效无桥接型** | 同一场景同时损害两个 T 目标，但 PA 图上无连接路径 | 说明两 T 之间可能有未被建模的中间约束 |
| **S3 概率性不满足无法收敛型** | 多个任务域均出现 P3，继续下钻后仍无法收束到 P4 | 说明底层可能缺少稳定性类公理，而非系统设计问题 |
| **S4 要求层补丁膨胀型** | 只能不断新增孤立 G 层补丁才能解释同一类问题，无法上收到 PA | 说明这些补丁有共同本质，需提炼为 PA 公理 |
| **S5 互斥结论型** | 不同任务对同一公理给出稳定冲突而体系无优先级规则 | 说明原公理表述有歧义或需要拆分 |

### 2.2 排除误触发

在确认启动 M4 之前，必须先排除以下误触发情形：

```
误触发检查（若以下任一成立，则不是 M4，是系统问题或 M3 分析问题）：
□ 是否是 v6 已知限制（L1-01 ~ L1-07）？→ 已知限制不进 M4
□ 是否是 G 层要求描述不清楚（模糊表述），而非体系遗漏？→ 修订 G 层描述即可
□ 是否是 PA 的 G 层落地不完整（PA 存在但 G 层漏掉了一些要求）？→ 补全 G 层要求即可
□ 是否是推演者未检查到 M1 中的相关映射路径？→ 重查 M1，不进 M4
□ 证据等级是否 ≥ E2？→ E0/E1 证据不足以判定体系缺口，先补场景
```

只有排除以上全部，才进入 M4 完整协议。

## 第三章：M4 完整工作协议

### 3.1 第一步：形式化描述缺口

将 M3 表7 中的 GAP- 条目扩展为完整的缺口描述文档：

```markdown
# GAP-[编号] 缺口描述

## 基本信息
- 缺口编号：GAP-XXX
- 触发信号类型：S1/S2/S3/S4/S5
- 来源 GAP 条目：M3 表7 中的 GAP-XXX
- 证据等级：E2 / E3
- 核心证据记录：[record_id 列表]

## 缺口描述
- 观察到的失败现象：
- 在现有 T-PA-G 体系中的归因尝试（已经检查过的路径）：
  - 尝试归因到 T?：[失败原因]
  - 尝试归因到 PA?：[失败原因]
  - 尝试归因到 G?：[失败原因]
- 无法归因的具体原因（说清楚"哪里空白"）：

## 与现有条目的关系
- 最接近的现有 T 条目及其差异：
- 最接近的现有 PA 条目及其差异：
- 最接近的现有 G 条目及其差异：

## 初步修正方向假设
- 若是 S1（遗漏）：新增什么内容？应在哪层？
- 若是 S2（无桥接）：在哪两个条目之间补充桥接？
- 若是 S3（无法收敛）：应提炼什么 PA 公理？
- 若是 S4（补丁膨胀）：共性是什么？应上收为哪类 PA？
- 若是 S5（互斥结论）：歧义在哪里？应如何拆分或明确边界？
```

### 3.2 第二步：可行性评估

对每个修正方向，评估以下三项：

| 评估项 | 问题 | 可接受的答案 |
|--------|------|-----------|
| **T 层服务** | 新条目服务于哪个 T 目标？ | 至少一个 T 目标（否则不应进入主干）|
| **PA 层是否已覆盖** | 是否现有 PA 已隐式包含该约束？ | 若是，则只需扩展 G 层，不需新增 PA |
| **与现有条目的矛盾** | 新条目是否与现有任何 T/PA/G 产生逻辑矛盾？ | 必须不产生矛盾，否则需先解决矛盾 |

### 3.3 第三步：修正草案

按以下格式起草修正方案：

```markdown
# 修正草案 AMEND-[编号]

## 来源缺口
- GAP 编号：GAP-XXX
- 缺口类型：S1/S2/S3/S4/S5

## 修正内容
- 修正类型：增补适用域 / 增补中间桥接公理 / 增补判定条件 / 上收若干 G 为 PA / 拆分原有 PA / 明确优先级边界
- 修正层级：T 层 / PA 层 / G 层
- 修正的具体文字（仅新增部分，不改动原有文字）：

  若新增 PA 公理（格式参照现有 PA 公理格式）：
    PA[N] [名称]：
    表述：
    含义：
    服务 T 目标：
    若缺失：

  若扩展 G 层要求（格式参照现有 G 要求格式）：
    G[N]-R[XXX] [名称]：
    PA 根源：
    服务 T：
    要求内容：
    层级：硬要求 / 判准 / 增强项

## 与原有体系的关系
- 新增内容服务的 T 目标：
- 新增内容具体化的 PA 公理（若是 G 层新增）：
- 与哪些现有条目的关系（补充/细化/依赖）：
- 是否影响现有任何条目的结论（若是，说明如何影响）：

## 自洽性检查（四条必须全部通过）
□ 覆盖性：每个 T 目标至少有一条 PA 服务（新增后仍成立）
□ 无悬空：每条 PA 公理服务至少一个 T 目标（新 PA 已服务 Tx）
□ G 追溯：新 G 要求可追溯到 PA（追溯链已填写）
□ 无循环依赖：新内容不与现有任何条目形成循环
```

### 3.4 第四步：影响范围评估

修正草案提出后，评估其对现有沙盘结论的影响：

```
对现有汇总矩阵的影响评估：
  哪些现有 P1 结论可能受影响？（若新 PA 的引入使原来"通过"的场景变为"部分通过"）
  哪些现有 P4 结论可以用新条目重新归因？（使根因链更精确）
  哪些现有 P5 结论可能变为可判定？（新条目提供了判定依据）

影响分级：
  无影响：新条目只解释了原来无法解释的情况，不改变现有结论
  轻微影响：少量现有结论需要精化（但不会从 P1 变 P4）
  中等影响：部分现有结论需要重验
  重大影响：大量现有结论需要重验 → 建议在修正后触发新一轮沙盘推演（M2/M3）
```

### 3.5 第五步：最终决策

| 决策 | 条件 | 行动 |
|------|------|------|
| **接受草案** | 自洽性检查全部通过，影响范围可接受 | 修订 T-PA-G 文档，并在沙盘汇总报告中记录"体系修订历史" |
| **修改草案** | 自洽性检查部分失败，或影响范围过大 | 返回步骤 3，重新起草 |
| **推迟** | 证据不足（E1 级别），或影响范围重大但无法立即重验 | 标注为"待确认缺口"，下一轮沙盘重点覆盖 |
| **否决** | 确认是 C 类误触发，或修正方向找不到 T 目标服务 | 关闭 GAP 条目，记录关闭原因 |

## 第四章：修正原则（四条绝对约束）

以下四条原则在任何情况下不得违反：

### 原则 1：增量补充，不替换

```
❌ 禁止：将现有 PA 公理重新定义，改变其原有含义
✅ 正确：在现有公理之外新增，明确声明与已有公理的关系

为何重要：现有 PA 公理是大量 G 层要求的追溯基础。改变 PA 含义会使数十条
         G 要求的追溯链失效，且改动影响难以穷举。
```

### 原则 2：层级归属严格

```
新增内容必须放在正确的层级：

若对所有多智能体系统普遍成立 → T 层或 PA 层
若仅对当前 v6 粒子体系成立 → G 层

❌ 严禁：将仅对 v6 体系成立的内容提升到 T/PA 层（会错误约束其他系统）
❌ 严禁：将普遍性约束降级到 G 层（会导致跨版本失去追溯性）

判断方法：将新内容的表述去掉"三粒子"、"ForegroundStack"等 v6 特化词汇，
         若仍然成立 → 可以是 PA/T 层；若不成立 → 只能是 G 层
```

### 原则 3：向上追溯链必须完整

```
每条新增公理或要求必须完成以下检查：
  □ 它服务于哪个 T 目标？（若无，则不应进入主干，改为附注或待定）
  □ 它是否已被现有 PA 公理隐式覆盖？（若是，只需扩展 G 层即可）
  □ 它与现有 PA 公理是否有逻辑矛盾？（若有，先解决矛盾再新增）
```

### 原则 4：修正后验证覆盖性与自洽性

```
修正后必须检查（参照 T-PA-G 文档附录 B 的自洽性校验）：
  ✓ 每个 T 目标至少有一条 PA 公理服务（覆盖性）
  ✓ 每条 PA 公理服务至少一个 T 目标（无悬空公理）
  ✓ G 层中每条要求均可追溯到 PA（裁剪元原则 Gm-R001）
  ✓ 新增公理不与现有公理产生循环依赖
```

## 第五章：典型缺口类型的修正模式

### 5.1 S1 无法归类型缺口的修正模式

**现象**：有失败，但 T/PA/G 无对应条目。

**修正思路**：

```
Step 1：确认该约束是普遍性的（对所有多智能体系统）还是 v6 特化的
Step 2：若普遍性 → 在 PA 层新增公理
        格式："系统必须保证 X 性质，否则 Y 目标面临威胁"
        服务 T：确认服务于哪个 T（通常是 T1-T5 之一）
Step 3：若 v6 特化 → 在 G 层新增要求，并追溯到最接近的 PA 公理
Step 4：若无法判断层级 → 先放 G 层，附注"待确认是否提升到 PA 层"
```

**典型案例**：长期运行下的记忆摘要语义退化（信息熵热寂）

```
原有情况：PA2（有界性）只管状态量的数量上界，未管语义保真
缺口：递归摘要导致历史语义不断压缩退化，这不是"数量超界"，而是"质量退化"
修正方向：
  若认为"语义保真性"是所有多智能体系统的普遍约束 → 在 PA 层新增 PA10（语义保真公理）
  若认为仅是 LLM 型 agent 的特化约束 → 在 G7（认知连续性）下新增 G7-R007
```

### 5.2 S2 双重失效无桥接型缺口的修正模式

**现象**：同一场景同时损害 T_a 和 T_b，但 PA 图上无连接路径。

**修正思路**：

```
Step 1：明确两个 T 目标在该场景下的失效路径
  T_a 失效路径：现象 → G_x → PA_y → T_a
  T_b 失效路径：现象 → G_u → PA_v → T_b
Step 2：检查 PA_y 和 PA_v 是否有隐式关联（但未在体系中说明）
  若有 → 在现有条目的说明中补充依赖关系，不需新增 PA
  若无 → 检查是否存在一个底层约束 PA_w，同时支撑 PA_y 和 PA_v
Step 3：若确认需新增 PA_w →
  PA_w 必须同时服务于 T_a 和 T_b
  PA_w 的表述应比 PA_y 和 PA_v 更基础（更接近 L0 层）
```

### 5.3 S4 要求层补丁膨胀型缺口的修正模式

**现象**：不断新增 G 层补丁解释同一类问题。

**修正思路**：

```
Step 1：收集所有相关的 G 层补丁，找出共性
Step 2：提炼共性：这些补丁的共同约束是什么？
  示例：多个 G 层补丁都在说"某类效应必须有终局"→ 共性是"效应有终局"(PA6)
Step 3：检查该共性是否已被现有 PA 覆盖
  若是 → 问题是 G 层落地不完整，而非体系遗漏（不需新 PA，补全 G 层即可）
  若否 → 新增 PA 公理，然后把所有相关 G 补丁重新追溯到该 PA
Step 4：清理冗余 G 层补丁（合并到新 PA 的具体化中）
```

## 第八章：M4 质量自检

在完成 M4 工作后，检查以下项目：

```
缺口识别质量
□ 已排除所有 C 类误触发（确认不是系统问题或 M1 未检查到的路径）
□ 证据等级 ≥ E2（E0/E1 不足以判定体系缺口）
□ 触发信号类型已明确（S1-S5 之一）

修正草案质量
□ 新增内容服务的 T 目标已明确
□ 层级归属正确（普遍性 → PA/T；v6 特化 → G）
□ 与现有条目无逻辑矛盾
□ 向上追溯链完整（G→PA→T）

自洽性验证
□ T 层覆盖性验证通过（每个 T 至少一条 PA 服务）
□ PA 无悬空验证通过（每条 PA 服务至少一个 T）
□ G 追溯性验证通过（每条 G 可追溯到 PA）
□ 无循环依赖

影响评估完整
□ 已评估对现有汇总矩阵的影响（无影响/轻微/中等/重大）
□ 若影响重大，已制定重验计划
```
